# План Stage 1. Admin Group Detail

## 0. Статус

- Статус плана: реализация и проверки выполнены, stage готов к pull request.
- Ветка: `feat/mpmf-1-stage-1-admin-group-detail`.
- Roadmap: `docs/roadmap/mpmf-1-implementation-roadmap.md`.
- Предыдущий завершенный stage: `MPMF 1.0 Implementation Roadmap`, PR #29.

## 1. Цель

Сделать рабочую административную карточку группы на маршруте
`/admin/groups/[groupId]`, чтобы администратор мог видеть состояние группы,
редактировать базовые данные и управлять активным составом без прямой работы в
Supabase.

## 2. Источники правды

Перед кодом прочитаны и используются:

- `docs/roadmap/README.md`;
- `docs/roadmap/mpmf-1-implementation-roadmap.md`;
- `docs/specs/README.md`;
- `docs/specs/02-feature-specs/group-management.md`;
- `docs/specs/03-technical-specs/page-contracts.md`;
- `docs/specs/03-technical-specs/pages-and-routes.md`;
- `docs/specs/03-technical-specs/api-actions.md`;
- `docs/specs/03-technical-specs/states-and-validation.md`;
- `supabase/schema.sql`;
- текущие страницы и data/actions в `app/admin`, `app/lib/data`.

## 3. Что входит

- Реализовать страницу `/admin/groups/[groupId]` вместо пустой заглушки.
- Показать название, курс, преподавателя, статус группы и базовую сводку.
- Показать активных учеников группы.
- Добавить форму `Изменить группу` для названия, преподавателя и статуса.
- Добавить форму `Добавить ученика` в текущую группу.
- Показать проблемные признаки:
  - нет преподавателя;
  - нет активных учеников;
  - нет активного расписания;
  - нет ближайших занятий.
- Показать read-only блоки расписания и ближайших занятий, если данные уже есть.
- Добавить рабочие переходы на журнал и оплату.
- Добавить переход из `/admin/groups` в карточку группы через действие
  `Открыть группу`.
- Оставить действия расписания и генерации занятий как границу следующего stage,
  если они не были уже реализованы в текущем коде.

## 4. Что не входит

- Создание или редактирование `ScheduleRule`.
- Генерация `Lesson` по расписанию.
- Редактирование журнала.
- Страница урока.
- Массовая оплата или изменение статусов оплат.
- Расширенная карточка курса, ученика или преподавателя.

## 5. Технический план

1. Расширить `app/lib/data/supabase-read.ts` типами и reader-функцией для
   `AdminGroupDetailData`.
2. Расширить `app/lib/data/admin-write.ts` записью обновления группы.
3. Обновить `app/admin/actions.ts`, чтобы `updateGroup`, `addStudentToGroup` и
   связанные revalidate/redirect реально работали для карточки группы.
4. Реализовать UI в `app/admin/groups/[groupId]/page.tsx` с существующими
   компонентами `SupabaseDataPage`, `DataTable`, `PageCreateAction` и локальными
   CSS-паттернами.
5. Обновить `/admin/groups`, чтобы список вел в `/admin/groups/[groupId]`.
6. При необходимости добавить небольшие CSS-правки в `app/globals.css`, только
   для уже используемой административной сетки и списков.

## 6. Проверка готовности

Stage считается готовым, если:

- `/admin/groups` показывает действие `Открыть группу`;
- `/admin/groups/[groupId]` открывается под админом и показывает данные группы;
- изменение базовых данных группы сохраняется в Supabase;
- добавление ученика в группу работает без дублей;
- проблемные признаки показываются по фактическим данным;
- кнопки журнала и оплаты ведут только на существующие маршруты;
- `npm.cmd run lint` проходит успешно;
- `npm.cmd run build` проходит успешно;
- Codex выполнил локальный smoke-check, если dev-сервер и Supabase доступны.

## 7. Ручной smoke-чеклист после автоматических проверок

- Войти как админ.
- Открыть `/admin/groups`.
- Нажать `Открыть группу` у существующей группы.
- Изменить название или статус группы и убедиться, что карточка обновилась.
- Добавить активного ученика в группу и убедиться, что он появился в составе.
- Проверить, что блоки проблем не показывают ложные проблемы на seed-группе.
