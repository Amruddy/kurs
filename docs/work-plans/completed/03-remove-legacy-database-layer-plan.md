# Remove Legacy Database Layer Plan

# План удаления старого слоя базы данных

## 0. Текущий статус

- Статус плана: завершен и смержен в `main`.
- Цель: полностью удалить текущий слой базы данных из приложения перед новым
  проектированием подключения Supabase.
- Старая локальная база данных не нужна и не переносится.
- Миграции старой локальной базы не нужны.
- В этом этапе не создается новая схема Supabase и не подключается новая база.

## 1. Решение

Проект временно работает без runtime-базы данных:

```text
Next.js app -> временные пустые состояния
```

На этом этапе нужно удалить Prisma, `pg`, локальные SQL-миграции, seed,
migrate-скрипты, smoke-сценарии, `.env`-пример подключения и все runtime-импорты
слоя БД из приложения.

Dev-вход должен остаться рабочим без базы: роли выбираются кнопками на `/login`
и хранятся в cookie только для ручной навигации по маршрутам.

Этот этап выполнен и зафиксирован отдельно от следующего Supabase-этапа.
Рабочая ветка этапа: `chore/remove-legacy-db-and-restructure-docs`.
Ветка фиксирует удаление старого слоя БД, временный режим без данных и
реструктуризацию документации: specs, roadmap и work-plans разделены.

## 2. Что входит

- удалить `prisma/`, `prisma.config.ts`, `app/lib/prisma.ts`;
- удалить `app/lib/db.ts`, `app/lib/db-types.ts`;
- удалить `database/` и SQL-migration файлы;
- удалить `scripts/seed.ts`, `scripts/migrate.ts`, `scripts/smoke-mvp.ts`;
- убрать DB-скрипты из `package.json`;
- убрать DB-зависимости и типы из `package.json`;
- убрать `DATABASE_URL` из `.env.example`;
- убрать обращения к БД из страниц, server actions и dev-auth;
- оставить маршруты приложения с временными пустыми состояниями;
- проверить, что приложение собирается без DB-зависимостей.

## 3. Что не входит

- перенос данных из старой локальной базы;
- создание новой схемы Supabase;
- миграции Supabase;
- seed-данные;
- smoke-сценарий, который требует данных;
- восстановление полного пользовательского функционала на этом этапе.

## 4. Проверка готовности

План считается выполненным, если:

- в runtime-коде нет импортов `@prisma/client`, `PrismaClient`, `pg`,
  `app/lib/prisma`, `app/lib/db`, `app/lib/db-types`;
- в репозитории нет активных Prisma/SQL migration/seed/migrate/smoke файлов;
- `package.json` не содержит DB-скриптов и DB-зависимостей;
- `.env.example` не содержит строки подключения к базе;
- `npm run build` проходит успешно;
- `npm run lint` проходит успешно или оставшиеся замечания не связаны с этим
  этапом.

## 5. Порядок завершения ветки

1. Codex показывает пользователю, что уже проверено, и дает короткий ручной
   smoke-чеклист.
2. Пользователь либо проверяет вручную, либо явно разрешает продолжать без
   ручной проверки.
3. Только после явного разрешения пользователя Codex коммитит текущий этап в
   текущую stage-ветку.
4. Codex пушит stage-ветку в GitHub.
5. Pull request открывает пользователь.
6. После merge пользователь сообщает Codex, что PR смержен.
7. Codex возвращается на `main`, обновляет `main` из GitHub и только потом
   создает новую ветку для следующего этапа.

Следующий этап подключения Supabase нельзя начинать в этой ветке и нельзя
начинать до merge этого DB reset этапа.
