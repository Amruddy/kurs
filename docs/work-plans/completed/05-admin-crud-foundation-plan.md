## 1. Цель

Вернуть минимальные административные формы и server actions поверх Supabase,
чтобы администратор мог вручную создать базовые учебные данные без прямой
работы в Supabase SQL Editor.

Архитектура этапа:

```text
Admin pages -> app/admin/actions.ts -> app/lib/data/* -> Supabase
```

## 2. Что входит в этап

- добавить кнопку и форму создания курса на `/admin/courses`;
- добавить кнопку и форму создания ученика на `/admin/students`;
- добавить кнопку и форму создания группы на `/admin/groups`;
- добавить назначение ученика в группу на `/admin/groups`;
- использовать существующий dev-auth и текущую организацию пользователя;
- сохранять записи в Supabase через server actions;
- после записи делать `revalidatePath` и возвращать пользователя на список;
- оставить интерфейс на русском языке;
- не возвращать Prisma и локальную базу.

## 3. Что не входит в этап

- Supabase Auth;
- RLS-политики;
- полное редактирование и архивирование всех сущностей;
- расписание и генерация уроков;
- создание преподавателей;
- платежи и история оплаты;
- журнал, посещаемость, домашние задания, материалы и прогресс;
- сложные фильтры, поиск и bulk-actions.

## 4. Технические правила

- Server actions не должны обращаться к Supabase напрямую, если операцию можно
  вынести в `app/lib/data/*`.
- Валидация на этом этапе минимальная, но обязательные поля должны проверяться
  до записи.
- Ошибочные данные не должны создавать пустые записи.
- `service role key` остается только на сервере.
- Новые записи всегда получают `organization_id` текущей dev-сессии.
- Для курса при создании сразу создается `course_progress_settings`.
- Для группы преподаватель выбирается из пользователей с ролью `teacher` в
  текущей организации.
- Для назначения ученика в группу повторное назначение не должно ломать
  страницу.

## 5. Проверка готовности

Этап считается готовым, если:

- `npm run lint` проходит успешно;
- `npm run build` проходит успешно;
- локально открывается `/login`;
- админ может создать курс на `/admin/courses`;
- админ может создать ученика на `/admin/students`;
- админ может создать группу на `/admin/groups`;
- админ может назначить ученика в группу на `/admin/groups`;
- созданные данные видны на admin-страницах после сохранения;
- в проекте нет Prisma и локальной базы;
- секреты не попали в Git.

## 6. Порядок завершения ветки

1. Codex выполняет автоматические проверки.
2. Codex выполняет базовую локальную smoke-проверку.
3. Codex показывает пользователю, что проверено, и дает короткий ручной
   smoke-чеклист.
4. Пользователь явно разрешает продолжать.
5. Codex коммитит stage-ветку.
6. Codex пушит stage-ветку в GitHub.
7. Pull request открывает пользователь.
8. После merge пользователь сообщает Codex.
9. Codex возвращается на `main`, обновляет `main` из GitHub и только потом
   начинает следующий этап.
