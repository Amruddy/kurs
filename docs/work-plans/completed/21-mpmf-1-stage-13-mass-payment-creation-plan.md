## 1. Цель

Закрыть добавление оплаты для администратора без изменения текущей структуры данных: администратор должен из одного окна добавить оплату всей группе или одному ученику выбранной группы, а система должна создать отдельные записи оплат конкретных учеников.

## 3. Что входит

- Единая кнопка `Добавить оплату` на `/admin/payments`.
- Кнопка `Добавить оплату` находится в шапке блока `Список оплат`.
- Фильтры на `/admin/payments` скрыты в раскрывающемся действии рядом со списком оплат; раскрытая панель должна быть заметной и читаемой.
- Добавление оплаты всей активной группе из одного окна.
- Добавление оплаты одному ученику из выбранной группы в том же окне.
- Удаление лишнего выбора курса из окна добавления оплаты: курс определяется через группу.
- Хранение результата как отдельных строк `payments` с `student_id`, `course_id` и, для группы, `group_id`.
- Защита от дублей за тот же контекст, период и срок оплаты.
- Запись истории создания в `payment_history`.
- Понятный результат после добавления: сколько оплат создано и сколько пропущено.
- Редактирование суммы, срока, периода и комментариев в списке `/admin/payments`.
- Редактирование суммы, срока, периода и комментариев в карточке ученика.
- Смена статуса оплаты только в общем списке `/admin/payments`.
- Имена учеников и контекст группы/курса в списке оплат кликабельны.

## 4. Что не входит

- Онлайн-оплата.
- Интеграции с банками, кассами и платежными провайдерами.
- Сложные тарифные планы.
- Новые таблицы или миграции, если текущих `payments`, `payment_history`, `course_id`, `group_id` и `student_id` достаточно.
- Полная переработка UI страницы оплат.

## 5. Технический план

1. Обновить roadmap, active-plan, completed-plan и `AGENTS.md` со Stage 12 на Stage 13.
2. Добавить data-layer функцию добавления оплат для группы.
3. Получать только активных учеников текущей организации.
4. Перед вставкой проверять существующие оплаты по ученику, курсу, группе, типу периода, датам периода и сроку оплаты.
5. Создавать недостающие строки `payments` и записи `payment_history`.
6. Добавить единый server action для окна `Добавить оплату`.
7. Добавить server action для редактирования деталей оплаты без смены статуса.
8. Добавить действия на `/admin/payments`, карточку ученика и сообщение о результате.
9. Выполнить автоматические проверки и локальный smoke-check.

## 6. Проверка готовности

- Администратор видит на `/admin/payments` одну кнопку `Добавить оплату`.
- В окне добавления есть только две области: `Всей группе` и `Одному ученику`.
- При выборе `Всей группе` поле ученика не показывается и не отправляется.
- При выборе `Одному ученику` сначала выбирается группа, затем ученик этой группы.
- Добавление оплаты группе создает оплаты только активным ученикам этой группы.
- Повторное добавление оплаты с тем же контекстом, периодом и сроком не создает дубли и показывает количество пропущенных записей.
- Сумма, срок, период и комментарии редактируются из `/admin/payments` и карточки ученика.
- Статус оплаты редактируется только из `/admin/payments`.
- Изменение статуса в списке оплат сохраняется выбором статуса без отдельной кнопки сохранения.
- Действия в строке оплаты располагаются вертикально, чтобы кнопка редактирования и выбор статуса читались как единый компактный блок.
- `npm.cmd run lint` проходит успешно.
- `npm.cmd run build` проходит успешно.
- Codex выполняет локальный smoke-check страницы оплат, если dev-сервер и Supabase доступны.
