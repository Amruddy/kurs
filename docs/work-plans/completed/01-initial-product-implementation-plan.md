# MPMF 1.0 Implementation Plan

# План разработки первого рабочего релиза

## 0. Текущий статус

- Текущий этап: `Stage 8: финальная smoke-проверка MPMF 1.0`.
- Статус этапа: завершен.
- Следующий технический коммит после реализации: `Complete MPMF 1.0 smoke flow`.
- Stage 0, Stage 1, Stage 2, Stage 3, Stage 4, Stage 5, Stage 6, Stage 7 и Stage 8 завершены.
- MPMF 1.0 готов к первому ручному тестированию после прохождения полного smoke-сценария.
- Новый этап разработки начинается только с отдельной ветки от актуального `main`.

## 1. Назначение документа

Этот документ переводит спецификации MPMF 1.0 в порядок разработки.

Он не добавляет новые функции. Если функция не описана в предыдущих спеках, она
не попадает в план разработки.

Перед началом Stage 0 нужно пройти checklist:

`docs/work-plans/00-before-stage-0-checklist.md`

Задача документа:

- разбить MPMF 1.0 на маленькие этапы;
- определить, какие сущности и страницы делать первыми;
- зафиксировать проверяемый результат каждого этапа;
- не смешивать базовый MPMF 1.0 с расширенными страницами после MPMF 1.0;
- дать понятный порядок коммитов.

Главный проверяемый путь MPMF 1.0:

1. Админ создает учебную основу.
2. Преподаватель проводит урок.
3. Ученик видит результат.
4. Админ создает оплату.
5. Ученик видит статус оплаты.

## 2. Принцип разработки

Разработка идет от минимального рабочего пути к расширению.

Каждый этап должен давать результат, который можно открыть в интерфейсе или
проверить вручную.

Не нужно сначала делать все страницы пустыми. Лучше сделать небольшой путь
полностью:

- данные;
- права;
- страница;
- действие;
- проверка результата.

Если этап нельзя проверить руками, он слишком большой или плохо определен.

## 3. Что не входит в первый рабочий план

В первом плане не делаем:

- маркетинговый сайт;
- мобильное приложение;
- онлайн-оплату;
- загрузку файлов, PDF, аудио, видео и изображений;
- чат;
- уведомления;
- сложную аналитику;
- экспорт;
- несколько учебных заведений в одном интерфейсе;
- сложную матрицу прав;
- отдельный рабочий путь индивидуального обучения;
- страницы курсов преподавателя `/teacher/courses` и `/teacher/courses/:courseId`;
- отдельные страницы индивидуального обучения `/teacher/individual` и
  `/teacher/individual/:enrollmentId`;
- отдельную страницу курса ученика `/student/courses/:courseId`;
- общий раздел материалов администратора `/admin/materials`;
- отдельные детальные страницы всех материалов и домашних заданий.

Эти части можно проектировать после прохождения базового smoke-сценария.

## 4. Stage 0: техническая основа

### Цель

Создать пустое приложение, в котором уже есть база для ролей, маршрутов, данных
и общего интерфейса.

### Входит

- Next.js-приложение;
- TypeScript;
- базовая структура App Router;
- активный слой базы данных временно отсутствует;
- старая локальная база, Prisma, миграции, seed и DB-smoke удалены без переноса
  данных;
- новый Supabase-подход должен проектироваться отдельным планом с чистой схемой;
- общий layout приложения;
- базовые страницы ошибок и пустых состояний;
- базовая тема интерфейса по `04-visual-rules.md`;
- seed-данные для одной организации, админа, преподавателя и ученика;
- стабильные тестовые логины для ручной проверки;
- dev-auth через кнопки "Войти как админ", "Войти как преподаватель",
  "Войти как ученик" без настоящей регистрации.

Stage 0 создает минимальную техническую модель пользователей, ролей и организации
только для запуска приложения и dev-auth. Полная настройка доступа, рабочих
областей и защиты маршрутов относится к Stage 1.

### Минимальные страницы

- `/login`;
- `/admin`;
- `/teacher`;
- `/student`.

На этом этапе страницы могут быть простыми, но должны открываться.

### Проверка готовности

- приложение запускается локально;
- приложение работает без активного слоя базы данных;
- dev-вход работает без seed-данных;
- тестовые пользователи выбираются кнопками на `/login`;
- можно открыть `/login`, `/admin`, `/teacher`, `/student`;
- роли в seed-данных различаются.

### Не входит

- настоящая регистрация;
- полноценная production-авторизация;
- восстановление пароля и email-приглашения;
- бизнес-действия;
- все CRUD-страницы.

## 5. Stage 1: организация, пользователи и роли

### Цель

Сделать основу доступа: пользователь работает внутри организации и видит свою
рабочую область.

Stage 1 не создает второй вариант auth и не дублирует Stage 0. Он берет
минимальную dev-auth основу из Stage 0 и доводит ее до проверяемой ролевой
маршрутизации и базовой защиты страниц.

### Сущности

- `Organization`;
- `User`;
- `OrganizationMember`;
- базовые роли `admin`, `director`, `teacher`, `student`;
- отдельное право `payments:write`.

### Входит

- вход через тестового пользователя или простую dev-auth схему;
- определение активной организации;
- определение доступных ролей пользователя;
- переключение рабочей области, если у пользователя несколько ролей;
- защита базовых маршрутов по роли.

### Проверка готовности

- админ попадает в `/admin`;
- преподаватель попадает в `/teacher`;
- ученик попадает в `/student`;
- пользователь без роли не получает доступ к чужой рабочей области;
- преподаватель-одиночка может иметь роли `teacher` и `admin`.

### Не входит

- восстановление пароля;
- приглашения пользователей;
- сложная матрица прав.

## 6. Stage 2: курсы, ученики, преподаватели и группы

### Цель

Админ должен создать учебную основу: курс, группу, преподавателя и учеников.

### Сущности

- `Course`;
- `CourseProgressSettings`;
- `Student`;
- `StudentContact`;
- `Group`;
- `GroupStudent`.

### Страницы

- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`.

### Действия

- создать курс таджвида;
- включить простой блок таджвид-прогресса для курса;
- создать ученика;
- создать преподавателя;
- создать группу;
- назначить курс группе;
- назначить преподавателя группе;
- добавить ученика в группу;
- убрать ученика из активного состава группы без удаления истории.

### Проверка готовности

- админ создает курс;
- админ создает преподавателя;
- админ создает ученика;
- админ создает группу;
- группа связана с курсом;
- у группы есть преподаватель;
- ученик добавлен в группу;
- группа видна на странице преподавателя.

### Не входит

- расписание;
- уроки;
- журнал;
- оплата;
- индивидуальное обучение как отдельный рабочий путь.

## 7. Stage 3: расписание и создание уроков

### Цель

Группа получает повторяющееся расписание, а система создает ближайшие уроки.

### Сущности

- `ScheduleRule`;
- `Lesson`.

### Страницы

- `/admin/groups/:groupId`;
- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`.

### Действия

- задать повторяющееся расписание группе;
- создать ближайшие уроки по расписанию на один календарный месяц;
- показать ближайший урок админу;
- показать ближайший урок преподавателю;
- открыть страницу группы преподавателя.

### Проверка готовности

- админ задает расписание группе;
- расписание может быть задано на весь срок курса;
- система создает уроки на один календарный месяц;
- повторная генерация не создает дубликаты уроков;
- у урока статус `scheduled`;
- преподаватель видит ближайший урок на `/teacher`;
- преподаватель может открыть группу.

### Не входит

- перенос урока;
- отмена урока;
- сложная генерация расписания на длинный период;
- индивидуальное расписание.

## 8. Stage 4: журнал, урок и посещаемость

### Цель

Преподаватель может открыть урок, заполнить запись урока и отметить посещаемость. В MPMF 1.0 у урока нет пользовательского статуса "начат" или "завершен".

### Сущности

- `JournalEntry`;

### Страницы

- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/attendance`;
- `/teacher/students`;
- `/teacher/students/:studentId`.

### Действия

- открыть календарный журнал группы;
- журнал группы показывает один выбранный месяц;
- в журнале показываются только реально созданные уроки, без пустых календарных дней;
- открыть сводную посещаемость преподавателя;
- показать список учеников с процентом посещаемости и количеством пропусков;
- открыть урок;
- создать или показать записи журнала по ученикам группы;
- отметить `present`, `absent`, `excused`;
- оставить пустую отметку;
- подтвердить посещаемость;
- добавить комментарий урока;
- сохранить урок;
- сохранить учебную запись урока.

### Проверка готовности

- преподаватель открывает урок;
- преподаватель отмечает одного ученика как `absent`;
- пустые отметки в прошедшем уроке считаются присутствием, если нет явной отметки `absent` или `excused`;
- журнал группы показывает результат;
- сводная посещаемость показывает учеников, процент посещаемости и количество
  пропусков за выбранный месяц.

### Не входит

- статусы проведения урока и действия "начать"/"завершить";
- редактирование прошедшего урока сложными правилами;
- история всех изменений журнала;
- админское редактирование учебных записей.

## 9. Stage 5: прогресс, домашнее задание и материалы

### Цель

После урока преподаватель фиксирует учебный результат, домашнее задание и
материал, а ученик видит только открытую часть.

### Сущности

- `ProgressRecord`;
- `StudentProgressRule`;
- `StudentProgressError`;
- `Homework`;
- `Material`.

### Страницы

- `/teacher/lessons/:lessonId`;
- `/teacher/homework`;
- `/teacher/materials`;
- `/teacher/students/:studentId`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`.

### Действия

- добавить прогресс ученику;
- добавить правило таджвида ученику;
- добавить ошибку чтения ученику;
- выбрать видимость прогресса для ученика;
- создать домашнее задание группе;
- создать индивидуальное задание ученику внутри группы;
- добавить текстовый материал;
- добавить материал-ссылку;
- скрыть материал от ученика;
- показать ученику только открытые материалы и прогресс.

### Проверка готовности

- преподаватель добавляет прогресс после урока;
- ученик видит открытую часть прогресса;
- внутренний комментарий преподавателя ученику не виден;
- преподаватель создает домашнее задание;
- ученик видит домашнее задание;
- преподаватель добавляет ссылку;
- ученик видит ссылку, если она `visible_to_students`.

### Не входит

- отправка выполненного домашнего задания учеником;
- проверка домашней работы в системе;
- файлы и медиа;
- сложные rubrics и графики прогресса.

## 10. Stage 6: кабинет ученика

### Цель

Ученик видит свой учебный минимум: ближайший урок, расписание, домашку,
материалы, посещаемость и прогресс.

### Страницы

- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`.

### Данные

- ближайший урок;
- активная группа;
- курс;
- преподаватель;
- домашние задания;
- открытые материалы;
- открытый прогресс;
- оценки и посещаемость.

### Проверка готовности

- ученик входит в кабинет;
- видит ближайший урок;
- видит расписание;
- видит домашнее задание;
- видит открытые материалы;
- видит свои оценки и посещаемость;
- видит только открытую часть прогресса;
- не видит других учеников.

### Не входит

- изменение учебных данных учеником;
- отправка домашней работы;
- родительский кабинет;
- уведомления.

## 11. Stage 7: ручная оплата

### Цель

Админ создает оплату ученику вручную, а ученик видит ее статус.

### Сущности

- `Payment`;
- `PaymentHistory`.

### Страницы

- `/admin/courses/:courseId` с блоком оплаты курса;
- `/admin/groups/:groupId` с блоком оплаты конкретной группы;
- `/student/payments`.

### Действия

- создать запись оплаты всем активным ученикам курса;
- создать запись оплаты всем активным ученикам конкретной группы;
- указать сумму;
- указать валюту;
- указать срок;
- указать период;
- поставить ручной статус `paid`, `pending`, `overdue` или `exempt`;
- изменить оплату конкретного ученика в карточке ученика;
- сохранить историю изменения;
- показать список неоплативших;
- показать оплату ученику.

### Проверка готовности

- админ создает оплату всем активным ученикам в настройках курса;
- админ создает оплату всем активным ученикам в карточке группы;
- индивидуальное изменение оплаты выполняется из карточки ученика;
- оплата привязана к курсу и может дополнительно быть привязана к группе;
- ученик видит только свою оплату;
- отдельная страница оплаты для админа или преподавателя в MPMF 1.0 не нужна;
- изменение оплаты попадает в историю.
- `pending` с прошедшим сроком может подсвечиваться как просроченная запись без
  автоматической смены сохраненного статуса.

### Не входит

- онлайн-эквайринг;
- автоматические платежи;
- автоматическая смена статуса оплаты;
- счета и квитанции;
- сложные тарифы;
- пакеты уроков.

## 12. Stage 8: финальная smoke-проверка MPMF 1.0

### Цель

Проверить, что основной путь работает от начала до конца.

### Smoke-сценарий

1. Создать организацию.
2. Создать админа.
3. Создать преподавателя.
4. Создать ученика.
5. Создать курс таджвида.
6. Создать группу.
7. Назначить курс группе.
8. Назначить преподавателя группе.
9. Добавить ученика в группу.
10. Задать расписание.
11. Создать ближайший урок.
12. Войти как преподаватель.
13. Открыть группу.
14. Открыть урок.
15. Отметить посещаемость.
16. Добавить комментарий.
17. Добавить прогресс.
18. Добавить домашнее задание.
19. Добавить материал-ссылку.
20. Войти как ученик.
21. Увидеть ближайший урок.
22. Увидеть домашнее задание.
23. Увидеть материал.
24. Увидеть посещаемость.
25. Увидеть открытую часть прогресса.
26. Войти как админ.
27. Открыть настройки курса и создать оплату всем активным ученикам курса.
28. Войти как ученик.
29. Увидеть статус оплаты.

### Готовность MPMF 1.0

MPMF 1.0 считается готовым к первому ручному тестированию, если весь smoke-сценарий
проходит без ручного исправления данных в базе.

## 13. Рекомендуемый порядок коммитов

1. `Complete MPMF 1.0 stage 0 setup`
2. `Add organization users and role routing`
3. `Implement courses groups students foundation`
4. `Add schedule rules and lesson generation`
5. `Build teacher journal and attendance flow`
6. `Add homework materials and progress`
7. `Build student cabinet MPMF 1.0`
8. `Implement manual payment tracking`
9. `Complete MPMF 1.0 smoke flow`

Коммит должен соответствовать одному проверяемому этапу. Если этап получается
слишком большим, его нужно разбить на подэтапы, но не смешивать разные
пользовательские пути в одном коммите.
