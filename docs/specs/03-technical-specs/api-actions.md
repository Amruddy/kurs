# API Actions

# Действия системы

## 1. Назначение документа

Этот документ описывает действия, которые должна уметь выполнять система.

Это не финальное описание REST API и не список конкретных URL. На этом этапе
важно зафиксировать:

- какое действие выполняется;
- кто может выполнить действие;
- какие данные нужны;
- что меняется в системе;
- какие права проверяются;
- какие важные правила нельзя нарушать.

## 2. Общие правила для всех действий

Каждое действие должно проверять:

- пользователь вошел в систему;
- пользователь относится к нужной организации;
- объект относится к той же организации;
- пользователь имеет нужную роль или отдельное право;
- преподаватель работает только со своими группами, индивидуальными учениками
  и учениками;
- ученик работает только со своими данными.

Если проверка не проходит, действие запрещается.

## 3. Авторизация и рабочая область

### 3.0. Auth MPMF 1.0 и seed-пользователи

В MPMF 1.0 вход может быть упрощенным.

До отдельного этапа Supabase Auth допустима dev-auth схема без настоящей
регистрации, восстановления пароля и email-приглашений.

Seed должен создать минимум:

- одну активную организацию;
- пользователя администратора с ролью `admin`;
- пользователя преподавателя с ролью `teacher`;
- пользователя ученика с ролью `student`;
- при необходимости отдельного пользователя преподавателя-одиночки с ролями
  `teacher` и `admin` внутри своей организации.

Тестовые логины должны быть стабильными и описанными в seed или README проекта,
чтобы smoke-сценарий можно было повторить руками.

На `/login`, пока используется dev-auth, доступны кнопки:

- "Войти как админ";
- "Войти как преподаватель";
- "Войти как ученик".

Каждая кнопка выбирает соответствующего seed-пользователя и создает
dev-сессию. Пользователь не вводит пароль в базовом MPMF 1.0.

Если пользователь имеет несколько рабочих областей, система должна явно дать
выбор рабочей области или переключатель роли. Автоматический переход без выбора
допустим только если у пользователя одна доступная рабочая область.

### 3.1. Войти в систему

Кто может: любой пользователь с учетной записью.

Данные:

- выбранный тестовый пользователь в dev-auth;
- логин и пароль позже, когда dev-auth заменяется настоящей авторизацией.

Результат:

- система определяет пользователя;
- система определяет его организации, роли и права;
- пользователь попадает в доступную рабочую область.

### 3.2. Получить доступные рабочие области

Кто может: вошедший пользователь.

Результат:

- список доступных рабочих областей: преподаватель, ученик,
  администратор/директор;
- текущая организация;
- роли и права внутри организации.

### 3.3. Добавить преподавателя в организацию

Кто может:

- админ/директор;
- преподаватель-одиночка для своей организации;
- пользователь с административным правом управления пользователями.

Данные:

- имя;
- email или другой логин;
- телефон, если используется;
- статус;
- роль `teacher`.

Результат:

- создается `User`, если такого пользователя еще нет;
- создается или обновляется `OrganizationMember`;
- пользователь получает роль `teacher` в текущей организации;
- преподавателя можно назначить на группу через `teacher_id`.

В MPMF 1.0 приглашения по email можно не делать. Достаточно создать учетную запись
или dev-доступ, чтобы преподаватель мог войти в систему.

## 4. Курсы

### 4.1. Создать курс

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с `courses:write`.

Данные:

- название;
- описание;
- тип курса;
- формат обучения;
- шкала оценки за урок: 5-балльная или 10-балльная;
- простые настройки прогресса: включен ли блок таджвид-прогресса для учеников курса.

Результат:

- создается `Course`;
- создаются или привязываются `CourseProgressSettings`;
- список правил курса не создается;
- курс получает `organization_id`.

### 4.2. Изменить курс

Кто может:

- админ/директор;
- преподаватель-одиночка для своих курсов;
- пользователь с `courses:write`.

Данные:

- название;
- описание;
- формат;
- статус.

Результат:

- обновляется курс;
- старая учебная история не меняется.

### 4.3. Архивировать курс

Кто может:

- админ/директор;
- преподаватель-одиночка для своих курсов;
- пользователь с `courses:write`.

Результат:

- курс получает статус `archived`;
- связанные прошлые уроки, журнал, прогресс, задания, материалы и оплаты не
  удаляются.

### 4.4. Изменить настройки прогресса курса

Кто может:

- админ/директор;
- преподаватель-одиночка для своих курсов;
- пользователь с `courses:write`.

Данные:

- включен ли простой блок прогресса;
- название блока в интерфейсе, если нужно.

Результат:

- обновляются настройки прогресса курса;
- правила и ошибки учеников не меняются.

## 5. Группы

### 5.1. Создать группу

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с `groups:write`.

Данные:

- курс;
- название;
- преподаватель;
- статус;
- расписание, если известно.

Результат:

- создается `Group`;
- группа связывается с курсом, преподавателем и организацией.

### 5.2. Изменить группу

Кто может:

- админ/директор;
- преподаватель-одиночка для своих групп;
- пользователь с `groups:write`.

Данные:

- название;
- преподаватель;
- статус;
- расписание.

Результат:

- обновляется группа;
- старая история группы не удаляется.

### 5.3. Добавить ученика в группу

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с `groups:write` или `students:write`.

Данные:

- `group_id`;
- `student_id`;
- дата добавления.

Результат:

- создается `GroupStudent`;
- ученик получает доступ к группе, расписанию, материалам и заданиям группы.

### 5.4. Убрать ученика из активного состава группы

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с `groups:write` или `students:write`.

Результат:

- связь `GroupStudent` получает статус `removed` или `completed`;
- заполняется `left_at`;
- прошлый журнал ученика в группе сохраняется.

## 6. Ученики

### 6.1. Создать ученика

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с `students:write`.

Данные:

- имя;
- контактные данные;
- начальный статус;
- группа, если назначается сразу.

Результат:

- создается `Student`;
- при необходимости создается связь с группой.

### 6.2. Изменить карточку ученика

Кто может:

- админ/директор;
- преподаватель-одиночка для своих учеников;
- пользователь с `students:write`.

Данные:

- имя;
- контактные данные;
- статус.

Результат:

- обновляется карточка ученика;
- учебная история не меняется.

### 6.3. Архивировать ученика

Кто может:

- админ/директор;
- преподаватель-одиночка для своих учеников;
- пользователь с `students:write`.

Результат:

- ученик получает статус `archived`;
- прошлые группы, уроки, прогресс, задания, материалы и оплаты сохраняются.

### 6.4. Изменить свои контактные данные

Кто может:

- ученик для себя.

Данные:

- телефон;
- email;
- дополнительные контактные данные, если они разрешены в интерфейсе.

Результат:

- обновляются только контактные данные ученика;
- учебная история не меняется;
- роли, группы, прогресс, посещаемость, задания, материалы и оплата не
  меняются.

## 7. Индивидуальное обучение

### 7.1. Назначить индивидуальное обучение

Кто может:

- админ/директор;

- пользователь с `students:write`.

Данные:

- ученик;
- курс;
- преподаватель;
- расписание, если известно.

Результат:

- создается `IndividualEnrollment`;
- ученик получает индивидуальный учебный контекст.

### 7.2. Завершить индивидуальное обучение

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с `students:write`.

Результат:

- индивидуальное обучение получает статус `completed` или `archived`;
- история индивидуального журнала сохраняется.

## 8. Расписание и занятия

### 8.1. Создать повторяющееся расписание

Кто может:

- админ/директор;
- преподаватель-одиночка;
- пользователь с административным правом на расписание.

Данные:

- группа;
- дни недели;
- время начала;
- время окончания;
- период действия;
- таймзона, если она отличается от таймзоны организации.

Результат:

- создаются `ScheduleRule`;
- будущие занятия могут быть созданы по этим правилам.

Правила:

- в MPMF 1.0 базовая таймзона организации - `Europe/Moscow`, если другое значение
  не задано явно;
- правило расписания можно задать на весь срок курса, например на год;
- `ends_on` может быть пустым, если дата завершения курса или расписания еще
  неизвестна;
- изменение правила расписания не переписывает уже прошедшие уроки;
- будущие уроки можно пересоздать или оставить вручную, но
  система не должна терять историю.

### 8.2. Создать занятия по расписанию

Кто может:

- система автоматически;
- админ/директор;
- преподаватель-одиночка.

Результат:

- создаются `Lesson` на конкретные даты.

Правила:

- в MPMF 1.0 достаточно создавать ближайшие уроки на один календарный месяц;
- один календарный месяц - это окно материализации уроков, а не максимальная
  длительность повторяющегося расписания;
- повторный запуск генерации должен быть безопасным и не создавать дубликаты;
- дубликатом считается урок с тем же `organization_id`, `schedule_rule_id` и
  тем же временем начала;
- генерация не создает уроки вне периода действия правила расписания;
- генерация не изменяет прошедшие уроки.

### 8.3. Перенести занятие

Кто может:

- админ/директор;
- преподаватель-одиночка;
- преподаватель школы для своих уроков, если школа разрешает.

Данные:

- новая дата;
- новое время;
- комментарий.

Результат:

- занятие получает новые дату и время;
- старая учебная запись не теряется.

### 8.4. Отменить занятие

Кто может:

- админ/директор;
- преподаватель-одиночка;
- преподаватель школы для своих уроков, если школа разрешает.

Результат:

- записи журнала не удаляются физически, если они уже были созданы.

## 9. Журнал и посещаемость

### 9.1. Открыть журнал группы

Кто может:

- преподаватель своей группы;
- преподаватель-одиночка своей группы;
- админ/директор для просмотра.

Результат:

- возвращается список занятий месяца;
- список учеников;
- отметки по датам;
- признак наличия сохраненной учебной записи по занятию.

### 9.2. Сохранить запись урока

Кто может:

- преподаватель урока;
- преподаватель-одиночка.
- пользователь с максимальным правом `journal:write:any`.

Данные:

- тема;
- комментарий;
- отметки учеников;
- ссылка на прогресс ученика, если преподаватель открыл его из урока;
- домашнее задание;
- материалы.

Результат:

- обновляется `Lesson`;
- создаются или обновляются `JournalEntry`;
- правила, ошибки и уровни прогресса ученика не меняются этим действием;
- учебные данные урока сохраняются без отдельного статуса проведения.

### 9.3. Отметить посещаемость

Кто может:

- преподаватель урока;
- преподаватель-одиночка.
- пользователь с максимальным правом `journal:write:any`.

Данные:

- ученик;
- урок;
- отметка: пусто, `present`, `absent`, `excused`;
- комментарий, если нужен.

Результат:

- обновляется `JournalEntry`;
- пустая отметка означает присутствие только для прошедшего урока.

### 9.4. Исправить отметку посещаемости

Кто может:

- преподаватель урока;
- преподаватель-одиночка.
- пользователь с максимальным правом `journal:write:any`.

Результат:

- отметка обновляется;
- при необходимости сохраняется история изменения.


### 9.5. Получить сводную посещаемость преподавателя

Кто может:

- преподаватель для своих учеников;
- преподаватель-одиночка для своих учеников;
- админ/директор для просмотра по школе.

Данные:

- период: текущий месяц, прошлый месяц или выбранный месяц;
- группа, если нужен фильтр;
- курс, если нужен фильтр;
- фильтр по пропускам, если нужен;
- фильтр по низкой посещаемости, если нужен.

Результат:

- список учеников;
- группа или курс ученика;
- количество прошедших уроков за период;
- количество посещенных уроков;
- количество пропущенных уроков;
- количество уважительных причин;
- процент посещаемости;
- дата последнего урока;
- короткий статус.

Правила:

- преподаватель видит только учеников своих групп;
- ученик не получает этот список;
- будущие уроки не учитываются;
- пустая отметка считается присутствием только для прошедшего урока.

## 10. Прогресс

### 10.1. Сохранить прогресс ученика

Кто может:

- преподаватель своих учеников;
- преподаватель-одиночка.

Данные:

- ученик;
- курс;
- урок, если обновление сделано после конкретного урока;
- правила, добавленные этому ученику;
- уровень по правилу ученика, если он поставлен;
- ошибки, добавленные этому ученику;
- что нужно повторить;
- комментарии;
- видимость для ученика.

Результат:

- создается или обновляется `ProgressRecord`, если нужно сохранить заметку или историю обновления;
- создаются или обновляются `StudentProgressRule`;
- создаются или обновляются `StudentProgressError`;
- ученик видит только открытые ему значения, правила, ошибки и комментарии.

### 10.2. Скрыть комментарий прогресса от ученика

Кто может:

- преподаватель своих учеников;
- преподаватель-одиночка.

Результат:

- комментарий остается виден преподавателю;
- ученик его не видит.

## 11. Домашние задания

### 11.1. Создать домашнее задание

Кто может:

- преподаватель своих групп;
- преподаватель-одиночка.

Данные:

- группа, индивидуальное обучение или конкретный ученик;
- название;
- текст задания;
- срок;
- связанный урок;
- материалы.

Результат:

- создается `Homework`;
- задание становится видно ученику, если оно относится к нему.

### 11.2. Изменить домашнее задание

Кто может:

- автор задания;
- преподаватель, который ведет эту группу или ученика;
- преподаватель-одиночка.

Результат:

- задание обновляется;
- при необходимости сохраняется история изменения.

### 11.3. Отменить или архивировать домашнее задание

Кто может:

- автор задания;
- преподаватель, который ведет эту группу или ученика;
- преподаватель-одиночка.

Результат:

- задание получает статус `cancelled` или `archived`;
- прошлая связь с уроком сохраняется.

## 12. Материалы

### 12.1. Добавить материал

Кто может:

- админ/директор;
- преподаватель своих курсов, групп, уроков и заданий;
- преподаватель-одиночка;
- пользователь с `materials:write`.

Данные:

- название;
- тип: `text` или `link`;
- текст или ссылка;
- область привязки: курс, группа, индивидуальное обучение, конкретный ученик,
  урок или домашнее задание;
- видимость.

Результат:

- создается `Material`;
- материал виден ученикам только при разрешенной видимости.

### 12.2. Изменить материал

Кто может:

- автор материала;
- админ/директор;
- преподаватель-одиночка для своих материалов;
- пользователь с `materials:write`.

Результат:

- материал обновляется;
- прошлые уроки не теряют учебный контекст.

### 12.3. Скрыть материал от учеников

Кто может:

- автор материала;
- админ/директор;
- преподаватель-одиночка;
- пользователь с `materials:write`.

Результат:

- материал получает статус `hidden` или видимость `teacher_only`;
- ученики больше не видят материал в кабинете;
- прошлые связи с уроками и заданиями сохраняются.

### 12.4. Архивировать материал

Кто может:

- автор материала;
- админ/директор;
- преподаватель-одиночка;
- пользователь с `materials:write`.

Результат:

- материал получает статус `archived`;
- материал не показывается в активных списках;
- прошлые уроки и задания сохраняют ссылку на материал.

## 13. Оплата

### 13.1. Создать запись оплаты

Кто может:

- админ/директор;
- преподаватель-одиночка;
- преподаватель школы с `payments:write`.

Данные:

- ученик;
- курс или группа;
- сумма;
- валюта;
- период;
- срок оплаты;
- статус.

Результат:

- создается `Payment`;
- оплата привязывается к конкретному учебному контексту.

### 13.2. Изменить оплату

Кто может:

- админ/директор;
- преподаватель-одиночка;
- преподаватель школы с `payments:write`.

Данные:

- сумма;
- период;
- срок;
- статус;
- комментарий.

Результат:

- обновляется `Payment`;
- создается запись `PaymentHistory`.

### 13.3. Получить список неоплативших

Кто может:

- админ/директор;
- преподаватель-одиночка;
- преподаватель школы для своих учеников.

Результат:

- список учеников со статусом `pending` или `overdue`;
- сумма;
- срок;
- группа;
- контактные данные, если доступны пользователю.

Правила:

- запись со статусом `pending` и прошедшим `due_at` попадает в список
  неоплативших как просроченная для отображения;
- это не меняет сохраненный статус на `overdue`;
- ручная смена статуса на `overdue` выполняется только действием изменения
  оплаты и попадает в `PaymentHistory`.

## 14. Кабинет ученика

### 14.1. Получить данные главного экрана ученика

Кто может:

- ученик для себя;
- преподаватель или админ через свои рабочие области, но не как кабинет ученика.

Результат:

- ближайший урок;
- курсы;
- группы и индивидуальное обучение;
- домашние задания;
- открытые материалы;
- открытый прогресс;
- оплата ученика.

### 14.2. Получить домашние задания ученика

Кто может:

- ученик для себя.

Результат:

- только задания, связанные с этим учеником;
- материалы, открытые ученику.

### 14.3. Получить прогресс ученика

Кто может:

- ученик для себя.

Результат:

- только правила, ошибки, уровни по правилам и комментарии, открытые преподавателем.

### 14.4. Получить оплату ученика

Кто может:

- ученик для себя.

Результат:

- только собственная оплата;
- без внутренних комментариев преподавателя или администратора.

## 15. Административная панель

### 15.1. Получить данные главного экрана администратора

Кто может:

- админ/директор;
- преподаватель-одиночка для своей организации.

Результат:

- количество курсов;
- количество групп;
- количество преподавателей;
- количество учеников;
- ближайшие занятия;
- проблемные группы;
- ученики с частыми пропусками;
- неоплатившие.

### 15.2. Получить проблемные группы

Кто может:

- админ/директор;
- преподаватель-одиночка для своих групп.

Результат:

- группы без расписания;
- группы без преподавателя;
- группы без активных учеников;
- группы с уроками без заполненного журнала.

## 16. Что уточнять дальше

Следующие технические спеки должны уточнить:

- обязательные поля для каждого действия;
- тексты ошибок;
- валидацию форм;
- фильтры и сортировки списков;
- какие действия входят в первый этап разработки;
- какие действия можно отложить после базового MPMF 1.0.
