# MVP Scope

# Состав первого рабочего релиза

## 1. Назначение документа

Этот документ отделяет первый рабочий релиз от всего, что описано в
спецификациях MVP.

В продуктовых и технических спеках описано много возможностей. Первый релиз
должен быть меньше: он должен дать работающий основной путь, который можно
проверить на реальном преподавателе, ученике и администраторе.

Документ отвечает на вопросы:

- что делаем в самом первом рабочем релизе;
- какие страницы обязательны сразу;
- какие действия обязательны сразу;
- что можно отложить, хотя оно уже описано в спеках;
- какой минимальный путь должен заработать первым.

## 2. Главный принцип первого релиза

Первый релиз должен доказать, что система может заменить минимальный бумажный
журнал преподавателя таджвида.

Главный путь:

1. Админ или преподаватель-одиночка создает курс таджвида.
2. Создает группу.
3. Добавляет учеников.
4. Задает расписание.
5. Преподаватель открывает урок.
6. Преподаватель начинает урок.
7. Отмечает посещаемость.
8. Добавляет короткий комментарий или прогресс.
9. Задает домашнее задание.
10. Завершает урок.
11. Ученик видит урок, задание и открытые материалы в кабинете.

Если этот путь работает, первый релиз считается полезным.

## 3. Core MVP

Core MVP - обязательная основа первого релиза.

Входит:

- вход в систему;
- организация через `organization_id`;
- роли `teacher`, `student`, `admin/director`;
- режим преподавателя-одиночки как преподаватель + админ своих данных;
- курс таджвида;
- простой блок прогресса таджвида;
- группы;
- ученики;
- назначение учеников в группу;
- назначение преподавателя на группу;
- повторяющееся расписание;
- создание занятий по расписанию;
- страница урока;
- календарный журнал группы;
- посещаемость;
- отдельная сводная посещаемость преподавателя;
- статус проведения урока;
- `attendance_status`;
- комментарий урока;
- базовый прогресс;
- домашнее задание;
- текстовые материалы и ссылки;
- ручная оплата;
- кабинет ученика;
- базовая админ-панель.

## 4. Teacher MVP

Первый релиз для преподавателя должен закрыть ежедневную работу.

Обязательные страницы:

- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`;
- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/students`;
- `/teacher/students/:studentId`;
- `/teacher/attendance`;
- `/teacher/homework`;
- `/teacher/materials`;
- `/teacher/payments`.

Обязательные действия:

- открыть свою группу;
- открыть календарный журнал группы;
- открыть сводную посещаемость и увидеть список учеников с процентом
  посещаемости и количеством пропусков;
- открыть урок;
- начать урок;
- отметить `absent`;
- отметить `excused`;
- явно поставить `present`, если преподавателю так удобно;
- оставить пустую отметку как присутствие только после подтверждения
  посещаемости или завершения урока;
- добавить комментарий урока;
- зафиксировать опоздание как комментарий;
- заполнить базовый прогресс;
- выбрать, что видит ученик;
- задать домашнее задание группе;
- задать индивидуальное задание ученику внутри группы;
- прикрепить текстовый материал или ссылку;
- сохранить урок;
- завершить урок;
- увидеть оплату своих учеников;
- для преподавателя-одиночки - изменить оплату своих учеников.

Можно отложить:

- расширенную аналитику преподавателя;
- сложные фильтры учеников;
- историю всех изменений журнала;
- несколько преподавателей на одну группу;
- сложные настройки шаблона курса.

## 5. Student MVP

Первый релиз для ученика должен дать простой кабинет.

Обязательные страницы:

- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

Обязательные действия:

- увидеть ближайший урок;
- увидеть свои курсы и группы;
- открыть домашнее задание;
- открыть текстовый материал или ссылку;
- увидеть открытую часть прогресса;
- увидеть посещаемость;
- увидеть подробную историю посещаемости в отдельном разделе;
- увидеть сумму, срок, период и статус оплаты.

Ученик может видеть ближайший запланированный урок до его завершения. Посещаемость
показывается только после подтверждения посещаемости или завершения урока.
Прогресс, домашние задания и материалы показываются только после сохранения и
с учетом настроек видимости.

Ученик в первом релизе не может:

- отправлять выполненное домашнее задание;
- менять учебные данные;
- редактировать прогресс;
- редактировать оплату;
- видеть других учеников;
- видеть внутренние комментарии преподавателя или администратора.

Можно отложить:

- редактирование учеником своих контактных данных;
- историю входов ученика;
- родительский кабинет;
- уведомления ученику.

## 6. Admin MVP

Первый релиз для администратора должен позволить создать основу учебного
процесса и контролировать ее.

Обязательные страницы:

- `/admin`;
- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`;
- `/admin/payments`.

Обязательные действия:

- создать курс таджвида;
- включить простой блок таджвид-прогресса для учеников курса;
- создать группу;
- назначить преподавателя;
- добавить учеников;
- задать расписание;
- открыть группу;
- открыть журнал группы для просмотра;
- создать или изменить оплату;
- увидеть список неоплативших;
- архивировать ученика;
- архивировать группу или курс.

Админ/директор в первом релизе не редактирует:

- учебные записи журнала;
- посещаемость;
- прогресс преподавателя;
- домашние задания преподавателя.

Можно отложить:

- отдельный директорский режим;
- сложную аналитику;
- графики;
- экспорт в Excel;
- гибкую матрицу всех прав;
- детальную активность преподавателей.

## 7. Payment MVP

Оплата в первом релизе - только ручной учет.

Входит:

- создать оплату ученику;
- указать сумму;
- указать валюту;
- указать срок;
- указать период;
- статус `paid`;
- статус `pending`;
- статус `overdue` как ручной статус;
- статус `exempt`;
- изменить оплату вручную;
- показать список неоплативших;
- показать оплату ученику в кабинете.

В MVP система может подсвечивать оплату как просроченную, если статус `pending`
и `due_at` уже прошел, но это не меняет сохраненный статус автоматически.
Автоматическая смена статуса оплаты не входит в MVP.

Не входит:

- онлайн-эквайринг;
- автоматические платежи;
- автоматическое изменение статуса оплаты;
- интеграции с банками;
- автоматические уведомления;
- счета и квитанции;
- сложные тарифы;
- пакеты уроков.

## 8. Materials MVP

Материалы в первом релизе - только текст и ссылки.

Входит:

- создать текстовый материал;
- создать ссылку;
- привязать материал к курсу;
- привязать материал к группе;
- привязать материал к уроку;
- привязать материал к домашнему заданию;
- скрыть материал от ученика;
- архивировать материал;
- показать открытые материалы ученику.

Не входит:

- загрузка файлов;
- PDF;
- аудио;
- видео;
- изображения;
- проверка доступности ссылок;
- единая библиотека повторно используемых материалов.

## 9. Progress MVP

Прогресс в первом релизе должен быть простым.

Входит:

- добавление правил таджвида конкретному ученику;
- добавление ошибок чтения конкретному ученику;
- уровень по правилу ученика: `excellent`, `good`, `satisfactory`, `poor`;
- оценка за урок по 5-балльной или 10-балльной шкале;
- текстовый комментарий;
- поле "что нужно повторить";
- видимость для ученика;
- внутренний комментарий преподавателя.

Не входит:

- сложные rubrics;
- большой фиксированный шаблон категорий таджвида;
- балльные шкалы;
- графики прогресса;
- автоматическая оценка;
- отдельные шаблоны для других предметов.

## 10. Routes первого релиза

Обязательные маршруты первого релиза:

- `/login`;
- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`;
- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/students`;
- `/teacher/students/:studentId`;
- `/teacher/attendance`;
- `/teacher/homework`;
- `/teacher/materials`;
- `/teacher/payments`;
- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`;
- `/admin`;
- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`;
- `/admin/payments`.

Можно отложить:

- отдельные детальные страницы всех материалов;
- отдельные детальные страницы всех домашних заданий;
- страницы курсов преподавателя `/teacher/courses` и `/teacher/courses/:courseId`;
- отдельные страницы индивидуального обучения `/teacher/individual` и
  `/teacher/individual/:enrollmentId`;
- расширенную страницу курса для ученика;
- отдельную страницу курса ученика `/student/courses/:courseId`;
- общий раздел материалов администратора `/admin/materials`;
- расширенную страницу преподавателя;
- отдельный рабочий путь индивидуального обучения, даже если модель данных уже
  содержит подготовленные сущности для него;
- отдельные отчеты и аналитические страницы.

## 11. Минимальный проверяемый сценарий

Минимальный сценарий, который должен работать первым:

1. Админ создает курс таджвида.
2. Админ создает группу.
3. Админ добавляет преподавателя.
4. Админ добавляет учеников.
5. Админ задает расписание.
6. Система создает ближайшее занятие.
7. Преподаватель входит в систему.
8. Преподаватель открывает группу.
9. Преподаватель открывает урок.
10. Преподаватель начинает урок.
11. Преподаватель отмечает одного отсутствующего.
12. Преподаватель подтверждает посещаемость.
13. Преподаватель добавляет комментарий и домашнее задание.
14. Преподаватель добавляет ссылку или текстовый материал.
15. Преподаватель сохраняет урок.
16. Преподаватель завершает урок.
17. Ученик входит в кабинет.
18. Ученик видит ближайший урок.
19. Ученик видит домашнее задание.
20. Ученик видит материал.
21. Ученик видит свою посещаемость.

После этого можно проверять оплату и прогресс как следующие сценарии первого
релиза.

## 12. Что точно не входит в первый релиз

Не входит:

- мобильное приложение;
- онлайн-оплата;
- автоматические уведомления;
- чат;
- видеозвонки;
- загрузка файлов и медиа;
- сертификаты;
- автоматическая проверка домашних заданий;
- тесты с автоматическими баллами;
- сложная аналитика;
- экспорт;
- несколько учебных заведений в интерфейсе;
- отдельный рабочий путь индивидуального обучения;
- маркетинговый сайт.

При этом модель данных уже должна быть подготовлена к будущему расширению,
особенно через `organization_id`, историю, soft-delete и будущие связи
индивидуального обучения без пользовательского интерфейса для этого пути.

## 13. Что делать после первого релиза

После проверки первого релиза можно расширять:

- оплату;
- прогресс;
- отчеты;
- фильтры;
- материалы;
- родительский доступ;
- уведомления;
- поддержку нескольких учебных заведений;
- другие предметы кроме таджвида.
