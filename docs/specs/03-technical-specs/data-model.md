# Data Model

# Техническая модель данных

## 1. Назначение документа

Этот документ фиксирует технические решения по данным проекта.

Он не заменяет Feature Specs, а переводит продуктовые правила в структуру,
которую можно будет использовать при проектировании базы данных, API и
страниц.

## 2. Учебное заведение

Учебное заведение - верхний организационный контекст данных.

В MVP можно считать, что в системе используется одно учебное заведение. Но
модель данных должна быть подготовлена к нескольким учебным заведениям, чтобы
позже не переписывать связи и историю.

Все основные сущности должны быть связаны с учебным заведением через
`organization_id`.

К основным сущностям относятся:

- пользователи и их роли внутри организации;
- курсы;
- группы;
- индивидуальное обучение;
- ученики;
- преподаватели;
- занятия;
- записи журнала;
- посещаемость;
- прогресс;
- домашние задания;
- материалы;
- оплаты.

Если пользователь в будущем относится к нескольким учебным заведениям, его
доступ должен определяться не только ролью пользователя, но и связью с
конкретной организацией.

## 3. История и удаление данных

Учебные данные нельзя физически удалять, если они уже связаны с прошедшим
обучением.

Физическое удаление допустимо только для ошибочно созданных объектов, которые
еще не связаны с учебной историей. В остальных случаях используется
архивирование, скрытие или soft-delete.

Не удаляются физически:

- занятия;
- записи журнала;
- посещаемость;
- прогресс;
- домашние задания, связанные с уроками;
- материалы, связанные с прошлыми уроками или заданиями;
- оплаты и изменения оплаты;
- связи ученика с группой или индивидуальным обучением, если по ним уже есть
  история.

Для таких сущностей нужно хранить технические поля вида:

- `created_at`;
- `updated_at`;
- `archived_at` или `deleted_at`, если объект скрыт или архивирован;
- `created_by`;
- `updated_by`, если это важно для аудита.

## 4. Материалы и прошлые уроки

Материал, связанный с прошлым уроком или домашним заданием, должен сохранять
учебный контекст.

Если материал больше не должен показываться ученикам, его можно скрыть или
архивировать. Но прошлая запись урока не должна терять ссылку на то, какой
материал был задан или использован.

Удаление материала и скрытие материала - разные действия:

- скрытие убирает материал из текущего кабинета ученика;
- архивирование убирает материал из активной работы преподавателя;
- удаление не должно разрушать прошлые уроки, задания и журнал.

## 5. Занятие

Занятие - это конкретный урок в конкретный день и время.

Занятие должно существовать отдельно от отметок учеников. Это важно, потому
что статус самого урока и отметки отдельных учеников отвечают на разные
вопросы:

- статус занятия отвечает на вопрос: был ли урок заполнен преподавателем;
- отметка ученика отвечает на вопрос: что известно по конкретному ученику на
  этом уроке.

## 6. Статус заполнения занятия

У занятия должен быть отдельный технический статус заполнения.

Базовые статусы MVP:

- `not_started` - занятие создано по расписанию, но преподаватель еще не
  заполнял журнал;
- `partially_filled` - в занятии есть часть данных, но запись еще не выглядит
  полностью завершенной;
- `filled` - занятие заполнено и может считаться полноценной записью журнала.

Этот статус хранится на уровне занятия, а не на уровне ячейки ученика.

## 7. Связь со статусом посещаемости

Пустая ячейка конкретного ученика в заполненном или частично заполненном
занятии означает присутствие ученика.

Это правило не должно использоваться для определения того, заполнен ли сам
урок. Заполненность урока определяется статусом занятия.

Пример:

- занятие имеет статус `not_started`;
- в ячейках учеников нет отметок;
- это не означает, что все ученики присутствовали;
- это означает, что преподаватель еще не заполнил урок.

Другой пример:

- занятие имеет статус `filled`;
- у конкретного ученика ячейка пустая;
- это означает, что ученик был на уроке без отдельной проблемы по посещаемости.

## 8. Когда менять статус занятия

Точные правила можно уточнить при проектировании интерфейса, но базовая логика
MVP такая:

- новое занятие из расписания получает `not_started`;
- если преподаватель добавил тему, комментарий, отметку посещаемости, оценку,
  прогресс, домашнее задание или материал, занятие становится
  `partially_filled`;
- если преподаватель явно сохраняет занятие как завершенную запись журнала,
  занятие становится `filled`.

Если в MVP не будет отдельной кнопки завершения урока, статус `filled` можно
выставлять автоматически после сохранения основной записи урока.

## 9. Зачем это нужно

Отдельный статус занятия решает главный конфликт календарного журнала:

- пустая ячейка ученика может означать присутствие;
- полностью незаполненный урок не должен выглядеть как урок, где все
  присутствовали.

Так журнал остается быстрым для преподавателя, но данные остаются понятными для
администратора, ученика и будущей аналитики.
