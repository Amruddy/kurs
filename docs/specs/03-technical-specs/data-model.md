# Data Model

# Техническая модель данных

## 1. Назначение документа

Этот документ фиксирует технические решения по данным проекта.

Он не заменяет Feature Specs, а переводит продуктовые правила в структуру,
которую можно будет использовать при проектировании базы данных, API и
страниц.

## 2. Учебное заведение

Учебное заведение - верхний организационный контекст данных.

В MVP можно считать, что в системе используется одно учебное заведение. Но
модель данных должна быть подготовлена к нескольким учебным заведениям, чтобы
позже не переписывать связи и историю.

Все основные сущности должны быть связаны с учебным заведением через
`organization_id`.

К основным сущностям относятся:

- пользователи и их роли внутри организации;
- курсы;
- группы;
- индивидуальное обучение;
- ученики;
- преподаватели;
- занятия;
- записи журнала;
- посещаемость;
- прогресс;
- домашние задания;
- материалы;
- оплаты.

Если пользователь в будущем относится к нескольким учебным заведениям, его
доступ должен определяться не только ролью пользователя, но и связью с
конкретной организацией.

### 2.1. Organization

`Organization` хранит учебное заведение или рабочее пространство
преподавателя-одиночки.

Минимальные поля:

- `id`;
- `name`;
- `type` - школа, учебный центр, частный преподаватель или другой тип;
- `status` - активна или архивирована;
- `created_at`;
- `updated_at`.

В MVP может существовать одна организация, но остальные сущности все равно
должны ссылаться на `organization_id`.

## 3. Пользователи, роли и права

### 3.1. User

`User` - человек, который входит в систему.

Минимальные поля:

- `id`;
- `name`;
- `email` или другой логин;
- `phone`, если используется;
- `status` - активный, приостановлен, архивный;
- `created_at`;
- `updated_at`.

Пользователь сам по себе не определяет полный доступ. Доступ определяется его
ролями и правами внутри организации.

### 3.2. OrganizationMember

`OrganizationMember` связывает пользователя с учебным заведением.

Минимальные поля:

- `id`;
- `organization_id`;
- `user_id`;
- `roles`;
- `permissions`;
- `status`;
- `created_at`;
- `updated_at`.

Примеры ролей MVP:

- `student`;
- `teacher`;
- `admin`;
- `director`.

Преподаватель-одиночка технически может быть пользователем с ролями `teacher`
и `admin` внутри своей организации.

### 3.3. Permission

`Permission` - отдельное право на конкретное действие.

В MVP не нужна сложная матрица для всех действий, но отдельные права нужны там,
где обычной роли недостаточно.

Первое обязательное отдельное право:

- `payments:write` - право редактировать оплату.

## 4. Курсы

### 4.1. Course

`Course` - учебное направление или предмет.

Минимальные поля:

- `id`;
- `organization_id`;
- `name`;
- `description`;
- `type` - в MVP рабочий тип `tajweed`;
- `format` - `group`, `individual` или `both`;
- `lesson_mark_scale` - `five_point` или `ten_point`, если курс использует оценки за урок;
- `status` - `active`, `archived`;
- `created_by`;
- `created_at`;
- `updated_at`;
- `archived_at`.

В MVP курс может быть создан администратором/директором или
преподавателем-одиночкой.

### 4.2. CourseProgressSettings

`CourseProgressSettings` хранит только факт, что для курса доступен простой прогресс.

Для курса таджвида это не большой фиксированный шаблон категорий.

В MVP настройки курса не хранят общий список правил для всех учеников.

Правила и ошибки хранятся на уровне конкретного ученика.

Минимальные поля:

- `id`;
- `course_id`;
- `name`;
- `is_progress_enabled`;
- `created_at`;
- `updated_at`.

При создании курса список правил не создается.

### 4.3. StudentProgressRule

`StudentProgressRule` - правило таджвида, которое преподаватель решил отслеживать у конкретного ученика.

Минимальные поля:

- `id`;
- `organization_id`;
- `student_id`;
- `course_id`;
- `name`;
- `level` - `excellent`, `good`, `satisfactory`, `poor`, если уровень поставлен;
- `note`;
- `is_visible_to_student`;
- `sort_order`;
- `is_active`;
- `created_at`;
- `updated_at`.

В MVP преподаватель может добавлять ученику новые правила, но не строит сложную рубрику прогресса.

## 5. Ученики

### 5.1. Student

`Student` - учебная карточка ученика внутри организации.

Минимальные поля:

- `id`;
- `organization_id`;
- `user_id`, если у ученика есть вход в личный кабинет;
- `name`;
- `phone`;
- `email`;
- `status` - `active`, `paused`, `archived`;
- `created_at`;
- `updated_at`;
- `archived_at`.

Ученик может существовать без пользовательского аккаунта, если школа пока ведет
его только как учебную запись. Если ученику нужен личный кабинет, его карточка
связывается с `User`.

### 5.2. StudentContact

`StudentContact` нужен для дополнительных контактов.

В MVP родительский аккаунт не входит, но контакт родителя или ответственного
лица может быть добавлен позже без изменения основной карточки ученика.

Минимальные поля:

- `id`;
- `student_id`;
- `name`;
- `relation`;
- `phone`;
- `email`;
- `is_primary`;
- `created_at`;
- `updated_at`.

## 6. Группы

### 6.1. Group

`Group` - учебная группа по конкретному курсу.

Минимальные поля:

- `id`;
- `organization_id`;
- `course_id`;
- `teacher_id`;
- `name`;
- `status` - `recruiting`, `active`, `paused`, `completed`, `archived`;
- `created_at`;
- `updated_at`;
- `archived_at`.

В MVP у группы один основной преподаватель. Несколько преподавателей на группу
можно добавить после MVP.

`teacher_id` указывает на `User.id`. При чтении и изменении группы система
проверяет, что этот пользователь является активным `OrganizationMember` той же
организации с ролью преподавателя. Админ может назначать преподавателя, но сам
становится значением `teacher_id` только если у него также есть роль
преподавателя.

### 6.2. GroupStudent

`GroupStudent` связывает ученика с группой.

Минимальные поля:

- `id`;
- `group_id`;
- `student_id`;
- `status` - `active`, `paused`, `removed`, `completed`;
- `joined_at`;
- `left_at`;
- `created_at`;
- `updated_at`.

Если ученик выбыл из группы, связь не удаляется физически. Она получает
`left_at` и соответствующий статус, чтобы история журнала сохранялась.

## 7. Индивидуальное обучение

### 7.1. IndividualEnrollment

`IndividualEnrollment` - индивидуальное обучение ученика по курсу.

Минимальные поля:

- `id`;
- `organization_id`;
- `course_id`;
- `student_id`;
- `teacher_id`;
- `status` - `active`, `paused`, `completed`, `archived`;
- `created_at`;
- `updated_at`;
- `archived_at`.

Индивидуальное обучение не должно технически притворяться группой. У него
собственная сущность и свой журнал одного ученика.

`teacher_id` указывает на `User.id`. При чтении и изменении индивидуального
обучения система проверяет активное членство преподавателя в той же
организации.

## 8. Расписание и занятия

### 8.1. ScheduleRule

`ScheduleRule` описывает повторяющееся расписание.

Минимальные поля:

- `id`;
- `organization_id`;
- `target_type` - `group` или `individual_enrollment`;
- `target_id`;
- `weekday`;
- `start_time`;
- `end_time`;
- `timezone`;
- `starts_on`;
- `ends_on`;
- `status` - `active`, `archived`;
- `created_at`;
- `updated_at`.

Одна группа может иметь несколько правил расписания, например занятия по
четвергам и воскресеньям в разное время.

В MVP используется таймзона организации. Если у организации не задана отдельная
таймзона, используется `Europe/Moscow`.

Период действия правила расписания может покрывать весь срок курса. Например,
если курс идет год, правило может иметь `starts_on` в начале курса и `ends_on`
в конце курса. Если дата окончания неизвестна, `ends_on` может быть пустым.

Ограничение "30 дней вперед" относится не к сроку действия расписания, а только
к созданию конкретных записей `Lesson`. Система хранит длинное правило
расписания, но материализует ближайшие уроки небольшим окном.

Правило расписания не меняет уже проведенные уроки. При изменении расписания
новые уроки создаются по новому правилу, а старые уроки остаются в истории.

### 8.2. Lesson

`Lesson` - конкретный урок в конкретный день и время.

Минимальные поля:

- `id`;
- `organization_id`;
- `course_id`;
- `group_id`, если это групповое занятие;
- `individual_enrollment_id`, если это индивидуальное занятие;
- `teacher_id`;
- `schedule_rule_id`, если урок создан из расписания;
- `scheduled_at`;
- `starts_at`;
- `ends_at`;
- `topic`;
- `summary`;
- `lesson_status` - `scheduled`, `in_progress`, `completed`, `cancelled`, `moved`;
- `attendance_status` - `not_checked`, `confirmed`;
- `created_at`;
- `updated_at`.

Урок может быть создан из повторяющегося расписания или вручную.

Для защиты от дублей урок, созданный из расписания, должен быть уникален в
рамках `organization_id`, `schedule_rule_id` и даты/времени начала. Повторный
запуск генерации не должен создавать второй урок на тот же слот.

`teacher_id` указывает на `User.id`. Доступ к уроку проверяется через активное
членство пользователя в организации и его связь с группой или индивидуальным
обучением.

## 9. Журнал, посещаемость и прогресс

### 9.1. JournalEntry

`JournalEntry` - учебная запись по ученику в рамках конкретного урока.

Минимальные поля:

- `id`;
- `lesson_id`;
- `student_id`;
- `attendance_mark` - пусто, `present`, `absent`, `excused`;
- `lesson_mark` - оценка или короткая отметка, если используется;
- `teacher_comment`;
- `internal_comment`;
- `is_visible_to_student`;
- `created_at`;
- `updated_at`.

Для группового урока обычно есть записи по ученикам группы. Для индивидуального
урока запись будет одна.

Пустая `attendance_mark` означает присутствие только если посещаемость урока
подтверждена через `attendance_status = confirmed` или если урок имеет `lesson_status = completed`.

### 9.2. ProgressRecord

`ProgressRecord` хранит заметку или историю обновления прогресса ученика.

Текущее состояние правил и ошибок хранится отдельно в `StudentProgressRule` и
`StudentProgressError`. Урок может быть связан с записью прогресса как причина
обновления, но правила не являются темой урока.

Минимальные поля:

- `id`;
- `organization_id`;
- `student_id`;
- `course_id`;
- `lesson_id`, если прогресс связан с уроком;
- `created_by`;
- `repeat_note` - что нужно повторить;
- `student_comment`;
- `internal_comment`;
- `visibility_settings` или отдельные поля видимости;
- `created_at`;
- `updated_at`.

### 9.3. StudentProgressError

`StudentProgressError` хранит ошибку или замечание, которое преподаватель решил отслеживать у конкретного ученика.

Минимальные поля:

- `id`;
- `organization_id`;
- `student_id`;
- `course_id`;
- `name`;
- `note`;
- `is_visible_to_student`;
- `last_progress_record_id`, если ошибка была обновлена через конкретную запись прогресса;
- `is_active`;
- `created_at`;
- `updated_at`.

Если правило или ошибка позже изменились или были отключены у ученика, старая история прогресса должна оставаться читаемой.

## 10. Домашние задания

### 10.1. Homework

`Homework` - домашнее задание для группы, индивидуального ученика или
конкретного ученика внутри группы.

Минимальные поля:

- `id`;
- `organization_id`;
- `course_id`;
- `group_id`, если задание для группы;
- `individual_enrollment_id`, если задание для индивидуального обучения;
- `student_id`, если задание индивидуальное внутри группы;
- `lesson_id`, если задание связано с уроком;
- `title`;
- `description`;
- `due_at`;
- `status` - `active`, `cancelled`, `archived`;
- `created_by`;
- `created_at`;
- `updated_at`;
- `archived_at`.

В MVP ученик не отправляет выполненное домашнее задание через систему.

## 11. Материалы

### 11.1. Material

`Material` - текстовый материал или ссылка.

Минимальные поля:

- `id`;
- `organization_id`;
- `course_id`, если материал относится к курсу;
- `group_id`, если материал относится к группе;
- `individual_enrollment_id`, если материал относится к индивидуальному
  обучению;
- `student_id`, если материал персонально назначен ученику внутри группы;
- `lesson_id`, если материал относится к уроку;
- `homework_id`, если материал относится к заданию;
- `title`;
- `type` - `text`, `link`;
- `content`;
- `url`;
- `visibility` - `teacher_only`, `visible_to_students`;
- `status` - `active`, `hidden`, `archived`;
- `created_by`;
- `created_at`;
- `updated_at`;
- `archived_at`.

Материал должен относиться хотя бы к одному учебному контексту: курсу, группе,
индивидуальному обучению, конкретному ученику, уроку или домашнему заданию.

Файлы, PDF, аудио, видео и изображения не входят в MVP.

## 12. Оплата

### 12.1. Payment

`Payment` - запись ручного учета оплаты.

Минимальные поля:

- `id`;
- `organization_id`;
- `student_id`;
- `course_id`;
- `group_id`, если оплата относится к группе;
- `individual_enrollment_id`, если оплата относится к индивидуальному обучению;
- `amount`;
- `currency`;
- `period_type` - `month`, `course`, `manual`;
- `period_start`;
- `period_end`;
- `due_at`;
- `status` - `paid`, `pending`, `overdue`, `exempt`;
- `comment`;
- `internal_comment`;
- `created_by`;
- `updated_by`;
- `created_at`;
- `updated_at`.

Оплата должна быть привязана к конкретному учебному контексту, чтобы не
смешивать задолженности ученика по разным группам или форматам обучения.

`status` хранится как ручной статус учета. В MVP система не обязана
автоматически менять `pending` на `overdue` после даты `due_at`.

Для списков и подсказок интерфейс может вычислять признак просрочки: если
`status = pending` и `due_at` раньше текущей даты, запись показывается как
требующая внимания. Это вычисление не заменяет сохраненный статус и не создает
запись `PaymentHistory`.

### 12.2. PaymentHistory

`PaymentHistory` хранит изменения оплаты.

Минимальные поля:

- `id`;
- `payment_id`;
- `changed_by`;
- `changed_at`;
- `field`;
- `old_value`;
- `new_value`;
- `comment`.

История нужна, потому что финансовые данные нельзя терять при исправлениях.

## 13. История и удаление данных

Учебные данные нельзя физически удалять, если они уже связаны с прошедшим
обучением.

Физическое удаление допустимо только для ошибочно созданных объектов, которые
еще не связаны с учебной историей. В остальных случаях используется
архивирование, скрытие или soft-delete.

Не удаляются физически:

- занятия;
- записи журнала;
- посещаемость;
- прогресс;
- домашние задания, связанные с уроками;
- материалы, связанные с прошлыми уроками или заданиями;
- оплаты и изменения оплаты;
- связи ученика с группой или индивидуальным обучением, если по ним уже есть
  история.

Для таких сущностей нужно хранить технические поля вида:

- `created_at`;
- `updated_at`;
- `archived_at` или `deleted_at`, если объект скрыт или архивирован;
- `created_by`;
- `updated_by`, если это важно для аудита.

## 14. Материалы и прошлые уроки

Материал, связанный с прошлым уроком или домашним заданием, должен сохранять
учебный контекст.

Если материал больше не должен показываться ученикам, его можно скрыть или
архивировать. Но прошлая запись урока не должна терять ссылку на то, какой
материал был задан или использован.

Физическое удаление материала в MVP не используется. Вместо него есть скрытие и
архивирование:

- скрытие убирает материал из текущего кабинета ученика;
- архивирование убирает материал из активной работы преподавателя;
- прошлые уроки, задания и журнал сохраняют связь с материалом.

## 15. Занятие как событие

Занятие - это конкретный урок в конкретный день и время.

Статус самого занятия должен существовать отдельно от отметок учеников. Это важно, потому что статус урока и отметки отдельных учеников отвечают на разные вопросы:

- `lesson_status` отвечает на вопрос: что произошло с уроком как событием;
- `attendance_status` отвечает на вопрос: подтвердил ли преподаватель посещаемость;
- отметка ученика отвечает на вопрос: что известно по конкретному ученику на этом уроке.

## 16. Статус проведения занятия

У занятия должен быть отдельный статус проведения.

Базовые статусы MVP:

- `scheduled` - урок запланирован, но преподаватель еще не начал его вести;
- `in_progress` - преподаватель открыл урок и указал, что урок начался;
- `completed` - преподаватель завершил урок, значит урок считается проведенным;
- `cancelled` - урок отменен;
- `moved` - урок перенесен на другое время.

Этот статус хранится на уровне занятия, а не на уровне ячейки ученика.

## 17. Связь со статусом посещаемости

Пустая ячейка конкретного ученика означает присутствие только после того, как преподаватель явно подтвердил посещаемость или завершил урок.

Правило MVP:

- если `lesson_status = cancelled` или `moved`, пустые отметки не считаются присутствием;
- если `lesson_status = scheduled` или `in_progress`, пустые отметки учеников не считаются присутствием;
- если `attendance_status = confirmed`, пустая отметка ученика считается присутствием;
- если `lesson_status = completed`, пустая отметка ученика считается присутствием, если нет явной отметки `absent` или `excused`.

Статусы `cancelled` и `moved` имеют приоритет над `attendance_status`: отмененный или перенесенный урок не должен создавать присутствие из пустых отметок.

`attendance_status` нужен, если преподаватель хочет подтвердить посещаемость отдельно от полного завершения урока.

Пример:

- занятие имеет `lesson_status = scheduled`;
- в ячейках учеников нет отметок;
- это не означает, что все ученики присутствовали;
- это означает, что преподаватель еще не начал или не завершил урок.

Другой пример:

- занятие имеет `lesson_status = completed`;
- у конкретного ученика ячейка пустая;
- это означает, что ученик был на уроке без отдельной проблемы по посещаемости.

## 18. Когда менять статус занятия

Базовая логика MVP:

- новое занятие из расписания получает `lesson_status = scheduled` и `attendance_status = not_checked`;
- когда преподаватель нажимает "Начать урок", занятие получает `lesson_status = in_progress`;
- когда преподаватель нажимает "Завершить урок", занятие получает `lesson_status = completed`;
- при подтверждении посещаемости занятие получает `attendance_status = confirmed`;
- отмена урока выставляет `lesson_status = cancelled`;
- перенос урока выставляет `lesson_status = moved` у старой записи или создает новую связанную запись на новую дату.

## 19. Зачем это нужно

Отдельный статус проведения занятия решает главный конфликт календарного журнала:

- пустая ячейка ученика может означать присутствие;
- но только после того, как понятно, что урок действительно проведен или посещаемость подтверждена.

Так журнал остается быстрым для преподавателя, но данные остаются понятными для администратора, ученика и будущей аналитики.
## 20. Главные связи MVP

Базовые связи:

- `Organization` имеет много `User` через `OrganizationMember`;
- `Organization` имеет много `Course`;
- `Course` имеет много `Group`;
- `Course` имеет много `IndividualEnrollment`;
- `Group` имеет много `Student` через `GroupStudent`;
- `Group` имеет много `Lesson`;
- `IndividualEnrollment` имеет много `Lesson`;
- `Lesson` имеет много `JournalEntry`;
- `JournalEntry` относится к одному `Student`;
- `Student` имеет много `ProgressRecord`;
- `Student` имеет много `StudentProgressRule`;
- `Student` имеет много `StudentProgressError`;
- `Homework` может относиться к группе, индивидуальному обучению, ученику или
  уроку;
- `Material` может относиться к курсу, группе, индивидуальному обучению,
  конкретному ученику, уроку или домашнему заданию;
- `Payment` относится к ученику и конкретному учебному контексту.
