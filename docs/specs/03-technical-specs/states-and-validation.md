# States and Validation

# Статусы и валидация

## 1. Назначение документа

Этот документ фиксирует статусы, обязательные поля, пустые состояния и базовые
правила проверки данных в MPMF 1.0.

Он нужен, чтобы формы, страницы, API/actions и база данных одинаково понимали:

- какие значения допустимы;
- какие поля обязательны;
- какие ошибки нужно показывать;
- какие данные нельзя сохранить;
- какие состояния считаются нормальными.

## 2. Общие правила валидации

Все действия должны проверять:

- обязательные поля заполнены;
- объект относится к текущей организации;
- пользователь имеет право выполнить действие;
- связанные объекты существуют;
- связанные объекты не архивированы, если действие требует активный объект;
- даты и периоды не противоречат друг другу;
- история учебных данных не удаляется физически.

Если данные некорректны, система не сохраняет изменения и показывает понятную
ошибку.

## 3. Общие технические статусы

Для большинства сущностей используются общие состояния:

- `active` - объект используется;
- `paused` - объект временно приостановлен;
- `archived` - объект скрыт из активной работы, но история сохраняется.

Не все сущности обязаны иметь все три статуса. Конкретные допустимые статусы
указаны ниже.

## 4. Организация

### 4.1. Статусы организации

Допустимые статусы:

- `active`;
- `archived`.

### 4.2. Обязательные поля организации

Обязательные поля:

- `name`;
- `type`.

Правила:

- организация не удаляется физически, если в ней есть учебная история;
- все основные сущности должны иметь `organization_id`.

## 5. Пользователь и членство в организации

### 5.1. Статусы пользователя

Допустимые статусы:

- `active`;
- `paused`;
- `archived`.

### 5.2. Обязательные поля пользователя

Обязательные поля:

- `name`;
- логин: `email`, `phone` или другой выбранный способ входа.

Правила:

- один пользователь может иметь несколько ролей;
- доступ определяется через организацию, роль и отдельные права;
- ученик может существовать как `Student` без аккаунта `User`.

## 6. Курс

### 6.1. Статусы курса

Допустимые статусы:

- `active`;
- `archived`.

### 6.2. Обязательные поля курса

Обязательные поля:

- `organization_id`;
- `name`;
- `type`;
- `format`;
- `is_progress_enabled` в простых настройках прогресса для курса таджвида.

Допустимые значения `type` в MPMF 1.0:

- `tajweed`.

Допустимые значения `format`:

- `group`;
- `individual`;
- `both`.

Допустимые значения `lesson_mark_scale`:

- `five_point`;
- `ten_point`.

Правила:

- `lesson_mark_scale` может быть пустым, если курс не использует оценку за
  урок;
- курс нельзя физически удалить, если по нему есть группы, ученики, уроки,
  прогресс, задания, материалы или оплаты;
- изменение настроек прогресса курса не должно менять уже сохраненный прогресс учеников.

## 7. Правила и ошибки прогресса ученика

### 7.1. Статусы правила и ошибки

Допустимые состояния:

- `active`;
- `inactive`.

В модели MPMF 1.0 правило и ошибка - две разные сущности:

- `StudentProgressRule`;
- `StudentProgressError`.

Общее поле `type` для них не используется.

Правила:

- правило или ошибку можно отключить у ученика, но старая история должна оставаться читаемой;
- преподаватель добавляет только те правила и ошибки, которые хочет отслеживать у конкретного ученика;
- преподаватель не обязан обновлять правила или ошибки на каждом уроке;
- видимость правила, ошибки, уровня или комментария для ученика должна храниться отдельно.

## 8. Ученик

### 8.1. Статусы ученика

Допустимые статусы:

- `active`;
- `paused`;
- `archived`.

### 8.2. Обязательные поля ученика

Обязательные поля:

- `organization_id`;
- `name`;
- `status`.

Контактные поля в MPMF 1.0 могут быть необязательными, если школа ведет ученика без
личного кабинета.

Правила:

- ученик может быть в нескольких группах;
- ученик может иметь индивидуальное обучение после базового группового MPMF 1.0;
- ученик может иметь аккаунт для личного кабинета, но это не обязательно для
  создания учебной карточки;
- редактирование учеником своих контактных данных откладывается после первого
  рабочего релиза;
- ученик не может менять свои учебные связи, статус, журнал, прогресс,
  домашние задания, материалы или оплату;
- архивирование ученика не удаляет историю.

## 9. Группа

### 9.1. Статусы группы

Допустимые статусы:

- `recruiting`;
- `active`;
- `paused`;
- `completed`;
- `archived`.

### 9.2. Обязательные поля группы

Обязательные поля:

- `organization_id`;
- `course_id`;
- `name`;
- `status`.

Преподаватель может быть назначен позже, если группа еще набирается или
создается администратором.

Правила:

- активная группа должна иметь курс;
- группа может временно не иметь расписания, но должна показываться как
  проблемная в админ-панели;
- группа может временно не иметь преподавателя, но должна показываться как
  проблемная в админ-панели;
- завершенная или архивная группа не должна принимать новых учеников без
  изменения статуса.

## 10. Связь ученика с группой

### 10.1. Статусы связи

Допустимые статусы:

- `active`;
- `paused`;
- `removed`;
- `completed`.

### 10.2. Обязательные поля связи

Обязательные поля:

- `group_id`;
- `student_id`;
- `status`;
- `joined_at`.

Правила:

- ученика нельзя добавить в архивную группу;
- при удалении ученика из активного состава связь не удаляется физически;
- если ученик выбыл, нужно заполнить `left_at`;
- прошлая история журнала сохраняется.

## 11. Индивидуальное обучение

### 11.1. Статусы индивидуального обучения

Допустимые статусы:

- `active`;
- `paused`;
- `completed`;
- `archived`.

### 11.2. Обязательные поля индивидуального обучения

Обязательные поля:

- `organization_id`;
- `course_id`;
- `student_id`;
- `teacher_id`;
- `status`.

Правила:

- индивидуальное обучение не является группой;
- у индивидуального обучения свой журнал одного ученика;
- завершение или архивирование не удаляет историю.

## 12. Расписание

### 12.1. Статусы правила расписания

Допустимые статусы:

- `active`;
- `archived`.

### 12.2. Обязательные поля правила расписания

Обязательные поля:

- `target_type`;
- `target_id`;
- `weekday`;
- `start_time`;
- `timezone`, если не используется таймзона организации;
- `starts_on`.

Правила:

- `target_type` может быть `group` или `individual_enrollment`;
- `ends_on` может быть пустым, если расписание бессрочное;
- `end_time` должен быть позже `start_time`, если используется;
- расписание не должно создавать занятия вне периода действия правила;
- правило расписания может действовать весь срок курса, в том числе год или
  больше;
- в MPMF 1.0 конкретные уроки по расписанию создаются ближайшим окном на один
  календарный месяц;
- один календарный месяц не ограничивает срок действия самого правила расписания;
- повторная генерация не должна создавать дубликаты уроков;
- изменение расписания не должно менять уже проведенные уроки.

## 13. Занятие

### 13.1. Урок без статуса проведения

В MPMF 1.0 урок не имеет отдельного пользовательского статуса проведения. Посещаемость считается по дате урока и отметке в журнале.

### 13.2. Обязательные поля занятия

Обязательные поля:

- `organization_id`;
- `course_id`;
- `teacher_id`;
- дата и время занятия;
- один учебный контекст: `group_id` или `individual_enrollment_id`.

Правила:

- занятие не должно одновременно относиться и к группе, и к индивидуальному обучению;
- изменение расписания не должно удалять уже созданные записи журнала;
- перенос занятия должен сохранять учебные записи и историю изменения.

Видимость для ученика:

- ученик может видеть ближайший запланированный урок;
- посещаемость ученика показывается по прошедшим урокам;
- прогресс, домашнее задание и материалы показываются ученику только если они
  сохранены и открыты для ученика по правилам видимости;
- внутренние комментарии преподавателя и администратора ученику не показываются.

## 14. Журнал и посещаемость

### 14.1. Допустимые отметки посещаемости

Допустимые значения:

- пусто - присутствие для прошедшего урока;
- `present` - явное присутствие;
- `absent` - отсутствовал;
- `excused` - отсутствовал по уважительной причине.

### 14.2. Обязательные поля записи журнала

Обязательные поля:

- `lesson_id`;
- `student_id`.

Правила:

- пустая отметка ученика не означает присутствие для будущего урока;
- пустая отметка ученика означает присутствие для прошедшего урока;
- опоздание фиксируется комментарием, без отдельного статуса MPMF 1.0;
- админ/директор смотрит журнал, но не редактирует учебные записи в MPMF 1.0;
- максимальное редактирование учебных записей возможно только через отдельное
  право `journal:write:any`.

## 15. Прогресс

### 15.1. Уровни прогресса

Допустимые значения уровня по правилу ученика:

- `excellent`;
- `good`;
- `satisfactory`;
- `poor`.

### 15.2. Обязательные поля прогресса

Обязательные поля `ProgressRecord`:

- `organization_id`;
- `student_id`;
- `course_id`;
- `created_by`.

Обязательные поля `StudentProgressRule`, если преподаватель добавляет правило ученику:

- `student_id`;
- `course_id`;
- `name`.

Обязательные поля `StudentProgressError`, если преподаватель добавляет ошибку ученику:

- `student_id`;
- `course_id`;
- `name`.

Правила:

- правила и ошибки относятся к конкретному ученику, а не к уроку или общему шаблону курса;
- урок может быть поводом обновить прогресс, но правило не становится темой урока;
- уровень ставится только по правилу, добавленному этому ученику, и не является обязательным;
- оценка за урок хранится отдельно в журнале и может быть 5-балльной или 10-балльной;
- текстовые комментарии заполняются только если преподаватель решил их оставить;
- ученик видит только значения с открытой видимостью;
- внутренние комментарии преподавателя не показываются ученику.

## 16. Домашнее задание

### 16.1. Статусы домашнего задания

Допустимые статусы:

- `active`;
- `cancelled`;
- `archived`.

### 16.2. Обязательные поля домашнего задания

Обязательные поля:

- `organization_id`;
- `course_id`;
- `title`;
- `description`;
- `created_by`;
- учебный контекст: группа, индивидуальное обучение, конкретный ученик или
  урок.

Правила:

- задание должно относиться хотя бы к одному учебному контексту;
- срок выполнения может быть пустым;
- ученик не отправляет выполненную работу через систему в MPMF 1.0;
- отмененное или архивное задание сохраняет связь с прошлым уроком.

## 17. Материал

### 17.1. Статусы материала

Допустимые статусы:

- `active`;
- `hidden`;
- `archived`.

### 17.2. Типы материала

Допустимые типы MPMF 1.0:

- `text`;
- `link`.

### 17.3. Видимость материала

Допустимые значения:

- `teacher_only`;
- `visible_to_students`.

### 17.4. Обязательные поля материала

Обязательные поля:

- `organization_id`;
- `title`;
- `type`;
- `created_by`;
- учебный контекст: курс, группа, индивидуальное обучение, конкретный ученик,
  урок или домашнее задание.

Дополнительные правила:

- для `text` нужен `content`;
- для `link` нужен `url`;
- ученик видит только материалы `visible_to_students` со статусом `active`;
- материал для индивидуального ученика может быть привязан к
  `individual_enrollment_id`;
- персональный материал ученику внутри группы может быть привязан к
  `student_id`, если этого не хватает через урок или домашнее задание;
- скрытие материала не удаляет связь с прошлыми уроками и заданиями;
- файлы, PDF, аудио, видео и изображения не входят в MPMF 1.0.

## 18. Оплата

### 18.1. Статусы оплаты

Допустимые статусы:

- `paid`;
- `pending`;
- `overdue`;
- `exempt`.

В MPMF 1.0 `overdue` - ручной статус учета. Система может вычислять признак
просрочки для отображения, если `status = pending` и `due_at` уже прошел, но не
меняет сохраненный статус автоматически.

### 18.2. Типы периода оплаты

Допустимые значения:

- `month`;
- `course`;
- `manual`.

### 18.3. Обязательные поля оплаты

Обязательные поля:

- `organization_id`;
- `student_id`;
- `amount`;
- `currency`;
- `period_type`;
- `due_at`;
- `status`;
- учебный контекст: курс или группа.

Правила:

- сумма не может быть отрицательной;
- валюта должна быть задана, если используется сумма;
- оплата должна относиться к конкретному учебному контексту;
- изменение оплаты должно сохраняться в истории;
- автоматическое вычисление просрочки для отображения не считается изменением
  оплаты и не создает историю;
- ученик видит только свою оплату;
- преподаватель школы редактирует оплату только с `payments:write`.

## 19. Минимальные формы MPMF 1.0

Этот раздел фиксирует минимальный состав форм первого рабочего релиза.

Форма может показывать дополнительные необязательные поля, если они уже есть в
спецификациях, но первый проход разработки должен закрыть минимум ниже.

### 19.1. Вход

Минимальные поля:

- выбор тестового пользователя кнопкой dev-входа.

Правила:

- в Stage 0 и Stage 1 используется тестовый вход без пароля;
- регистрация, восстановление пароля и приглашения не входят в MPMF 1.0.

### 19.2. Курс

Минимальные поля:

- название;
- тип курса, в MPMF 1.0 `tajweed`;
- формат: `group`, `individual` или `both`;
- включен ли простой прогресс таджвида;
- шкала оценки за урок, если используется.

### 19.3. Преподаватель

Минимальные поля:

- имя;
- логин: email, телефон или другой выбранный способ входа;
- статус.

Преподаватель должен получить роль `teacher` в текущей организации.

### 19.4. Ученик

Минимальные поля:

- имя;
- статус;
- контактные данные, если ученик получает личный кабинет.

Ученик может быть создан без аккаунта `User`, если личный кабинет пока не
выдается.

### 19.5. Группа

Минимальные поля:

- название;
- курс;
- статус;
- преподаватель, если группа уже готова к занятиям.

Группа может быть создана без расписания, но должна показываться как
проблемная до добавления расписания.

### 19.6. Назначение ученика в группу

Минимальные поля:

- группа;
- ученик;
- дата добавления.

Связь не удаляется физически при выбытии ученика.

### 19.7. Расписание

Минимальные поля:

- учебный контекст: группа.
- день недели;
- время начала;
- время окончания, если используется;
- дата начала действия правила;
- дата окончания действия правила, если расписание не бессрочное.

### 19.8. Урок

Минимальные поля и действия:

- тема, если преподаватель хочет ее заполнить;
- отметки посещаемости;
- комментарий урока;
- домашнее задание и материалы, если они добавляются после урока.

### 19.9. Домашнее задание

Минимальные поля:

- название;
- описание;
- учебный контекст;
- срок, если он нужен;
- связанный урок, если задание создано после урока.

### 19.10. Материал

Минимальные поля:

- название;
- тип: `text` или `link`;
- текст для типа `text`;
- ссылка для типа `link`;
- учебный контекст;
- видимость.

### 19.11. Оплата

Минимальные поля:

- ученик;
- учебный контекст;
- сумма;
- валюта;
- период;
- срок оплаты;
- статус.

Изменение оплаты должно создавать запись истории.

## 20. Пустые состояния

Пустые состояния не являются ошибкой.

Нормальные пустые состояния MPMF 1.0:

- нет курсов;
- нет групп;
- нет учеников;
- нет ближайших занятий;
- журнал еще не заполнен;
- домашнее задание не задано;
- материалов нет;
- прогресс еще не открыт ученику;
- оплата не настроена.

Если у пользователя есть право создать недостающий объект, интерфейс может
показывать действие создания.

Если права нет, интерфейс просто объясняет, что данных пока нет.

## 21. Ошибки

Базовые ошибки MPMF 1.0:

- нет доступа;
- объект не найден;
- обязательное поле не заполнено;
- объект относится к другой организации;
- объект архивирован;
- неверный статус;
- неверная дата или период;
- нет права редактирования;
- действие невозможно, потому что есть учебная история.

Тексты ошибок должны быть понятными пользователю и не раскрывать чужие данные.

Например, если пользователь пытается открыть объект другой организации, система
может показывать "объект не найден", а не объяснять, что объект существует в
другой школе.

## 22. Что уточнять дальше

Следующие спеки должны уточнить:

- точные тексты ошибок;
- фильтры и сортировки;
- какие формы входят в первый этап разработки;
- какие поля можно сделать необязательными в первом релизе;
- какие изменения нужно логировать в истории.
