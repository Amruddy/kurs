# Permissions

# Права доступа

## 1. Назначение документа

Этот документ фиксирует минимальные технические правила доступа для MVP.

Подробная ролевая модель может расширяться позже, но MVP должен иметь понятные
базовые права, чтобы не смешивать данные школы, преподавателя-одиночки и
ученика.

## 2. Базовые принципы

Права доступа зависят от трех вещей:

- пользователь;
- организация;
- конкретное право или роль внутри этой организации.

Один пользователь может иметь несколько ролей. Например, он может быть
преподавателем и администратором одновременно.

Преподаватель-одиночка не является отдельной технической ролью. Это
пользователь, который имеет права преподавателя и административные права на
свои данные.

## 3. Роли MVP

В MVP используются роли:

- `student`;
- `teacher`;
- `admin`;
- `director`.

Администратор и директор в MVP могут быть одной практической ролью:
`admin/director`.

Преподаватель-одиночка технически не является отдельной ролью. Это пользователь
с правами `teacher` и административными правами внутри своей организации.

## 4. Основные отдельные права

Отдельные права нужны там, где роль сама по себе недостаточно точна.

Минимальные отдельные права MVP:

- `payments:write` - редактирование оплаты;
- `courses:write` - создание и редактирование курсов;
- `groups:write` - создание и редактирование групп;
- `students:write` - создание и редактирование учеников;
- `materials:write` - создание и редактирование материалов;
- `admin:access` - доступ к административной области.

В MVP можно не показывать пользователю эти технические названия, но система
должна иметь понятную проверку прав.

## 5. Курсы

| Пользователь | Видит курсы | Создает/редактирует курсы |
|---|---|---|
| Админ/директор | все курсы школы | да |
| Преподаватель школы | свои назначенные курсы | только если есть `courses:write` |
| Преподаватель-одиночка | свои курсы | да |
| Ученик | только свои курсы | нет |

Преподаватель школы обычно работает с курсами, которые создал администратор.
Он может видеть шаблон прогресса курса, но не должен менять школьные курсы без
отдельного права.

## 6. Группы

| Пользователь | Видит группы | Создает/редактирует группы |
|---|---|---|
| Админ/директор | все группы школы | да |
| Преподаватель школы | свои группы | только если есть `groups:write` |
| Преподаватель-одиночка | свои группы | да |
| Ученик | только свои группы | нет |

Преподаватель школы может открывать свои группы, видеть состав, расписание,
журнал и учебные данные. Изменение состава группы или статуса группы в школе
по умолчанию относится к административным действиям.

## 7. Ученики

| Пользователь | Видит учеников | Создает/редактирует учеников |
|---|---|---|
| Админ/директор | всех учеников школы | да |
| Преподаватель школы | учеников своих групп и индивидуального обучения | только если есть `students:write` |
| Преподаватель-одиночка | своих учеников | да |
| Ученик | только себя | ограниченно или нет |

В MVP ученик не редактирует учебные записи. Если нужно разрешить ученику менять
свои контактные данные, это должно быть отдельным правилом в следующих
технических спеках.

## 8. Расписание и занятия

| Пользователь | Видит расписание | Создает/редактирует расписание |
|---|---|---|
| Админ/директор | все расписание школы | да |
| Преподаватель школы | свое расписание | только если есть административное право |
| Преподаватель-одиночка | свое расписание | да |
| Ученик | только свои занятия | нет |

Преподаватель школы может вести конкретный урок в журнале, но изменение
повторяющегося расписания группы в школе по умолчанию считается
административным действием.

## 9. Журнал и посещаемость

| Пользователь | Видит журнал | Редактирует журнал |
|---|---|---|
| Админ/директор | журналы школы | нет |
| Преподаватель школы | свои группы и индивидуальных учеников | да |
| Преподаватель-одиночка | свои группы и индивидуальных учеников | да |
| Ученик | только открытые ему данные | нет |

Админ/директор может смотреть журнал для контроля, но в MVP не редактирует
учебные записи преподавателя.

Редактирование журнала включает:

- отметку посещаемости;
- исправление отметки;
- комментарий урока;
- отметку опоздания как комментарий;
- оценку или короткую запись урока;
- статус заполнения занятия.

## 10. Прогресс

| Пользователь | Видит прогресс | Редактирует прогресс |
|---|---|---|
| Админ/директор | прогресс учеников школы | нет |
| Преподаватель школы | прогресс своих учеников | да |
| Преподаватель-одиночка | прогресс своих учеников | да |
| Ученик | только открытый ему прогресс | нет |

Преподаватель сам выбирает, какие комментарии и категории прогресса видит
ученик.

Внутренние заметки преподавателя не показываются ученику.

## 11. Домашние задания

| Пользователь | Видит домашние задания | Создает/редактирует домашние задания |
|---|---|---|
| Админ/директор | задания школы для контроля | нет |
| Преподаватель школы | задания своих групп и учеников | да |
| Преподаватель-одиночка | свои задания | да |
| Ученик | свои задания | нет |

В MVP ученик не сдает выполненную работу через систему и не меняет статус
домашнего задания.

## 12. Материалы

| Пользователь | Видит материалы | Создает/редактирует материалы |
|---|---|---|
| Админ/директор | материалы школы | да |
| Преподаватель школы | материалы своих курсов, групп, уроков и заданий | да |
| Преподаватель-одиночка | свои материалы | да |
| Ученик | только открытые ему материалы | нет |

Ученик не видит материалы со статусом `teacher_only`, `hidden` или
`archived`.

Связанные с прошлыми уроками материалы не удаляются физически.

## 13. Оплата

| Пользователь | Видит оплату | Редактирует оплату |
|---|---|---|
| Админ/директор | все в школе | да |
| Преподаватель школы | своих учеников | только если есть `payments:write` |
| Преподаватель-одиночка | своих учеников | да |
| Ученик | только свою | нет |

## 14. Право `payments:write`

`payments:write` - отдельное право на редактирование оплаты.

Оно нужно, чтобы преподаватель школы мог видеть оплату своих учеников, но не
менять финансовые данные без явного разрешения школы.

Пользователь с `payments:write` может:

- изменить статус оплаты;
- изменить сумму;
- изменить срок оплаты;
- изменить период оплаты;
- исправить ошибочную запись оплаты.

В MVP история изменений оплаты должна сохраняться или быть подготовлена к
сохранению, потому что финансовые данные нельзя терять при исправлениях.

## 15. Границы видимости оплаты

Админ/директор видит оплату только внутри своей организации.

Преподаватель школы видит оплату только тех учеников, которые относятся к его
группам или индивидуальному обучению.

Преподаватель-одиночка видит и редактирует оплату только своих учеников.

Ученик видит только свою оплату и не видит:

- оплату других учеников;
- внутренние комментарии администратора;
- внутренние комментарии преподавателя, если они не открыты ученику.

## 16. Кабинет ученика

Кабинет ученика показывает только данные самого ученика.

Ученик видит:

- свои курсы;
- свои группы и индивидуальное обучение;
- свои ближайшие занятия;
- свои домашние задания;
- открытые ему материалы;
- свою посещаемость;
- открытый ему прогресс;
- свою оплату.

Ученик не видит:

- других учеников;
- оплату других учеников;
- внутренние заметки преподавателя;
- внутренние комментарии администратора;
- административную панель;
- скрытые материалы;
- скрытые категории или комментарии прогресса.

## 17. Административная область

Административная область доступна пользователям с ролью `admin`, `director` или
правом `admin:access`.

Преподаватель школы не видит административную область без дополнительных прав.

Преподаватель-одиночка видит упрощенную административную область только своих
данных:

- свои курсы;
- свои группы;
- своих учеников;
- свое расписание;
- свои оплаты;
- свои материалы.

## 18. Главное правило проверки доступа

Любое действие должно проверять:

- пользователь вошел в систему;
- пользователь относится к нужной организации;
- пользователь имеет нужную роль или отдельное право;
- объект относится к той же организации;
- для преподавателя объект относится к его группе, индивидуальному обучению или
  его ученикам;
- для ученика объект относится только к нему самому.

Если хотя бы одно условие не выполняется, действие запрещается.

## 19. Что уточнять дальше

В следующих технических спеках нужно уточнить:

- какие права проверяются у каждого API/action;
- какие кнопки скрываются в интерфейсе при отсутствии прав;
- может ли ученик редактировать свои контактные данные;
- кто может менять повторяющееся расписание в школе;
- кто может редактировать материалы уровня курса;
- какие действия требуют истории изменений.
