# Production Auth And Real Accounts

# Настоящая авторизация и реальные аккаунты

## 1. Назначение

Этот документ фиксирует следующий большой блок после `MPMF 1.0`: замену dev-входа на настоящую авторизацию через Supabase Auth и реальные аккаунты пользователей.

`MPMF 1.0` сознательно использовал dev-auth кнопками. Production Auth должен сделать продукт пригодным для реального входа администратора, преподавателя и ученика по своим учетным данным.

## 2. Источники и техническая основа

Техническая основа:

- Supabase Auth;
- server-side auth для Next.js через cookie-based Supabase SSR client;
- Supabase Auth Admin API для серверного создания и приглашения пользователей.

Официальные источники Supabase:

- `https://supabase.com/docs/guides/auth/server-side/creating-a-client`;
- `https://supabase.com/docs/reference/javascript/auth-admin-inviteuserbyemail`;
- `https://supabase.com/docs/reference/javascript/auth-admin-createuser`.

Правила:

- `SUPABASE_SERVICE_ROLE_KEY` используется только на сервере;
- Auth Admin API нельзя вызывать из браузера;
- server code не должен доверять неподтвержденной cookie-сессии без проверки Supabase Auth claims;
- пароли не хранятся в таблицах приложения.

## 3. Основное решение

В production-режиме вход работает через Supabase Auth:

- email;
- пароль;
- восстановление пароля;
- выход из аккаунта;
- cookie-based session для Server Components, Server Actions и Route Handlers.

Публичная самостоятельная регистрация не входит в первый production-auth этап. Аккаунты создаются и приглашаются только через административные действия.

В production используется реальный Supabase invite/recovery flow:

- администратор или преподаватель-одиночка вводит email пользователя;
- система отправляет реальное email-приглашение через Supabase Auth;
- пользователь сам задает пароль из письма приглашения или восстановления;
- администратор, преподаватель-одиночка и интерфейс продукта не задают и не показывают пароль пользователя;
- простые временные пароли допустимы только для dev/seed/smoke-режима и не используются в production.

Dev-auth остается только для локальной разработки и smoke-проверки, если явно включен dev-флагом окружения. В production dev-auth недоступен.

## 4. Связь Supabase Auth и доменной модели

Внутренняя таблица `users` остается доменной карточкой человека внутри продукта.

Supabase Auth user связывается с `users` через отдельное поле:

- `users.auth_user_id`.

Причина:

- текущие связи (`organization_members.user_id`, `groups.teacher_id`, `students.user_id`, `created_by`) уже используют внутренний `users.id`;
- отдельное поле позволяет подключить настоящий auth без переписывания всей учебной истории.

Правило:

- один Supabase Auth user соответствует одному активному `users` профилю внутри текущей модели;
- `users.email` остается рабочим email профиля;
- `users.auth_user_id` используется для поиска профиля после входа.

## 5. Жизненный цикл аккаунта

Минимальные состояния аккаунта:

- `profile_only` - есть карточка пользователя в `users`, но нет auth-доступа;
- `invited` - приглашение отправлено, пользователь еще не завершил вход;
- `active` - пользователь может входить;
- `disabled` - вход отключен, учебная история сохраняется.

В первом этапе допускается хранить эти состояния в `users.auth_status`.

## 6. Создание и приглашение пользователей

В первом production-auth этапе управление auth-доступом выполняется внутри админских разделов преподавателей и учеников. Отдельный общий экран `/admin/users` можно добавить позже, но он не нужен для первого прохода.

Администратор может:

- создать преподавателя как профиль без входа;
- создать ученика как профиль без входа;
- пригласить преподавателя по email;
- пригласить ученика по email;
- повторно отправить приглашение;
- отключить доступ пользователя без удаления учебной истории.

Преподаватель-одиночка может выполнять те же действия только для своей организации, своих учеников и своих учебных контекстов.

При приглашении:

- система создает или использует существующий `users` профиль;
- создает или обновляет `organization_members`;
- для ученика связывает `students.user_id` с `users.id`;
- вызывает Supabase Auth Admin API только на сервере;
- отправляет реальное email-приглашение через Supabase Auth;
- сохраняет `auth_user_id` после создания или обнаружения auth-пользователя;
- переводит `auth_status` в `invited`;
- не создает production-пароль вручную.

## 7. Вход и рабочая область

После входа система:

1. получает подтвержденного Supabase Auth пользователя;
2. находит `users` по `auth_user_id`;
3. находит активные `organization_members`;
4. определяет доступные роли и права;
5. выбирает рабочую область или показывает выбор, если ролей несколько.

Если профиль или активное членство не найдено, пользователь не попадает в рабочие разделы.

## 8. Страницы

Нужно обновить или добавить:

- `/login` - email/password вход, восстановление доступа, dev-вход только в dev-режиме;
- `/auth/callback` - обработка callback/redirect от Supabase Auth;
- `/auth/reset-password` - установка нового пароля после письма восстановления;
- `/profile` - минимальный профиль: имя, email, организация, роли, выход;
- админские карточки преподавателя и ученика - действие приглашения, повторного приглашения или отключения доступа.

## 9. Безопасность

Обязательные правила:

- нельзя открыть чужую страницу подстановкой ID;
- ученик видит только свою `students` карточку и связанные учебные данные;
- преподаватель видит только свои группы и учеников;
- админ видит данные своей организации;
- отключенный пользователь не входит в систему;
- dev-auth не работает в production;
- service role key не попадает в клиентский bundle.

## 10. Smoke-проверка

Новый smoke-flow должен проверять:

- вход админа по email/password;
- вход преподавателя по email/password;
- вход ученика по email/password;
- logout;
- redirect неавторизованного пользователя на `/login`;
- запрет доступа к чужой рабочей области;
- сохранение старого cross-role route smoke уже после настоящего входа.

## 11. Что не входит в первый production-auth этап

- публичная регистрация школы;
- OAuth/social login;
- SSO;
- MFA/passkeys;
- родительские аккаунты;
- multi-organization self-service;
- ручные production-пароли, которые назначает администратор;
- billing/online payment;
- production deploy и домен, если они не выделены отдельным этапом.
