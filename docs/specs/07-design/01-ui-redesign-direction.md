# Направление визуального редизайна

## 1. Назначение документа

Этот документ фиксирует направление визуальной переработки интерфейса после
завершения MVP.

Редизайн не добавляет новую функциональность. Он меняет только:

- визуальную систему;
- компоновку существующих экранов;
- навигацию;
- карточки, таблицы, формы, статусы и пустые состояния;
- адаптивность и визуальную читаемость.

Редизайн не меняет:

- маршруты;
- роли;
- права доступа;
- бизнес-логику;
- модели данных;
- server actions;
- MVP-сценарии;
- набор доступных пользователю действий.

Если для визуального решения кажется нужной новая функция, ее нельзя добавлять в
рамках дизайн-этапа. Ее нужно вынести в отдельную feature spec после дизайна.

## 2. Главный принцип

Выбранный референс используется только как визуальное направление.

Из референса можно брать:

- общий светлый стиль;
- левый sidebar;
- спокойную рабочую область;
- карточки с мягкой тенью;
- плотность отступов;
- аккуратные списки;
- форму главных страниц ролей;
- оформление вкладок, бейджей, кнопок и полей;
- mobile-подход с компактной нижней навигацией.

Из референса нельзя брать как требование:

- новые функции;
- новые данные;
- новые метрики;
- загрузку файлов;
- уведомления;
- онлайн-оплату;
- чат;
- новые пользовательские сценарии;
- новые маршруты.

Все содержимое экранов берется только из текущего MVP. Данные и блоки можно
переставлять свободно, если смысл страницы и существующие действия не меняются.

## 3. Основа направления

Целевое направление: светлый образовательный рабочий интерфейс для школы,
преподавателя и ученика.

Интерфейс должен выглядеть как ежедневная рабочая система, а не как
маркетинговый сайт.

Основные качества:

- чистый светлый фон;
- постоянная боковая навигация на desktop;
- белые рабочие панели;
- мягкие тени;
- тонкие borders;
- спокойные карточки;
- понятные списки и таблицы;
- формы без визуального шума;
- компактные статусы;
- достаточно воздуха, но без пустой декоративности;
- визуальная плотность, подходящая для регулярной работы.

Для `/admin`, `/teacher` и `/student` используется термин `главная страница`.
Английское название для этих экранов в пользовательском интерфейсе и проектных
текстах не используется.

## 4. Референс

Основной референс: первый выбранный визуальный вариант с преподавательской и
ученической рабочей областью.

Ключевые признаки референса:

- слева sidebar с брендом, пользователем, навигацией и выходом внизу;
- справа широкая рабочая область;
- на главной странице есть приветствие и основные рабочие блоки;
- карточки имеют белый фон, тонкий border, мягкую тень и radius среднего
  размера;
- списки уроков, домашних заданий, материалов и активности читаются как
  рабочие списки, а не как декоративные плитки;
- кнопки и бейджи используют цвет только для смысла и акцента;
- страницы уроков используют вкладки и двухколоночную компоновку;
- мобильный вид не является механическим сжатием desktop, а получает
  компактную навигацию и карточный порядок.

Референс задает только визуальную систему. Интерфейс должен оставаться продуктом
`Курс` и соответствовать русскоязычному учебному контексту.

## 5. Границы дизайн-трека

Первый дизайн-трек должен привести существующий MVP к единому визуальному виду.

Входит:

- общий shell приложения;
- sidebar;
- topbar или компактная верхняя зона рабочей области;
- рабочая область страницы;
- главные страницы ролей;
- карточки состояния;
- таблицы;
- формы;
- кнопки;
- вкладки;
- статусы;
- пустые состояния;
- страницы ошибок;
- адаптация desktop и mobile.

Не входит:

- новые графики, если для них нет данных в MVP;
- новые разделы меню;
- новые страницы;
- новые фильтры;
- drag-and-drop;
- загрузка файлов;
- онлайн-оплата;
- уведомления;
- чат;
- родительский доступ;
- новые отчеты;
- темная тема;
- маркетинговая главная страница.

## 6. Роли и приоритет экранов

Редизайн должен охватить все существующие MVP-роли.

Приоритет 1:

- `/login`;
- общий layout;
- `/admin`;
- `/teacher`;
- `/student`.

Приоритет 2:

- `/admin/courses`;
- `/admin/groups`;
- `/admin/students`;
- `/admin/teachers`;
- `/teacher/groups`;
- `/teacher/lessons/:lessonId`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

Приоритет 3:

- детальные страницы курса, группы и ученика;
- журнал группы;
- страницы материалов и домашних заданий преподавателя;
- страницы ошибок и пустые состояния.

## 7. Layout

Desktop layout:

- слева постоянный sidebar;
- sidebar содержит бренд, блок пользователя, навигацию текущей роли и выход;
- справа основная рабочая область;
- верхняя зона рабочей области содержит заголовок, краткое описание и
  контекстные действия, если они уже есть в MVP;
- контент использует широкую рабочую сетку, а не узкую ленточную страницу;
- таблицы получают достаточно ширины;
- основные действия находятся рядом с заголовком страницы или внутри
  контекстного блока;
- карточки не должны вкладываться в другие декоративные карточки без нужды.

Mobile layout:

- постоянный sidebar скрывается;
- основная навигация становится компактной;
- для главных ученических и преподавательских экранов допустима нижняя
  навигация;
- таблицы получают горизонтальную прокрутку или карточный вид;
- основные действия остаются доступными без долгого поиска;
- текст и кнопки не должны пересекаться или обрезаться.

## 8. Цвета

Базовое направление:

- фон приложения: очень светлый серо-голубой или нейтральный;
- рабочие панели: белые;
- основной текст: темный нейтральный;
- вторичный текст: серый;
- borders: светло-серые;
- success: спокойный зеленый;
- warning: янтарный;
- danger: мягкий красный.

Акценты могут зависеть от роли:

- администратор: нейтральный синий или зеленый акцент;
- преподаватель: зеленый акцент;
- ученик: фиолетовый акцент.

Акцентный цвет используется дозированно:

- активный пункт меню;
- primary button;
- выбранная вкладка;
- важный индикатор;
- фокус элемента;
- смысловой статус.

Интерфейс не должен становиться одноцветным. Цвет не должен заменять
структуру, иконку или текст.

## 9. Типографика

Текст должен быть спокойным и рабочим.

Правила:

- крупные заголовки только на главных страницах ролей и важных detail-экранах;
- внутри карточек заголовки компактные;
- таблицы используют плотный, но читаемый размер;
- подписи и meta-текст меньше основного текста;
- letter spacing не используется как декоративный прием;
- русские тексты должны помещаться в кнопки, бейджи и таблицы.

## 10. Компоненты

Нужны единые компоненты или устойчивые паттерны:

- `AppShell`;
- `Sidebar`;
- `Topbar`;
- `PageHeader`;
- `MetricCard` или карточка состояния, если данные уже есть в MVP;
- `Panel`;
- `DataTable`;
- `StatusBadge`;
- `Tabs`;
- `FormField`;
- `EmptyState`;
- `ActionButton`;
- `InlineAction`;
- `SectionHeader`.

Компоненты должны быть простыми. Не нужно вводить сложную дизайн-систему ради
самой дизайн-системы.

## 11. Таблицы и формы

Таблицы остаются главным рабочим паттерном для списков, где нужно сравнение.

Правила:

- четкая шапка;
- читаемые строки;
- умеренная высота строк;
- статус через бейдж;
- основные действия в правой части строки или рядом с заголовком;
- без лишних декоративных карточек вокруг каждой строки, если таблица лучше.

Формы:

- понятные label;
- достаточная ширина полей;
- компактные подсказки;
- ошибки рядом с полем;
- primary action в конце формы;
- cancel/back action менее заметная.

## 12. Главные страницы

Главная страница не должна быть набором случайных карточек.

Каждая роль имеет свой фокус:

- админ: учебная основа, ближайшие уроки, ученики, группы, преподаватели,
  оплата;
- преподаватель: ближайший урок, группы, ученики, журнал, домашние задания,
  материалы;
- ученик: ближайший урок, расписание, домашнее задание, материалы, прогресс,
  посещаемость, оплата.

Карточки допускаются только если они помогают быстро понять состояние или
перейти к существующему действию.

Новые показатели нельзя придумывать ради красивой сетки. Если данных нет в MVP,
блок не добавляется.

## 13. Иконки

Иконки можно использовать в sidebar, кнопках и статусах.

Правила:

- иконки должны быть одного стиля;
- декоративные иконки без функции не нужны;
- кнопка с понятной иконкой должна иметь доступное название;
- если используется библиотека иконок, предпочтительно выбрать одну и не
  смешивать разные наборы.

## 14. Проверка качества

После дизайн-этапа нужно проверить:

- `npm.cmd run lint`;
- `npm.cmd run build`;
- `npm.cmd run smoke:mvp`;
- открытие ключевых страниц всех ролей;
- desktop viewport;
- mobile viewport;
- отсутствие визуальных пересечений;
- читаемость таблиц;
- доступность основных действий;
- сохранение полного MVP smoke-сценария.

## 15. Критерий готовности

Дизайн-трек считается готовым, если существующая функциональность MVP работает
без изменений, а основные страницы выглядят как единый светлый образовательный
рабочий интерфейс с понятной навигацией, карточками, таблицами, формами и
статусами.
