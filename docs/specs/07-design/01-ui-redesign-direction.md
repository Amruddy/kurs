# Направление визуального редизайна

## 1. Назначение документа

Этот документ фиксирует направление визуальной переработки интерфейса после
завершения MVP.

Редизайн не добавляет новую функциональность. Он меняет только:

- визуальную систему;
- компоновку существующих экранов;
- навигацию;
- карточки, таблицы, формы, статусы и пустые состояния;
- адаптивность и визуальную читаемость.

Редизайн не меняет:

- маршруты;
- роли;
- права доступа;
- бизнес-логику;
- модели данных;
- server actions;
- MVP-сценарии;
- набор доступных пользователю действий.

Если для визуального решения кажется нужной новая функция, ее нельзя добавлять в
рамках дизайн-этапа. Ее нужно вынести в отдельную feature spec после дизайна.

## 2. Основа направления

Целевое направление: светлый образовательный SaaS dashboard.

Интерфейс должен выглядеть как рабочая система для школы, преподавателя и
ученика, а не как маркетинговый сайт.

Основные качества:

- чистый светлый фон;
- спокойная боковая навигация;
- мягкий фиолетовый акцент;
- аккуратные белые рабочие панели;
- читаемые таблицы;
- понятные карточки состояния;
- умеренные тени;
- много воздуха, но без пустой декоративности;
- визуальная плотность, подходящая для ежедневной работы.

## 3. Референсы

Референс student dashboard используется как вдохновение для ученического
кабинета:

- крупный спокойный первый экран;
- левое меню;
- карточки ближайших задач;
- календарь или расписание;
- блоки прогресса;
- визуальное ощущение легкости.

Референс TeachFlow используется как более близкая основа для продукта:

- sidebar слева;
- светлая рабочая область;
- dashboard-карточки;
- таблицы курсов, оплат и оценок;
- компактная верхняя панель;
- фиолетовый primary accent;
- чистые формы и фильтры;
- спокойные бейджи статусов.

Референсы задают направление, но не копируются пиксель-в-пиксель. Интерфейс
должен оставаться продуктом `Курс` и соответствовать русскоязычному учебному
контексту.

## 4. Границы первого дизайн-этапа

Первый дизайн-этап должен привести существующий MVP к единому визуальному виду.

Входит:

- общий shell приложения;
- sidebar;
- topbar;
- рабочая область страницы;
- карточки метрик;
- таблицы;
- формы;
- кнопки;
- вкладки;
- статусы;
- пустые состояния;
- страницы ошибок;
- адаптация desktop и mobile.

Не входит:

- новые графики, если для них нет данных в MVP;
- новые разделы меню;
- drag-and-drop;
- загрузка файлов;
- онлайн-оплата;
- уведомления;
- чат;
- родительский доступ;
- новые отчеты;
- темная тема.

## 5. Роли и приоритет экранов

Редизайн должен охватить все существующие MVP-роли.

Приоритет 1:

- `/login`;
- общий layout;
- `/admin`;
- `/teacher`;
- `/student`.

Приоритет 2:

- `/admin/courses`;
- `/admin/groups`;
- `/admin/students`;
- `/admin/teachers`;
- `/teacher/groups`;
- `/teacher/lessons/:lessonId`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

Приоритет 3:

- детальные страницы курса, группы и ученика;
- журнал группы;
- страницы материалов и домашних заданий преподавателя;
- страницы ошибок и пустые состояния.

## 6. Layout

Desktop layout:

- слева постоянный sidebar;
- справа основная рабочая область;
- сверху внутри рабочей области компактная topbar;
- контент ограничен разумной шириной, но не выглядит как узкая ленточная
  страница;
- таблицы получают достаточно ширины;
- основные действия находятся рядом с заголовком страницы или внутри
  контекстного блока.

Mobile layout:

- sidebar не должен занимать экран постоянно;
- навигация сворачивается в компактное меню или нижнюю навигацию;
- таблицы получают горизонтальную прокрутку или карточный вид;
- основные действия остаются доступными без долгого поиска;
- текст и кнопки не должны пересекаться или обрезаться.

## 7. Цвета

Базовое направление:

- фон приложения: очень светлый серо-голубой или нейтральный;
- рабочие панели: белые;
- основной текст: темный нейтральный;
- вторичный текст: серый;
- primary accent: фиолетовый;
- secondary accent: мягкий синий;
- success: спокойный зеленый;
- warning: янтарный;
- danger: мягкий красный;
- borders: светло-серые.

Фиолетовый акцент должен использоваться дозированно:

- активный пункт меню;
- primary button;
- выбранная вкладка;
- важный индикатор;
- фокус элемента.

Интерфейс не должен становиться полностью фиолетовым.

## 8. Типографика

Текст должен быть спокойным и рабочим.

Правила:

- крупные заголовки только на главных dashboard-экранах;
- внутри карточек заголовки компактные;
- таблицы используют плотный, но читаемый размер;
- подписи и meta-текст меньше основного текста;
- letter spacing не используется как декоративный прием;
- русские тексты должны помещаться в кнопки, бейджи и таблицы.

## 9. Компоненты

Нужны единые компоненты:

- `AppShell`;
- `Sidebar`;
- `Topbar`;
- `PageHeader`;
- `MetricCard`;
- `Panel`;
- `DataTable`;
- `StatusBadge`;
- `Tabs`;
- `FormField`;
- `EmptyState`;
- `ActionButton`;
- `InlineAction`;
- `SectionHeader`.

Компоненты должны быть простыми. Не нужно вводить сложную дизайн-систему ради
самой дизайн-системы.

## 10. Таблицы и формы

Таблицы должны стать главным рабочим паттерном для списков.

Правила:

- четкая шапка;
- читаемые строки;
- умеренная высота строк;
- статус через бейдж;
- основные действия в правой части строки или рядом с заголовком;
- без лишних декоративных карточек вокруг каждой строки, если таблица лучше.

Формы:

- понятные label;
- достаточная ширина полей;
- компактные подсказки;
- ошибки рядом с полем;
- primary action в конце формы;
- cancel/back action менее заметная.

## 11. Dashboard-экраны

Dashboard не должен быть набором случайных карточек.

Каждая роль имеет свой главный фокус:

- админ: состояние учебного процесса, ближайшие уроки, ученики, группы, оплата;
- преподаватель: ближайший урок, группы, ученики, журнал, домашние задания;
- ученик: ближайший урок, расписание, домашнее задание, материалы, прогресс,
  оплата.

Карточки допускаются только если они помогают быстро понять состояние или
перейти к действию.

## 12. Иконки

Иконки можно использовать в sidebar, кнопках и статусах.

Правила:

- иконки должны быть одного стиля;
- декоративные иконки без функции не нужны;
- кнопка с понятной иконкой должна иметь доступное название;
- если используется библиотека иконок, предпочтительно выбрать одну и не смешивать
  разные наборы.

## 13. Проверка качества

После дизайн-этапа нужно проверить:

- `npm.cmd run lint`;
- `npm.cmd run build`;
- `npm.cmd run smoke:mvp`;
- открытие ключевых страниц всех ролей;
- desktop viewport;
- mobile viewport;
- отсутствие визуальных пересечений;
- читаемость таблиц;
- доступность основных действий;
- сохранение полного MVP smoke-сценария.

## 14. Критерий готовности

Дизайн-этап считается готовым, если существующая функциональность MVP работает
без изменений, а основные страницы выглядят как единый светлый SaaS dashboard с
понятной навигацией, карточками, таблицами, формами и статусами.
