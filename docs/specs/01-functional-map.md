# Functional Map

# Карта возможностей платформы

## 1. Назначение документа

Этот документ описывает, что пользователи могут делать в системе.

Global Spec отвечает на вопрос: что это за проект и зачем он нужен.

Functional Map отвечает на вопросы:

- что пользователь может получить в системе;
- какие основные действия он выполняет;
- какие пути существуют;
- какие ситуации могут случиться;
- какие функции нужны до технического проектирования.

В этом документе не описываются:

- база данных;
- API;
- конкретные страницы;
- дизайн интерфейса;
- компоненты;
- техническая архитектура.

Главный фокус MVP: платформа для школ таджвида и преподавателей таджвида. В будущем система может расширяться на любые образовательные направления через настройки курса.

## 2. Роли MVP

В MVP есть четыре пользовательских контекста:

- преподаватель;
- ученик;
- администратор/директор;
- преподаватель-одиночка.

Важно: преподаватель-одиночка не является отдельной технической ролью. Это пользователь, который одновременно выполняет функции преподавателя и администратора.

## 3. Преподаватель

Преподаватель - главный пользователь MVP.

Он работает с курсами, группами, индивидуальными учениками, журналом, посещаемостью, прогрессом, домашними заданиями, материалами и оплатой.

### 3.1. Что преподаватель видит после входа

После входа преподаватель должен получить доступ к своей рабочей области:

- доступные курсы;
- группы по курсам;
- индивидуальные ученики;
- ближайшие занятия;
- журнал;
- список учеников, которым нужно напомнить об оплате;
- короткие признаки проблем: пропуски, слабый прогресс, неоплата.

### 3.2. Что преподаватель может делать с курсами

Преподаватель может:

- видеть свои курсы;
- открыть курс таджвида;
- видеть, какие группы относятся к курсу;
- видеть, какие индивидуальные ученики относятся к курсу;
- видеть, что для курса доступен таджвид-прогресс;
- использовать простой блок таджвид-прогресса;
- добавлять правила и ошибки конкретным ученикам.

Преподаватель не должен в MVP строить полноценный конструктор любых предметов.

### 3.3. Что преподаватель может делать с группами

Преподаватель может:

- видеть список своих групп;
- открыть конкретную группу;
- видеть название группы;
- видеть курс группы;
- видеть список учеников;
- видеть расписание группы;
- видеть журнал группы;
- видеть статус группы;
- видеть финансовые статусы учеников по группе.

### 3.4. Что преподаватель может делать с индивидуальными учениками

Преподаватель может:

- видеть список индивидуальных учеников;
- открыть индивидуального ученика;
- видеть курс индивидуального ученика;
- видеть расписание индивидуальных занятий;
- вести журнал индивидуального обучения;
- задавать домашнее задание индивидуальному ученику;
- отмечать прогресс индивидуального ученика;
- видеть оплату индивидуального ученика.

Индивидуальный ученик не должен называться группой в продукте.

### 3.5. Что преподаватель может делать в журнале

Журнал - центральная рабочая зона преподавателя.

Преподаватель может:

- открыть журнал группы;
- открыть журнал индивидуального ученика;
- выбрать месяц;
- видеть занятия за месяц;
- видеть отметки учеников по датам занятий;
- открыть конкретную запись урока;
- внести подробности урока;
- сохранить изменения.

В записи урока могут быть:

- отметка урока: пусто как присутствие, `П`, `Н`, `У` или оценка;
- короткая запись;
- признак прогресса;
- признак домашнего задания;
- заметка преподавателя.

В журнале по каждому уроку должна быть понятная запись: единая отметка урока, комментарий, прогресс или другое обозначение. Оценка за каждый урок не обязательна, но если она ставится, она ставится в том же месте, где отмечается посещаемость. Шкала оценки за урок может быть 5-балльной или 10-балльной.

Для быстрых отметок могут использоваться короткие обозначения, например:

- `П` - присутствовал;
- `Н` - отсутствовал;
- `У` - отсутствовал по уважительной причине;
- оценка или короткий комментарий по уроку.

Опоздание можно указать как комментарий к уроку, без отдельного обязательного обозначения.

Точные обозначения нужно утвердить в Feature Spec журнала и посещаемости.

Отсутствие какой-либо отметки в ячейке урока означает, что ученик был на уроке, только после подтверждения посещаемости или завершения урока преподавателем. Если урок еще не завершен, пустая ячейка не считается присутствием.

### 3.6. Что преподаватель может делать с посещаемостью

Преподаватель может отметить:

- отсутствовал;
- уважительная причина.

Преподаватель может:

- быстро отметить отсутствие;
- указать опоздание в комментарии к уроку;
- убрать ошибочную отметку;
- оставить комментарий к отсутствию;
- видеть историю посещаемости ученика;
- видеть частые пропуски.

### 3.7. Что преподаватель может делать с прогрессом таджвида

Преподаватель может фиксировать прогресс через простой выбор параметров.

В MVP не нужен большой фиксированный шаблон таджвида.

Преподаватель открывает конкретного ученика и ведет:

- правила таджвида, добавленные этому ученику;
- уровень ученика по добавленному правилу;
- ошибки чтения;
- что нужно повторить;
- комментарий;
- видимость для ученика.

Для уровня по правилу ученика используются простые значения:

- отлично;
- хорошо;
- удовлетворительно;
- плохо;

Преподаватель может:

- поставить уровень по правилу ученика;
- изменить уровень;
- оставить текстовый комментарий;
- отметить проблемные зоны;
- отметить, что ученик должен повторить;
- видеть историю прогресса ученика.

Прогресс должен иметь разные уровни видимости:

- часть прогресса видит только преподаватель;
- часть прогресса может видеть ученик;
- ученик видит только открытые правила, уровни, ошибки и комментарии, которые преподаватель считает нужным открыть.

### 3.8. Что преподаватель может делать с домашними заданиями

Преподаватель может:

- создать домашнее задание для группы;
- создать домашнее задание для индивидуального ученика;
- указать текст задания;
- указать срок;
- связать задание с уроком;
- прикрепить материал;
- оставить комментарий;
- изменить задание;
- видеть, какое задание было дано на прошлом уроке.

В MVP ученик не сдает домашнее задание через систему. Домашнее задание нужно только для фиксации: преподаватель указал, ученик увидел, дальше ученик сдает его вживую или на уроке.

### 3.9. Что преподаватель может делать с материалами

Преподаватель может:

- добавить материал к курсу;
- добавить материал к группе;
- добавить материал к конкретному уроку;
- добавить материал к домашнему заданию;
- прикрепить ссылку;
- добавить текстовое описание или текстовый материал;
- видеть список материалов;
- скрыть или удалить материал.

В MVP материалы - это только текстовое описание/текстовый материал и ссылки.

Загрузка файлов, PDF, аудио и других медиа не входит в начальную версию. Это можно добавить позже.

### 3.10. Что преподаватель может делать с оплатой

Преподаватель может:

- видеть статус оплаты учеников;
- видеть сумму оплаты;
- видеть срок оплаты;
- видеть задолженность;
- отметить оплату вручную, если школа выдала ему право редактирования оплаты;
- изменить статус оплаты, если школа выдала ему право редактирования оплаты;
- открыть список тех, кому нужно напомнить;
- фильтровать неоплативших.

Преподаватель в школе по умолчанию видит оплату, но редактирует ее только при наличии отдельного права.

Преподаватель-одиночка редактирует оплату всегда, потому что он одновременно выполняет функции преподавателя и администратора.

В MVP система не отправляет автоматические уведомления. Она только показывает, кому нужно напомнить.

Финальный набор статусов оплаты MVP:

- оплачен;
- ожидает оплаты;
- просрочено;
- освобожден от оплаты.

Период оплаты может быть разным:

- за месяц;
- за курс;
- вручную заданный период.

## 4. Ученик

Ученик использует систему как личный кабинет.

Он должен быстро понимать:

- когда следующий урок;
- что задано;
- какие материалы открыть;
- что у него с посещаемостью;
- какой у него прогресс;
- когда и сколько нужно оплатить.

### 4.1. Что ученик видит после входа

После входа ученик должен получить доступ к своей учебной области:

- свои курсы;
- свои группы или индивидуальное обучение;
- ближайший урок;
- домашнее задание;
- материалы;
- краткий прогресс;
- статус оплаты.

### 4.2. Что ученик может делать с расписанием

Ученик может:

- видеть ближайшие уроки;
- видеть дату и время урока;
- видеть курс;
- видеть группу или индивидуальный формат;
- видеть преподавателя.

### 4.3. Что ученик может делать с домашними заданиями

Ученик может:

- открыть домашнее задание;
- видеть текст задания;
- видеть срок;
- видеть связанный урок;
- открыть прикрепленные материалы;
- видеть комментарий преподавателя.

В MVP ученик не отправляет выполненное домашнее задание через систему.

### 4.4. Что ученик может делать с материалами

Ученик может:

- видеть доступные материалы;
- открыть материал курса;
- открыть материал группы;
- открыть материал домашнего задания;
- открыть ссылку;
- прочитать текстовое описание или текстовый материал.

### 4.5. Что ученик может видеть по прогрессу

Ученик может:

- видеть открытый преподавателем прогресс;
- видеть комментарии преподавателя;
- видеть, что нужно повторить;
- видеть проблемные зоны;
- видеть открытые правила и уровни по ним, если преподаватель открыл их для ученика.

Важно: ученик не должен видеть прогресс как грубую систему оценок. Для таджвида прогресс должен помогать понять, что улучшать.

### 4.6. Что ученик может видеть по оплате

Ученик может:

- видеть сумму ближайшей оплаты;
- видеть срок оплаты;
- видеть статус оплаты;
- видеть задолженность;
- понимать, за какой курс или период нужна оплата.

В MVP ученик не оплачивает внутри системы. Он только видит информацию по оплате.

## 5. Администратор / директор

В MVP администратор и директор могут быть одной ролью.

Администратор/директор отвечает за организацию учебного процесса.

### 5.1. Что администратор/директор видит после входа

После входа администратор/директор должен получить доступ к управлению учебным процессом:

- список курсов;
- список групп;
- список преподавателей;
- список учеников;
- ближайшие занятия;
- общую посещаемость;
- статусы оплаты;
- список неоплативших;
- проблемные группы или учеников.

### 5.2. Что администратор/директор может делать с курсами

Администратор/директор может:

- создать курс;
- использовать простой блок таджвид-прогресса;
- указать название курса;
- указать описание курса;
- выбрать формат: группы, индивидуальное обучение или оба варианта;
- включить простой блок таджвид-прогресса для учеников курса;
- смотреть правила, ошибки и уровни в карточке конкретного ученика;
- архивировать или отключить курс.

### 5.3. Что администратор/директор может делать с группами

Администратор/директор может:

- создать группу;
- выбрать курс;
- назначить преподавателя;
- добавить учеников;
- указать расписание;
- изменить состав группы;
- изменить статус группы;
- открыть журнал группы;
- видеть оплату по группе.

Администратор/директор может смотреть журнал, но не редактирует учебные записи. Редактирование журнала доступно преподавателю.

### 5.4. Что администратор/директор может делать с преподавателями

Администратор/директор может:

- добавить преподавателя;
- назначить преподавателя на группу;
- видеть группы преподавателя;
- видеть нагрузку преподавателя;
- видеть активность преподавателя;
- отключить преподавателя от группы.

### 5.5. Что администратор/директор может делать с учениками

Администратор/директор может:

- добавить ученика;
- открыть карточку ученика;
- назначить ученика в группу;
- назначить ученика на индивидуальное обучение;
- видеть посещаемость ученика;
- видеть прогресс ученика;
- видеть оплату ученика;
- изменить контактные данные ученика;
- убрать ученика из группы;
- архивировать ученика.

### 5.6. Что администратор/директор может делать с оплатой

Администратор/директор может:

- видеть оплату по всем группам;
- видеть оплату по конкретной группе;
- видеть оплату по конкретному ученику;
- видеть сумму;
- видеть срок оплаты;
- видеть задолженность;
- отметить оплату вручную;
- изменить статус оплаты;
- открыть список тех, кому нужно напомнить.

Автоматическая отправка уведомлений не входит в MVP.

### 5.7. Что администратор/директор может видеть по контролю

Администратор/директор может видеть:

- количество курсов;
- количество групп;
- количество учеников;
- количество преподавателей;
- активные группы;
- группы без расписания;
- группы без преподавателя;
- учеников с частыми пропусками;
- учеников с задолженностью;
- общую картину учебного процесса.

## 6. Преподаватель-одиночка

Преподаватель-одиночка - это пользователь, который сам преподает и сам управляет своей учебной организацией.

Он должен иметь возможности преподавателя и базовые возможности администратора.

### 6.1. Что преподаватель-одиночка может делать

Преподаватель-одиночка может:

- создать свой курс;
- включить простой блок таджвид-прогресса;
- создать группу;
- добавить индивидуального ученика;
- добавить учеников;
- задать повторяющееся расписание;
- вести журнал;
- отмечать посещаемость;
- вести прогресс;
- задавать домашние задания;
- добавлять материалы;
- вести статусы оплаты;
- видеть список неоплативших.

### 6.2. Чем он отличается от преподавателя в школе

Преподаватель в школе обычно работает с теми курсами и группами, которые создал администратор.

Преподаватель-одиночка сам создает и организует:

- курс;
- группу;
- индивидуальных учеников;
- расписание;
- оплату.

Он действует как преподаватель и администратор одновременно. Сначала он может создать курс, затем группу или индивидуального ученика, затем расписание и правила оплаты.

Для одного преподавателя путь должен быть коротким и не должен требовать сложных административных действий.

## 7. Основные проблемные ситуации

### 7.1. Ученик отсутствовал

Система должна позволить:

- быстро отметить отсутствие;
- добавить уважительную причину;
- оставить комментарий;
- увидеть частые пропуски.

### 7.2. Ученик опоздал

Система должна позволить:

- указать опоздание в комментарии к уроку;
- оставить дополнительный комментарий;
- видеть историю комментариев, если преподаватель фиксировал опоздания.

### 7.3. Ученик плохо продвигается

Система должна позволить:

- отметить проблемное правило или ошибку;
- поставить уровень по правилу: отлично, хорошо, удовлетворительно или плохо;
- написать, что нужно повторить;
- видеть историю замечаний.

### 7.4. Ученик не оплатил

Система должна позволить:

- увидеть задолженность;
- увидеть срок оплаты;
- открыть список неоплативших;
- отметить оплату вручную после получения оплаты.

### 7.5. У группы изменилось расписание

Система должна позволить:

- изменить повторяющееся расписание;
- сохранить новые даты занятий;
- не потерять старые записи журнала.

Расписание в MVP должно поддерживать повторяющиеся занятия. Например, курс идет три месяца, уроки проходят по четвергам и воскресеньям, а время начала может задаваться отдельно для каждого дня недели.

### 7.6. Преподаватель ведет индивидуального ученика

Система должна позволить:

- открыть индивидуального ученика без группы;
- видеть его расписание;
- вести его журнал;
- задавать домашние задания;
- отмечать прогресс;
- учитывать оплату.

## 8. Границы Functional Map для MVP

В MVP включаем:

- курсы таджвида с простым блоком прогресса;
- группы;
- индивидуальных учеников;
- учеников;
- преподавателей;
- календарный журнал;
- посещаемость;
- прогресс таджвида;
- домашние задания;
- материалы в формате текста и ссылок;
- ручной учет оплаты;
- личный кабинет ученика;
- базовые возможности администратора/директора;
- режим преподавателя-одиночки.

В MVP не включаем:

- полноценный конструктор любых предметов;
- загрузку файлов, PDF, аудио и других медиа;
- автоматические платежи;
- автоматическую отправку уведомлений;
- видеозвонки;
- встроенный чат;
- автоматическую проверку домашних заданий;
- сложную аналитику;
- мобильное приложение.
