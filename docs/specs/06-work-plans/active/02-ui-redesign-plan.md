# UI Redesign Work Plan

# План визуального редизайна MVP

## 0. Текущий статус

- Текущий трек: `UI Redesign: визуальная переработка MVP без изменения функциональности`.
- Статус трека: проектирование.
- Источник направления дизайна: `docs/specs/07-design/01-ui-redesign-direction.md`.
- MVP Stage 0-8 завершены, следующий дизайн-трек начинается от актуального `master`.
- Новый этап редизайна нельзя реализовывать напрямую в `master`.

## 1. Назначение документа

Этот документ переводит дизайн-направление в порядок маленьких проверяемых
этапов разработки.

План не добавляет новые функции. Он отвечает только за визуальную переработку
существующего MVP.

Если в процессе редизайна появляется идея новой функции, она не попадает в этот
план. Ее нужно вынести в отдельную feature spec после завершения дизайн-трека.

## 2. Главные ограничения

В рамках этого плана нельзя менять:

- маршруты;
- роли;
- права доступа;
- Prisma-модели;
- миграции;
- seed как источник новых возможностей;
- server actions ради новой логики;
- MVP smoke-сценарий;
- набор действий на страницах.

Можно менять:

- CSS;
- layout;
- визуальные компоненты;
- порядок существующих блоков на странице, если смысл страницы не меняется;
- подписи, если они остаются в рамках существующего действия;
- адаптивное поведение;
- пустые состояния;
- визуальное отображение статусов.

## 3. Проверяемая цель

После редизайна существующий MVP должен выглядеть как единый светлый
образовательный SaaS dashboard:

- стабильный sidebar;
- компактная topbar;
- аккуратные dashboard-страницы;
- единые карточки;
- читаемые таблицы;
- спокойные формы;
- понятные статусы;
- нормальная мобильная адаптация.

При этом полный smoke-сценарий MVP должен проходить без изменений.

## 4. Не входит в дизайн-трек

Не входит:

- новые аналитические графики;
- новые dashboard-метрики, если их нет в текущих данных;
- новые страницы;
- новые фильтры;
- онлайн-оплата;
- загрузка файлов;
- уведомления;
- чат;
- родительский доступ;
- темная тема;
- смена продуктового названия;
- маркетинговая главная страница.

## 5. Stage UI 0: общий shell и дизайн-токены

### Цель

Создать визуальную основу приложения, не меняя функциональность страниц.

### Входит

- базовые CSS-токены: цвета, spacing, radius, shadow, typography;
- общий фон приложения;
- единая ширина рабочей области;
- базовые стили кнопок;
- базовые стили форм;
- базовые стили таблиц;
- базовые стили карточек и панелей;
- визуальная подготовка для sidebar/topbar.

### Проверка готовности

- `/login`, `/admin`, `/teacher`, `/student` открываются;
- текущая навигация не ломается;
- тексты не обрезаются;
- `npm.cmd run lint` проходит;
- `npm.cmd run build` проходит;
- `npm.cmd run smoke:mvp` проходит.

### Коммит

`Add UI redesign foundation`

## 6. Stage UI 1: sidebar, topbar и общий app layout

### Цель

Заменить текущую верхнюю навигацию на более рабочую структуру: sidebar на
desktop и компактную навигацию на mobile.

### Входит

- `AppShell`;
- `Sidebar`;
- `Topbar`;
- группировка навигации по ролям;
- активное состояние текущего раздела;
- компактный mobile-вариант;
- сохранение всех существующих ссылок.

### Не входит

- новые разделы меню;
- скрытие существующих MVP-разделов;
- новые роли;
- новая auth-логика.

### Проверка готовности

- админ видит админские разделы;
- преподаватель видит преподавательские разделы;
- ученик видит ученические разделы;
- переходы по существующим маршрутам работают;
- mobile viewport не имеет горизонтального развала из-за navigation;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign app shell navigation`

## 7. Stage UI 2: dashboard-экраны ролей

### Цель

Привести главные страницы ролей к единому dashboard-виду.

### Страницы

- `/admin`;
- `/teacher`;
- `/student`.

### Входит

- `PageHeader`;
- `MetricCard`;
- `Panel`;
- визуальный блок ближайшего урока;
- карточки текущего состояния;
- быстрые действия существующих страниц;
- аккуратные пустые состояния.

### Не входит

- новые метрики;
- новые графики;
- новые действия;
- новые данные.

### Проверка готовности

- админ видит курсы, группы, учеников, преподавателей и оплату в существующем
  объеме;
- преподаватель видит ближайший урок и группы;
- ученик видит ближайший урок, домашнее задание, материалы, прогресс и оплату в
  существующем объеме;
- нет визуального перегруза карточками;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign role dashboards`

## 8. Stage UI 3: списки, таблицы и формы администратора

### Цель

Сделать админские рабочие страницы визуально цельными и удобными для повторной
работы.

### Страницы

- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`.

### Входит

- единый вид таблиц;
- единый вид форм;
- section headers;
- status badges;
- компактные действия;
- визуальное оформление блоков оплаты в существующих местах.

### Не входит

- отдельная страница оплаты админа;
- новые фильтры;
- новые формы;
- новые поля.

### Проверка готовности

- админ может пройти существующий путь создания учебной основы;
- формы остаются понятными;
- таблицы читаются на desktop;
- mobile имеет приемлемую прокрутку;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign admin management pages`

## 9. Stage UI 4: рабочие страницы преподавателя

### Цель

Сделать преподавательский путь визуально быстрым: группа, урок, журнал,
посещаемость, домашние задания, материалы и ученики.

### Страницы

- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`;
- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/attendance`;
- `/teacher/students`;
- `/teacher/students/:studentId`;
- `/teacher/homework`;
- `/teacher/materials`.

### Входит

- единый вид урока;
- читаемый журнал;
- компактные отметки посещаемости;
- визуальное разделение прогресса, ДЗ и материалов;
- формы без изменения состава полей;
- рабочие таблицы и списки.

### Не входит

- статус "начать урок" или "завершить урок";
- новые отметки посещаемости;
- загрузка файлов;
- новые отчеты.

### Проверка готовности

- преподаватель открывает группу;
- открывает урок;
- отмечает посещаемость;
- добавляет комментарий, прогресс, домашнее задание и материал;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign teacher workspace`

## 10. Stage UI 5: кабинет ученика

### Цель

Сделать ученический кабинет визуально спокойным и понятным: ближайший урок,
расписание, домашнее задание, материалы, прогресс, посещаемость и оплата.

### Страницы

- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

### Входит

- dashboard-визуал ученического кабинета;
- блок ближайшего урока;
- карточки домашнего задания и материалов;
- понятные статусы оплаты;
- читаемый прогресс;
- аккуратная посещаемость.

### Не входит

- отправка домашнего задания;
- загрузка файлов;
- родительский доступ;
- онлайн-оплата.

### Проверка готовности

- ученик видит ближайший урок;
- видит ДЗ;
- видит материал;
- видит посещаемость;
- видит открытую часть прогресса;
- видит статус оплаты;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign student cabinet`

## 11. Stage UI 6: responsive и финальный visual smoke

### Цель

Проверить весь редизайн на ключевых desktop и mobile viewport без изменения
функциональности.

### Входит

- проверка desktop;
- проверка mobile;
- исправление визуальных пересечений;
- проверка длинных русских текстов;
- проверка таблиц;
- проверка форм;
- финальный smoke.

### Проверка готовности

- `npm.cmd run lint`;
- `npm.cmd run build`;
- `npm.cmd run smoke:mvp`;
- ручной или Codex smoke ключевых страниц:
  - `/login`;
  - `/admin`;
  - `/teacher`;
  - `/student`;
  - `/teacher/lessons/:lessonId`;
  - `/student/payments`;
- нет очевидных визуальных поломок на mobile.

### Коммит

`Complete UI redesign smoke`

## 12. Рекомендуемый порядок коммитов

1. `Add UI redesign foundation`
2. `Redesign app shell navigation`
3. `Redesign role dashboards`
4. `Redesign admin management pages`
5. `Redesign teacher workspace`
6. `Redesign student cabinet`
7. `Complete UI redesign smoke`

Каждый коммит должен соответствовать одному проверяемому дизайн-этапу и не
должен добавлять новую функциональность.
