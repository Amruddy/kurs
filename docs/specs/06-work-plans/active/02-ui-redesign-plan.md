# UI Redesign Work Plan

# План визуального редизайна MVP

## 0. Текущий статус

- Текущий трек: `UI Redesign: визуальная переработка MVP без изменения функциональности`.
- Статус трека: перепланирование по выбранному визуальному референсу.
- Источник направления дизайна: `docs/specs/07-design/01-ui-redesign-direction.md`.
- Выбранный референс используется только как визуальный стиль.
- Для UI 0-6 данные, действия, маршруты и роли берутся только из текущего MVP.
- Публичная главная страница `/` добавлена в MVP как отдельный будущий этап,
  потому что это лицо проекта до входа в систему.
- MVP Stage 0-8 завершены, следующий дизайн-трек начинается от актуального `master`.
- Новый этап редизайна нельзя реализовывать напрямую в `master`.

## 1. Назначение документа

Этот документ переводит дизайн-направление в порядок маленьких проверяемых
этапов разработки.

UI 0-6 не добавляют новые функции. Они отвечают только за визуальную
переработку существующего MVP.

Если в процессе редизайна появляется идея новой функции, она не попадает в этот
план. Ее нужно вынести в отдельную feature spec после завершения дизайн-трека.

Публичная главная страница `/` уже вынесена в отдельное требование и может быть
реализована отдельным stage после приведения рабочих страниц к единому дизайну.

## 2. Главные ограничения

В рамках этого плана нельзя менять:

- маршруты;
- роли;
- права доступа;
- Prisma-модели;
- миграции;
- seed как источник новых возможностей;
- server actions ради новой логики;
- MVP smoke-сценарий;
- набор действий на страницах.

Исключение: отдельный future stage публичной главной страницы может изменить
поведение маршрута `/` с редиректа на `/login` на полноценную публичную страницу.

Можно менять:

- CSS;
- layout;
- визуальные компоненты;
- порядок существующих блоков на странице, если смысл страницы не меняется;
- подписи, если они остаются в рамках существующего действия;
- адаптивное поведение;
- пустые состояния;
- визуальное отображение статусов.

Можно свободно переставлять существующие данные и блоки внутри страницы, если
это помогает сделать интерфейс понятнее и красивее.

## 3. Проверяемая цель

После редизайна существующий MVP должен выглядеть как единый светлый
образовательный рабочий интерфейс:

- стабильный sidebar на desktop;
- аккуратная верхняя зона рабочей области;
- красивые главные страницы ролей;
- единые карточки;
- читаемые таблицы;
- спокойные формы;
- понятные статусы;
- нормальная мобильная адаптация;
- mobile-навигация без горизонтального развала.

При этом полный smoke-сценарий MVP должен проходить без изменений.

## 4. Не входит в дизайн-трек

Не входит:

- новые аналитические графики;
- новые показатели, если их нет в текущих данных;
- новые страницы;
- новые маршруты;
- новые фильтры;
- онлайн-оплата;
- загрузка файлов;
- уведомления;
- чат;
- родительский доступ;
- темная тема;
- смена продуктового названия;
- сложный маркетинговый сайт за пределами публичной главной страницы MVP.

## 5. Stage UI 0: дизайн-токены и базовая визуальная основа

### Цель

Создать визуальную основу приложения по выбранному референсу, не меняя
функциональность страниц.

### Входит

- базовые CSS-токены: цвета, spacing, radius, shadow, typography;
- общий светлый фон приложения;
- белые рабочие панели;
- базовые стили кнопок;
- базовые стили форм;
- базовые стили таблиц;
- базовые стили карточек;
- базовые стили бейджей статусов;
- подготовка role-based accent: админ, преподаватель, ученик.

### Не входит

- новый sidebar;
- перестройка главных страниц;
- изменение маршрутов;
- новые данные.

### Проверка готовности

- `/login`, `/admin`, `/teacher`, `/student` открываются;
- текущая навигация не ломается;
- тексты не обрезаются;
- визуальная основа не добавляет новые действия;
- `npm.cmd run lint` проходит;
- `npm.cmd run build` проходит.

### Коммит

`Add UI redesign foundation`

## 6. Stage UI 1: общий shell и навигация

### Цель

Заменить текущую верхнюю навигацию на рабочую структуру из референса: sidebar на
desktop и компактная навигация на mobile.

### Входит

- `AppShell`;
- `Sidebar`;
- верхняя зона рабочей области;
- бренд в sidebar;
- блок текущего пользователя и роли;
- навигация текущей роли;
- активное состояние текущего раздела;
- выход внизу sidebar;
- compact/mobile navigation;
- сохранение всех существующих ссылок.

### Не входит

- новые разделы меню;
- скрытие существующих MVP-разделов;
- новые роли;
- новая auth-логика;
- новые уведомления.

### Проверка готовности

- админ видит админские разделы;
- преподаватель видит преподавательские разделы;
- ученик видит ученические разделы;
- переходы по существующим маршрутам работают;
- mobile viewport не имеет горизонтального развала из-за navigation;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign app shell navigation`

## 7. Stage UI 2: главные страницы ролей

### Цель

Сделать красивые главные страницы для `/admin`, `/teacher` и `/student` по
визуальному направлению первого референса.

### Страницы

- `/admin`;
- `/teacher`;
- `/student`.

### Входит

- `PageHeader` или аналогичный паттерн заголовка;
- приветствие или понятный заголовок роли;
- карточки состояния только на основе текущих MVP-данных;
- блок ближайшего урока, если данные уже есть на странице;
- блоки существующих быстрых переходов;
- аккуратные списки существующих сущностей;
- пустые состояния;
- role-based accent.

### Роль администратора

Главная страница администратора может показывать только существующие MVP-смыслы:

- учебная основа;
- курсы;
- группы;
- ученики;
- преподаватели;
- ближайшие уроки;
- оплаты в существующем объеме.

### Роль преподавателя

Главная страница преподавателя может показывать только существующие MVP-смыслы:

- ближайший урок;
- группы;
- ученики;
- журнал;
- посещаемость;
- домашние задания;
- материалы.

### Роль ученика

Главная страница ученика может показывать только существующие MVP-смыслы:

- ближайший урок;
- расписание;
- домашнее задание;
- материалы;
- прогресс;
- посещаемость;
- оплата.

### Не входит

- новые показатели;
- новые графики;
- новые действия;
- новые данные;
- новые уведомления.

### Проверка готовности

- админ видит существующую учебную основу в более аккуратной компоновке;
- преподаватель видит существующие рабочие данные в более аккуратной компоновке;
- ученик видит существующие учебные данные в более аккуратной компоновке;
- для `/admin`, `/teacher` и `/student` используется название "главная страница";
- нет визуального перегруза карточками;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign role main pages`

## 8. Stage UI 3: страницы администратора

### Цель

Сделать админские рабочие страницы визуально цельными и удобными для повторной
работы.

### Страницы

- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`.

### Входит

- единый вид таблиц;
- единый вид форм;
- section headers;
- status badges;
- компактные действия;
- визуальное оформление блоков оплаты в существующих местах;
- аккуратная компоновка существующих деталей курса, группы и ученика.

### Не входит

- отдельная новая страница оплаты админа;
- новые фильтры;
- новые формы;
- новые поля;
- новые действия.

### Проверка готовности

- админ может пройти существующий путь создания учебной основы;
- формы остаются понятными;
- таблицы читаются на desktop;
- mobile имеет приемлемую прокрутку или карточную компоновку;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign admin management pages`

## 9. Stage UI 4: рабочие страницы преподавателя

### Цель

Сделать преподавательский путь визуально быстрым и похожим по качеству на
выбранный референс: группа, урок, журнал, посещаемость, домашние задания,
материалы и ученики.

### Страницы

- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`;
- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/attendance`;
- `/teacher/students`;
- `/teacher/students/:studentId`;
- `/teacher/homework`;
- `/teacher/materials`.

### Входит

- единый вид урока;
- вкладки на странице урока, если они не меняют существующие действия;
- двухколоночная компоновка урока на desktop;
- читаемый журнал;
- компактные отметки посещаемости;
- визуальное разделение прогресса, ДЗ и материалов;
- формы без изменения состава полей;
- рабочие таблицы и списки.

### Не входит

- статус "начать урок" или "завершить урок", если он не входит в текущий MVP;
- новые отметки посещаемости;
- загрузка файлов;
- новые отчеты;
- новые уведомления.

### Проверка готовности

- преподаватель открывает группу;
- открывает урок;
- отмечает посещаемость;
- добавляет комментарий, прогресс, домашнее задание и материал;
- существующие действия остались доступны;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign teacher workspace`

## 10. Stage UI 5: страницы ученика

### Цель

Сделать ученический путь спокойным и понятным: главная страница, расписание,
домашнее задание, материалы, прогресс, посещаемость и оплата.

### Страницы

- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

### Входит

- визуал главной страницы ученика по выбранному референсу;
- блок ближайшего урока;
- карточки или списки домашнего задания и материалов;
- понятные статусы оплаты;
- читаемый прогресс;
- аккуратная посещаемость;
- mobile-представление без перегруза.

### Не входит

- отправка домашнего задания;
- загрузка файлов;
- родительский доступ;
- онлайн-оплата;
- новые уведомления.

### Проверка готовности

- ученик видит ближайший урок;
- видит ДЗ;
- видит материал;
- видит посещаемость;
- видит открытую часть прогресса;
- видит статус оплаты;
- smoke-сценарий MVP проходит.

### Коммит

`Redesign student pages`

## 11. Stage UI 6: responsive и финальный visual smoke

### Цель

Проверить весь редизайн на ключевых desktop и mobile viewport без изменения
функциональности.

### Входит

- проверка desktop;
- проверка mobile;
- исправление визуальных пересечений;
- проверка длинных русских текстов;
- проверка таблиц;
- проверка форм;
- проверка главных страниц ролей;
- финальный smoke.

### Проверка готовности

- `npm.cmd run lint`;
- `npm.cmd run build`;
- `npm.cmd run smoke:mvp`;
- ручной или Codex smoke ключевых страниц:
  - `/login`;
  - `/admin`;
  - `/teacher`;
  - `/student`;
  - `/teacher/lessons/:lessonId`;
  - `/student/payments`;
- нет очевидных визуальных поломок на mobile;
- полный MVP-сценарий не изменился.

### Коммит

`Complete UI redesign smoke`

## 12. Stage UI 7: публичная главная страница проекта

### Цель

Сделать `/` полноценной публичной главной страницей проекта. Это лицо продукта
до входа в систему.

### Страница

- `/`.

### Входит

- главный экран с названием продукта и коротким объяснением;
- блок "для кого": школа, преподаватель, ученик;
- блок основных MVP-возможностей:
  - группы;
  - расписание;
  - журнал и посещаемость;
  - прогресс;
  - домашние задания;
  - материалы;
  - ручной учет оплаты;
- понятный переход на `/login`;
- тот же визуальный язык, что и в рабочем интерфейсе;
- desktop и mobile версия без горизонтального развала.

### Не входит

- регистрация;
- тарифы;
- онлайн-оплата;
- блог;
- чат;
- форма заявки;
- новые личные функции;
- новые роли;
- новые интеграции.

### Проверка готовности

- `/` открывается без авторизации;
- `/login` продолжает работать;
- вход админа, преподавателя и ученика не ломается;
- публичная главная выглядит частью того же продукта;
- smoke-сценарий MVP проходит.

### Коммит

`Add public project homepage`

## 13. Рекомендуемый порядок коммитов

1. `Add UI redesign foundation`
2. `Redesign app shell navigation`
3. `Redesign role main pages`
4. `Redesign admin management pages`
5. `Redesign teacher workspace`
6. `Redesign student pages`
7. `Complete UI redesign smoke`
8. `Add public project homepage`

Каждый коммит должен соответствовать одному проверяемому этапу. UI 0-6 не
добавляют новую функциональность; UI 7 добавляет только публичную главную
страницу `/`, уже описанную в отдельной feature spec.

## 14. Рабочий цикл

Цикл для каждого этапа:

1. Обновить `master` из GitHub.
2. Создать отдельную ветку этапа от актуального `master`.
3. Реализовать только этот этап.
4. Запустить автоматические проверки.
5. Выполнить Codex smoke-проверку, если возможно локально.
6. Кратко показать пользователю, что проверено.
7. По явной команде пользователя закоммитить и запушить ветку.
8. Пользователь открывает и мержит pull request.
9. После подтверждения merge вернуться на `master`, обновить его из GitHub и
   только потом начинать следующий этап.

Нельзя начинать следующий этап, пока предыдущий этап не смержен.
