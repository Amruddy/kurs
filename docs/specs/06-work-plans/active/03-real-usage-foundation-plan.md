# Real Usage Foundation Plan

# План перехода к рабочему использованию

## 0. Текущий статус

- Базовый MVP завершен.
- System A UI завершен.
- Следующее направление: подключить Supabase Auth и Supabase PostgreSQL так,
  чтобы системой можно было пользоваться с реальными пользователями.
- Этот план начинается только после merge завершенных UI-веток в `master`.
- Новая реализация каждого stage делается в отдельной ветке от актуального
  `master`.

## 1. Цель

Перевести проект из demo/dev-режима в рабочий режим:

1. Пользователь открывает публичную главную страницу `/`.
2. Нажимает `Войти`.
3. Входит через `/login` по email и паролю.
4. Система определяет его роли в организации.
5. Админ может приглашать преподавателей.
6. Админ или преподаватель-одиночка может выдавать кабинет ученику.
7. Преподаватель и ученик видят только свои данные.

Название продукта для интерфейса, metadata, Vercel и публичной главной:
`Deshar`. В русскоязычном описании можно пояснять смысл как `Дешар`, но в
UI-бренде используется латиница, чтобы название нормально работало в домене,
Vercel-проекте и письмах-приглашениях.

## 2. Главная страница `/`

Публичная главная не должна сужать продукт только до таджвида.

Основной заголовок:

`Система для школ и преподавателей`

Подзаголовок:

`Ведите группы, расписание, уроки, домашние задания, прогресс учеников и оплаты в одном рабочем кабинете.`

Блоки:

- `Для школы`: курсы, группы, преподаватели, ученики и оплаты.
- `Для преподавателя`: уроки, посещаемость, задания, материалы и прогресс.
- `Для ученика`: расписание, домашние задания, материалы, прогресс и статус оплаты.

Кнопка:

- `Войти`

Таджвид остается внутри продукта: в названиях курсов, прогрессе, seed-данных и
учебной логике.

## 3. Архитектурное решение

- База данных: Supabase PostgreSQL.
- Доступ к application-данным: Prisma.
- Авторизация: Supabase Auth.
- Хостинг и выкладка: Vercel.
- Пароли, сессии, приглашения и восстановление доступа хранит Supabase Auth.
- Application `User` хранит профиль человека для учебной системы.
- `OrganizationMember` хранит роли и права внутри организации.
- `Student` остается учебной карточкой и может существовать без кабинета.
- `Student.user_id` заполняется только если ученику нужен вход в систему.

Обязательные переменные окружения для рабочих stage:

- `DATABASE_URL`;
- `DIRECT_URL`, если он нужен для миграций Supabase;
- `NEXT_PUBLIC_SUPABASE_URL`;
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`;
- `SUPABASE_SERVICE_ROLE_KEY`;
- `NEXT_PUBLIC_SITE_URL`.

`SUPABASE_SERVICE_ROLE_KEY` используется только на сервере и не должен попадать
в клиентский код.

### 3.1. Vercel deployment

Рабочая выкладка проекта идет через Vercel.

Правила:

- GitHub-репозиторий подключается к Vercel;
- production и preview deploy выполняются через Vercel;
- все рабочие env-переменные задаются в Vercel Environment Variables;
- секреты Supabase не записываются в код, спеки, README или `.env.example` со
  значениями;
- `NEXT_PUBLIC_SITE_URL` в production должен соответствовать основному
  Vercel-домену или подключенному домену проекта;
- Supabase Auth Site URL и Redirect URLs должны включать production URL
  проекта и нужные Vercel preview/local адреса для проверки;
- email-приглашения преподавателей и учеников должны вести на рабочий домен,
  а не на локальный `localhost`.

Перед рабочей проверкой нужно убедиться, что в Vercel заданы все переменные
окружения и production build проходит без dev-auth как основного входа.

## 4. Что не входит в это направление

Не делаем:

- публичную регистрацию;
- публичный каталог курсов;
- онлайн-оплату;
- чат;
- загрузку файлов;
- родительский кабинет;
- несколько организаций в одном интерфейсе для одного владельца;
- сложный onboarding-конструктор;
- хранение паролей в application-базе;
- переписывание всего UI без необходимости.

## 5. Stage 0: план и спеки

### Цель

Зафиксировать решение до кода.

### Входит

- обновить спеки страниц `/` и `/login`;
- обновить спеки действий авторизации, приглашения преподавателя и кабинета
  ученика;
- обновить модель данных под связь `User` с Supabase Auth;
- обновить правила доступа;
- создать этот поэтапный план.

### Проверка готовности

- изменения только в документации;
- в плане есть порядок внедрения;
- следующий stage можно начать отдельной веткой.

## 6. Stage 1: Supabase DB/Auth foundation

### Цель

Подготовить техническую основу Supabase без изменения основного пользовательского
потока.

### Входит

- установить официальный Supabase SDK для Next.js;
- добавить серверные helpers для Supabase client;
- описать обязательные env-переменные в примере окружения;
- описать, какие env-переменные нужно добавить в Vercel;
- добавить в Prisma `User.auth_user_id` с уникальным индексом;
- создать миграцию PostgreSQL;
- обновить seed или отдельный setup-скрипт для связи application `User` с
  Supabase Auth;
- оставить dev-auth доступным только как локальный fallback, если он явно
  включен.

### Проверка готовности

- `npm run build` проходит;
- Prisma client генерируется;
- миграция применима к PostgreSQL/Supabase;
- dev-сценарий не сломан;
- production build готов к запуску на Vercel;
- в коде нет service role key на клиентской стороне.

## 7. Stage 2: real session и защита маршрутов

### Цель

Заменить основную dev-auth сессию на реальную Supabase Auth сессию.

### Входит

- получить текущего Supabase Auth пользователя на сервере;
- найти application `User` по `auth_user_id`;
- загрузить активные `OrganizationMember`;
- определить доступные рабочие области;
- обновить `requireWorkspace`;
- сохранить логику выбора роли для пользователя с несколькими ролями;
- закрыть `/admin`, `/teacher`, `/student` для пользователей без нужной роли.

### Проверка готовности

- пользователь без сессии попадает на `/login`;
- админ открывает только админскую область;
- преподаватель открывает только преподавательскую область;
- ученик открывает только кабинет ученика;
- пользователь с ролями `admin` и `teacher` может переключать рабочую область;
- чужие данные по-прежнему не видны.

## 8. Stage 3: login/logout через email и пароль

### Цель

Сделать рабочую страницу входа.

### Входит

- форма `/login` с email и паролем;
- вход через Supabase Auth;
- понятные ошибки входа;
- logout;
- корректный redirect после входа;
- dev-кнопки скрыты в рабочем режиме.

### Проверка готовности

- пользователь входит по email/паролю;
- пользователь выходит из системы;
- после logout защищенные страницы недоступны;
- пользователь с одной ролью попадает в свою рабочую область;
- пользователь с несколькими ролями может выбрать рабочую область.

## 9. Stage 4: публичная главная `/`

### Цель

Сделать главный экран сайта, который объясняет продукт и ведет во вход.

### Входит

- новая страница `/` с текстами из этого плана;
- кнопка `Войти`;
- аккуратный UI в стиле System A;
- отсутствие дублей левой навигации;
- отсутствие публичной регистрации и лишних блоков.

### Проверка готовности

- `/` открывается без авторизации;
- `/login` открывается по кнопке;
- вошедший пользователь может продолжить работу;
- страница не выглядит как отдельный рекламный сайт и не обещает функций вне
  MVP.

## 10. Stage 5: приглашение преподавателя

### Цель

Админ добавляет преподавателя, а преподаватель сам задает пароль по письму.

### Входит

- форма на `/admin/teachers`: имя, email, телефон при необходимости;
- создание или приглашение Supabase Auth пользователя;
- создание application `User`;
- создание `OrganizationMember` с ролью `teacher`;
- статус приглашения или понятное состояние ошибки;
- возможность назначить преподавателя на группу после создания.

### Проверка готовности

- админ приглашает преподавателя;
- преподаватель получает приглашение и задает пароль;
- преподаватель входит через `/login`;
- преподаватель видит только свои группы;
- админ видит преподавателя в списке.

## 11. Stage 6: кабинет ученика по приглашению

### Цель

Ученик может оставаться учебной карточкой без входа, но при необходимости
получает кабинет.

### Входит

- при создании или редактировании ученика есть действие `Выдать доступ`;
- email ученика обязателен только для кабинета;
- создание или приглашение Supabase Auth пользователя;
- создание application `User`;
- связь `Student.user_id`;
- `OrganizationMember` с ролью `student`;
- вход ученика через `/login`.

### Проверка готовности

- ученик без кабинета остается обычной учебной записью;
- ученику можно выдать кабинет позже;
- ученик входит и видит только свои уроки, домашние задания, материалы,
  прогресс и оплату;
- ученик не видит админскую и преподавательскую области.

## 12. Stage 7: настройка первого пользователя

### Цель

Сделать понятный путь для первой школы или преподавателя-одиночки.

### Входит

- setup-действие или server-only скрипт для первой организации;
- режим `школа`: первый пользователь получает роль `admin`;
- режим `преподаватель-одиночка`: первый пользователь получает роли `admin` и
  `teacher`;
- создание начальной организации;
- связь Supabase Auth пользователя с application `User`.

### Проверка готовности

- можно создать первую школу и первого админа;
- можно создать преподавателя-одиночку;
- после входа первый пользователь может создать курс, группу, учеников,
  расписание и оплату.

## 13. Stage 8: рабочий smoke-flow

### Цель

Проверить продукт как реальную систему, а не dev-demo.

### Smoke-сценарий

1. Открыть `/`.
2. Перейти на `/login`.
3. Войти как первый админ.
4. Создать курс.
5. Создать преподавателя и отправить приглашение.
6. Преподаватель принимает приглашение и входит.
7. Админ создает ученика.
8. Админ выдает ученику кабинет.
9. Ученик принимает приглашение и входит.
10. Админ создает группу и расписание.
11. Преподаватель видит свою группу и уроки.
12. Преподаватель добавляет домашнее задание, материал и прогресс.
13. Ученик видит только свои данные.
14. Админ создает оплату.
15. Ученик видит статус оплаты.

### Проверка готовности

- smoke-сценарий проходит без dev-auth;
- smoke-сценарий проходит на Vercel production или preview deployment;
- реальные роли работают;
- приглашения работают;
- ссылки из email-приглашений ведут на Vercel-домен проекта;
- Supabase Auth redirects настроены под Vercel;
- service role key не раскрывается клиенту;
- UI остается в единой системе System A.

## 14. Порядок фиксации

Каждый stage:

1. Начинается отдельной веткой от актуального `master`.
2. Реализуется небольшим проверяемым набором изменений.
3. Проходит автоматическую проверку.
4. Проходит базовую smoke-проверку Codex.
5. Показывается пользователю с кратким отчетом.
6. Коммитится и пушится только после команды `фиксируй`.
7. Следующий stage начинается только после merge предыдущего stage и обновления
   `master`.
