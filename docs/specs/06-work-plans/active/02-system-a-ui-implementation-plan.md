# План внедрения System A UI

## 0. Текущий статус

- Выбранное направление дизайна: `System A · Баланс`.
- Источник визуального направления: `docs/design-concepts/symmetry-system-options.html`.
- Спека дизайна обновлена в `docs/specs/04-visual-rules.md`.
- MVP Stage 0-8 завершены.
- Внедрение UI еще не начато.
- Новый код UI нельзя делать напрямую в `master`; нужна отдельная ветка от актуального `master`.

## 1. Цель

Применить единую симметричную дизайн-систему `System A · Баланс` ко всем MVP-страницам проекта.

Главная цель - не просто сделать одну красивую страницу, а привести весь продукт к одному визуальному языку:

- одинаковая навигация;
- одинаковая сетка;
- одинаковые карточки;
- одинаковые таблицы;
- одинаковые кнопки;
- одинаковые статусы;
- одинаковые формы;
- одинаковые пустые состояния;
- разделы из левой навигации не дублируются на главных экранах ролей как
  отдельные кнопки или плитки;
- единое поведение на desktop и mobile.

## 2. Что входит

Входит только то, что уже описано в MVP-спеках.

Рабочие области:

- преподаватель;
- ученик;
- администратор/директор;
- преподаватель-одиночка через рабочую область преподавателя и упрощенную админскую область.

Страницы преподавателя:

- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`;
- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/students`;
- `/teacher/students/:studentId`;
- `/teacher/attendance`;
- `/teacher/homework`;
- `/teacher/materials`;
- `/teacher/payments`.

Страницы ученика:

- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

Страницы администратора:

- `/admin`;
- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`;
- `/admin/payments`.

Общие страницы:

- `/`;
- `/login`;
- `/forbidden`;
- `not-found`;
- `error`.

## 3. Что не входит

Не добавляем:

- чат;
- онлайн-оплату;
- загрузку файлов;
- PDF, аудио, видео и изображения как материалы;
- графики ради графиков;
- сложную аналитику;
- экспорт;
- уведомления;
- публичный маркетинговый сайт;
- новые маршруты, которых нет в MVP;
- отдельный UI индивидуального обучения.

## 4. Дизайн-система

### 4.1. Базовая сетка

Desktop:

- shell с левой навигацией;
- верхняя панель страницы;
- рабочая область на 12 колонок;
- основной верхний блок занимает 8 колонок;
- правый верхний блок занимает 4 колонки;
- четыре карточки показателей занимают всю ширину одинаковыми колонками;
- нижний основной блок занимает 8 колонок;
- нижний правый блок занимает 4 колонки;
- все блоки одного уровня должны начинаться на одной горизонтальной линии.

Mobile:

- левая навигация скрывается;
- используется нижняя компактная навигация;
- рабочая область становится одноколоночной;
- таблицы получают горизонтальную прокрутку или заменяются адаптированным списком там, где таблица неудобна.

### 4.2. Компоненты

Нужно выделить и использовать единообразно:

- `AppShell`;
- `SidebarNav`;
- `MobileNav`;
- `PageHeader`;
- `DashboardGrid`;
- `HeroBlock`;
- `SignalPanel`;
- `MetricCard`;
- `Panel`;
- `ActionPanel`;
- `DataTable`;
- `JournalTable`;
- `StatusBadge`;
- `JournalMark`;
- `Button`;
- `IconButton`;
- `FormField`;
- `EmptyState`;
- `RoleSwitcher`.

Компоненты должны быть простыми и соответствовать текущему стеку проекта.
Новые библиотеки не добавлять без отдельной причины.

### 4.3. Цвета и статусы

Использовать цветовую основу из `04-visual-rules.md`.

Статусы посещаемости:

- `П` - мягкий зеленый;
- `Н` - мягкий красный;
- `У` - янтарный;
- пустая отметка будущего урока - нейтральная.

Статусы оплаты:

- `paid` - зеленый;
- `pending` - янтарный или синий;
- `overdue` - красный;
- `exempt` - нейтральный или мягкий фиолетовый.

Статусы прогресса:

- `excellent` - зеленый;
- `good` - зеленый;
- `satisfactory` - янтарный или синий;
- `poor` - красный.

## 5. Этапы внедрения

### UI Stage 0: подготовка спецификации и плана

Цель:

- зафиксировать выбор `System A · Баланс`;
- зафиксировать правила симметрии;
- создать план внедрения.

Готовность:

- `04-visual-rules.md` содержит выбранное направление;
- активный план создан;
- код приложения не менялся.

### UI Stage 1: фундамент shell и компонентов

Цель:

- создать базовую дизайн-систему без переделки всех страниц сразу.

Входит:

- глобальные CSS tokens;
- базовый app shell;
- светлая левая навигация;
- мобильная нижняя навигация;
- базовые компоненты кнопок, панелей, карточек, статусов и таблиц;
- единые состояния hover, focus, disabled.
- на главных экранах ролей убраны кнопки-переходы, которые повторяют левую навигацию.

Проверка:

- `/login` открывается;
- `/teacher`, `/student`, `/admin` открываются;
- навигация не ломает доступ к существующим страницам;
- mobile viewport не имеет горизонтального переполнения из-за shell.

### UI Stage 2: страницы преподавателя

Цель:

- применить System A к ежедневному рабочему пути преподавателя.

Входит:

- `/teacher`;
- `/teacher/groups`;
- `/teacher/groups/:groupId`;
- `/teacher/groups/:groupId/journal`;
- `/teacher/lessons/:lessonId`;
- `/teacher/students`;
- `/teacher/students/:studentId`;
- `/teacher/attendance`;
- `/teacher/homework`;
- `/teacher/materials`;
- `/teacher/payments`.

Проверка:

- ближайший урок виден сразу;
- журнал сохраняет читаемую таблицу;
- карточки показателей одинаковые;
- правые блоки выровнены с основными;
- формы и таблицы используют общие компоненты;
- mobile-режим пригоден для открытия урока и отметки посещаемости.

### UI Stage 3: страницы ученика

Цель:

- применить System A к кабинету ученика без админских действий и без редактирования учебных данных.

Входит:

- `/student`;
- `/student/schedule`;
- `/student/homework`;
- `/student/materials`;
- `/student/progress`;
- `/student/attendance`;
- `/student/payments`.

Проверка:

- ученик видит ближайший урок;
- домашнее задание и материалы легко найти;
- прогресс показывает только открытую часть;
- оплата только отображается;
- нет кнопок отправки домашнего задания, онлайн-оплаты или редактирования данных.

### UI Stage 4: страницы администратора

Цель:

- применить System A к настройке и контролю учебного процесса.

Входит:

- `/admin`;
- `/admin/courses`;
- `/admin/courses/:courseId`;
- `/admin/groups`;
- `/admin/groups/:groupId`;
- `/admin/students`;
- `/admin/students/:studentId`;
- `/admin/teachers`;
- `/admin/payments`.

Проверка:

- админ быстро видит курсы, группы, учеников, преподавателей и оплату;
- списки и карточки используют общие компоненты;
- проблемы групп и оплат заметны;
- админ не получает UI для редактирования учебных записей преподавателя в журнале.

### UI Stage 5: общие страницы и полировка

Цель:

- привести к единому стилю страницы входа, ошибок, пустых состояний и крайние responsive-сценарии.

Входит:

- `/`;
- `/login`;
- `/forbidden`;
- `not-found`;
- `error`;
- пустые состояния;
- ошибки форм;
- финальная проверка mobile.

Проверка:

- главная входная точка проекта `/` не выглядит как отдельный маркетинговый лендинг;
- если `/` перенаправляет на `/login`, визуально проверяется `/login` как входная страница проекта;
- все страницы выглядят как один продукт;
- нет разнобоя в кнопках, статусах, таблицах и отступах;
- нет обрезанного текста в кнопках и карточках;
- нет лишних функций вне MVP.

## 6. Порядок работы для каждого UI stage

Для каждого этапа:

1. Вернуться на актуальный `master`.
2. Обновить `master` из GitHub.
3. Создать отдельную ветку stage от `master`.
4. Реализовать только scope этого stage.
5. Запустить автоматические проверки.
6. Выполнить локальную smoke-проверку.
7. Кратко описать пользователю, что проверено.
8. Дождаться команды пользователя продолжать.
9. Закоммитить и запушить ветку.

Следующий UI stage начинать только после merge предыдущего.

## 7. Smoke-проверка дизайна

Минимальная проверка после каждого UI stage:

- `npm run build`;
- открыть соответствующие страницы локально;
- проверить desktop ширину около 1440px;
- проверить mobile ширину около 390px;
- проверить навигацию;
- проверить hover/focus/disabled состояния;
- проверить, что таблицы не ломают layout;
- проверить, что статус читается не только цветом.

## 8. Риски

- Если внедрять весь UI одним коммитом, будет трудно проверить регрессии.
- Если сначала не выделить общие компоненты, страницы снова начнут расходиться визуально.
- Если слишком сильно уплотнить таблицы, mobile-сценарии преподавателя станут неудобными.
- Если добавить новые элементы вне спеки, UI начнет обещать функции, которых нет в MVP.

## 9. Решение

Дальше внедрять дизайн только по этому плану.

Первый кодовый этап после утверждения пользователя:

`UI Stage 1: фундамент shell и компонентов`.
