# Проверка спецификаций перед Stage 0

## 1. Назначение документа

Этот документ нужен перед переходом от спецификаций к технической реализации.

Он фиксирует, что проект идет по правилу:

1. Сначала spec.
2. Потом plan.
3. Потом code.

Если чего-то нет в спецификациях, это не разрабатывается. Если в спецификациях
неясно, сначала уточняется спецификация. Если план противоречит спецификации,
разработка останавливается до исправления документации.

## 2. Карта документов

Перед разработкой Stage 0 должны быть согласованы документы:

- `00-global-spec.md` - кто использует продукт, где и зачем;
- `01-functional-map.md` - какие возможности нужны пользователям;
- `02-feature-specs/` - поведение отдельных функций;
- `03-technical-specs/` - данные, страницы, API/actions, права и валидация;
- `04-visual-rules.md` - общий стиль интерфейса;
- `04-interface-by-role.md` - рабочие области по ролям;
- `05-user-stories.md` - реальные пользовательские сценарии;
- `06-work-plans/` - порядок маленьких проверяемых этапов.

## 3. Что должно быть понятно до кода

Перед Stage 0 должны быть понятны следующие решения.

### 3.1. Язык проекта

- Документация проекта ведется на русском языке.
- Пользовательские тексты интерфейса по умолчанию на русском языке.
- Технические идентификаторы, маршруты, команды, названия таблиц и API остаются
  в техническом формате.

### 3.2. Границы MVP

Должно быть понятно:

- какие страницы входят в первый рабочий релиз;
- какие страницы отложены после MVP;
- какие действия входят в первый рабочий релиз;
- какие действия описаны на будущее, но не реализуются в первом проходе;
- где проходит граница между групповым MVP и индивидуальным обучением.

Особое правило:

- индивидуальное обучение может быть подготовлено в модели данных;
- отдельный рабочий путь индивидуального обучения не реализуется до прохождения
  базового группового smoke-сценария.

### 3.3. Роли и вход

Должно быть понятно:

- какие роли есть в MVP;
- какие тестовые пользователи создаются в seed;
- как пользователь попадает в рабочую область после входа;
- как пользователь с несколькими ролями выбирает рабочую область;
- какие маршруты защищены ролью или правом;
- чем преподаватель-одиночка отличается от преподавателя школы.

Для Stage 0 достаточно dev-auth или тестового входа. Полноценная регистрация,
восстановление пароля и приглашения пользователей не входят в Stage 0.

Решение зафиксировано:

- Stage 0 и Stage 1 используют dev-auth через кнопки на `/login`;
- кнопки входа: "Войти как админ", "Войти как преподаватель", "Войти как
  ученик";
- seed создает организацию, админа, преподавателя и ученика;
- тестовые логины должны быть стабильными;
- пользователь с несколькими ролями выбирает рабочую область.

### 3.4. Минимальные формы

До реализации должны быть понятны минимальные поля форм:

- вход в систему;
- создание курса;
- создание преподавателя;
- создание ученика;
- создание группы;
- назначение ученика в группу;
- назначение преподавателя группе;
- создание расписания;
- открытие и проведение урока;
- создание домашнего задания;
- создание материала;
- создание оплаты.

Если форма не нужна в Stage 0, это должно быть явно указано в плане этапа.

Решение зафиксировано:

- минимальные поля форм описаны в `03-technical-specs/states-and-validation.md`;
- Stage 0 открывает только базовые страницы и не реализует бизнес-формы;
- CRUD-формы начинаются с соответствующих этапов work plan.

### 3.5. Расписание и создание уроков

До реализации расписания должны быть понятны правила:

- какая таймзона используется;
- на какой период вперед создаются ближайшие уроки;
- как система не создает дубликаты уроков при повторной генерации;
- что происходит при изменении правила расписания;
- что входит в MVP, а что откладывается.

Для Stage 0 эти правила могут быть еще не реализованы, но они должны быть
зафиксированы до Stage 3.

Решение зафиксировано:

- базовая таймзона MVP - `Europe/Moscow`, если у организации не задано другое;
- правило расписания можно задать на весь срок курса или без даты окончания;
- конкретные уроки по расписанию создаются ближайшим окном на 30 дней вперед;
- 30 дней не ограничивают срок действия самого расписания;
- повторная генерация не создает дубликаты;
- изменение расписания не переписывает уже проведенные уроки.

### 3.6. Статусы урока и посещаемость

Должно быть понятно:

- чем `lesson_status` отличается от `attendance_status`;
- когда пустая отметка считается присутствием;
- что видит ученик до завершения урока;
- какие данные скрыты от ученика;
- кто может исправлять учебные записи.

Решение зафиксировано:

- ученик может видеть ближайший запланированный урок до завершения;
- посещаемость видна ученику только после подтверждения или завершения урока;
- прогресс, домашние задания и материалы видны только после сохранения и по
  настройкам видимости;
- внутренние комментарии ученику не показываются.

### 3.7. Оплата

Должно быть понятно:

- оплата в MVP ведется вручную;
- онлайн-оплата не входит в MVP;
- кто может создавать и изменять оплату;
- что видит ученик;
- что видит преподаватель;
- является ли `overdue` ручным статусом или вычисляемым признаком.

До реализации оплаты это решение должно быть зафиксировано в технической
спецификации.

Решение зафиксировано:

- `overdue` хранится как ручной статус оплаты;
- `pending` с прошедшим `due_at` может подсвечиваться как просрочка;
- автоматическая смена сохраненного статуса оплаты не входит в MVP.

### 3.8. Пустые состояния и ошибки

Должно быть понятно:

- какие пустые состояния нормальны;
- какие ошибки показываются пользователю;
- какие ошибки не должны раскрывать чужие данные;
- какие действия показываются в пустом состоянии при наличии прав.

## 4. Готовность к Stage 0

Stage 0 можно начинать, если:

- есть актуальный MVP scope;
- есть рабочий plan для Stage 0;
- понятно, какие страницы Stage 0 должны открываться;
- понятно, какие seed-данные нужны;
- понятно, что Stage 0 не делает бизнес-действия;
- понятно, что auth в Stage 0 может быть упрощенной;
- понятно, какие визуальные правила применяются к базовому layout;
- нет противоречий между `mvp-scope.md`, `pages-and-routes.md`,
  `data-model.md`, `api-actions.md`, `states-and-validation.md` и work plan.

## 5. Что делать при проблеме

Если в спецификациях чего-то нет:

- не реализовывать;
- добавить или уточнить spec;
- после этого обновить work plan.

Если в спецификациях неясно:

- не угадывать;
- уточнить правило в соответствующем документе;
- только потом планировать реализацию.

Если work plan противоречит spec:

- остановить разработку;
- исправить spec или work plan;
- продолжать только после согласования.
