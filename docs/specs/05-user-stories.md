# User Stories

# Пользовательские сценарии MVP

## 1. Назначение документа

Этот документ описывает реальные пути пользователей в первой версии платформы.

Предыдущие спецификации отвечают на вопросы:

- что делает система;
- какие сущности есть в базе;
- какие страницы нужны;
- какие действия доступны;
- какие роли и ограничения существуют.

User Stories отвечают на другой вопрос: как человек проходит рабочий путь от начала до результата.

Документ нужен, чтобы перед разработкой проверить:

- не потеряна ли важная часть основного сценария;
- хватает ли страниц и действий для MVP;
- какие сценарии должны быть проверены вручную после реализации;
- какой порядок разработки будет самым логичным.

## 2. Главный пользовательский путь MVP

Главный путь MVP должен доказать, что платформа может заменить минимальный бумажный журнал преподавателя таджвида.

Базовый сценарий:

1. Админ или преподаватель-одиночка создает курс таджвида.
2. Создает группу.
3. Добавляет преподавателя.
4. Добавляет учеников.
5. Задает расписание.
6. Система создает ближайшие уроки.
7. Преподаватель открывает урок.
8. Отмечает посещаемость.
9. Добавляет комментарий, прогресс, домашнее задание и материал.
10. Завершает урок.
11. Ученик входит в кабинет.
12. Ученик видит ближайший урок, домашнее задание, материалы, посещаемость, прогресс и оплату.

Если этот путь работает, первый релиз считается полезным.

## 3. Роли в сценариях

В MVP есть основные пользовательские контексты:

- админ/директор;
- преподаватель;
- ученик;
- преподаватель-одиночка.

Преподаватель-одиночка не является отдельной технической ролью. Это пользователь, который совмещает возможности преподавателя и базового администратора для своих данных.

## 4. История: админ создает учебную основу

### 4.1. Ситуация

В учебном центре начинается новый курс таджвида. Админу нужно создать основу учебного процесса: курс, группу, преподавателя, учеников и расписание.

### 4.2. Участник

Админ/директор.

### 4.3. Цель

Подготовить группу так, чтобы преподаватель мог начать вести уроки.

### 4.4. Предусловия

- Админ вошел в систему.
- У админа есть доступ к своей организации.
- В системе может еще не быть курсов, групп и учеников.

### 4.5. Основной сценарий

1. Админ открывает `/admin`.
2. Переходит в раздел курсов.
3. Создает курс таджвида.
4. Проверяет, что для курса включен простой блок таджвид-прогресса.
5. Переходит в раздел групп.
6. Создает новую группу.
7. Выбирает курс таджвида для группы.
8. Назначает преподавателя.
9. Добавляет учеников.
10. Задает повторяющееся расписание.
11. Сохраняет группу.
12. Система создает ближайшие занятия по расписанию.
13. Админ открывает страницу группы и проверяет состав, преподавателя и расписание.

### 4.6. Результат

- Есть активный курс.
- Есть активная группа.
- У группы есть преподаватель.
- В группе есть ученики.
- У группы есть расписание.
- Преподаватель может открыть группу и ближайший урок.

### 4.7. Важные проверки

- Нельзя создать группу без курса.
- Нельзя назначить ученика в чужую организацию.
- Нельзя назначить преподавателя из чужой организации.
- Расписание должно создавать уроки предсказуемо.
- Старые записи журнала не должны теряться при изменении расписания.

## 5. История: преподаватель проводит урок

### 5.1. Ситуация

Преподаватель проводит обычный урок таджвида и хочет быстро зафиксировать посещаемость, прогресс, домашнее задание и материалы.

### 5.2. Участник

Преподаватель.

### 5.3. Цель

Провести урок в системе без лишней административной нагрузки.

### 5.4. Предусловия

- Преподаватель вошел в систему.
- У преподавателя есть назначенная группа.
- У группы есть ближайший урок.
- В группе есть ученики.

### 5.5. Основной сценарий

1. Преподаватель открывает `/teacher`.
2. Видит ближайшие занятия и свои группы.
3. Открывает нужную группу.
4. Переходит в журнал или сразу открывает ближайший урок.
5. Нажимает действие начала урока.
6. Видит список учеников группы.
7. Отмечает отсутствующего ученика как `absent`.
8. При необходимости отмечает уважительную причину как `excused`.
9. Оставляет пустую отметку или явно ставит `present` для присутствующих.
10. Подтверждает посещаемость.
11. Добавляет общий комментарий к уроку.
12. При необходимости ставит оценку за урок по выбранной шкале.
13. Если нужно, открывает конкретного ученика и обновляет его прогресс: добавленные ученику правила, уровни по этим правилам, ошибки и что нужно повторить.
14. Выбирает, какая часть прогресса видна ученику.
15. Добавляет домашнее задание.
16. Прикрепляет текстовый материал или ссылку.
17. Сохраняет урок.
18. Завершает урок.

### 5.6. Результат

- Урок имеет завершенный статус.
- Посещаемость зафиксирована.
- Комментарий сохранен.
- Прогресс сохранен.
- Домашнее задание создано.
- Материал доступен ученику, если он открыт для ученика.

### 5.7. Важные проверки

- Пустая отметка считается присутствием только после завершения урока.
- До завершения урока пустая отметка не должна считаться присутствием.
- Ученик не видит внутренние комментарии преподавателя.
- Ученик видит только открытые для него части прогресса.
- Админ может смотреть журнал, но не редактирует учебные записи преподавателя в MVP.

## 6. История: преподаватель смотрит группу и журнал

### 6.1. Ситуация

Преподавателю нужно быстро понять состояние группы: кто занимается, какие уроки прошли, кто часто отсутствует и что задано.

### 6.2. Участник

Преподаватель.

### 6.3. Цель

Получить рабочую картину группы без открытия лишних разделов.

### 6.4. Основной сценарий

1. Преподаватель открывает список групп.
2. Выбирает группу.
3. Видит название группы, курс, расписание и список учеников.
4. Открывает календарный журнал.
5. Выбирает месяц.
6. Видит уроки месяца.
7. Открывает конкретный урок.
8. Проверяет посещаемость, комментарий, прогресс и домашнее задание.
9. При необходимости редактирует еще не завершенный урок или добавляет недостающую запись.

### 6.5. Результат

- Преподаватель понимает состояние группы.
- У него есть быстрый доступ к урокам и ученикам.
- Журнал можно использовать как основной рабочий экран.

## 7. История после базового MVP: преподаватель ведет индивидуального ученика

Этот сценарий описывает развитие после прохождения базового группового MVP.
Модель данных готовится заранее, но отдельный рабочий путь индивидуального
обучения не входит в первый smoke-сценарий.

### 7.1. Ситуация

Преподаватель занимается с учеником индивидуально, без группы.

### 7.2. Участник

Преподаватель.

### 7.3. Цель

Вести индивидуальное обучение так же удобно, как групповое, но без искусственного создания группы.

### 7.4. Основной сценарий

1. Преподаватель открывает индивидуальных учеников.
2. Создает или выбирает ученика.
3. Назначает курс таджвида.
4. Задает индивидуальное расписание.
5. Открывает ближайший индивидуальный урок.
6. Отмечает посещаемость.
7. Записывает прогресс.
8. Добавляет домашнее задание.
9. Прикрепляет материал.
10. Завершает урок.

### 7.5. Результат

- Индивидуальный ученик не называется группой в продукте.
- У него есть свои уроки, прогресс, домашка, материалы и оплата.
- Ученик видит свои данные в кабинете.

## 8. История: ученик открывает личный кабинет

### 8.1. Ситуация

Ученик хочет понять, когда следующий урок, что нужно подготовить и какие материалы дал преподаватель.

### 8.2. Участник

Ученик.

### 8.3. Цель

Получить простую учебную картину без доступа к административным данным.

### 8.4. Предусловия

- Ученик вошел в систему.
- Ученик привязан к группе.
- Преподаватель уже создал урок, домашнее задание или материал.

### 8.5. Основной сценарий

1. Ученик открывает `/student`.
2. Видит ближайший урок.
3. Видит свои курсы и группы.
4. Открывает домашние задания.
5. Видит задание от преподавателя.
6. Открывает материалы.
7. Видит текстовый материал или ссылку.
8. Открывает прогресс.
9. Видит только те комментарии, правила и ошибки, которые преподаватель открыл для ученика.
10. Открывает посещаемость.
11. Видит свои посещения и пропуски.
12. Открывает оплату.
13. Видит сумму, срок, период и статус оплаты.

### 8.6. Результат

- Ученик понимает, что готовить к следующему уроку.
- Ученик видит открытые материалы.
- Ученик видит свой прогресс.
- Ученик видит только свои данные.

### 8.7. Важные проверки

- Ученик не видит других учеников.
- Ученик не редактирует прогресс.
- Ученик не редактирует посещаемость.
- Ученик не редактирует оплату.
- Ученик не видит внутренние комментарии преподавателя.

## 9. История: админ контролирует оплату

### 9.1. Ситуация

Админу нужно понять, кто оплатил обучение, кто должен оплатить и кому нужно напомнить.

### 9.2. Участник

Админ/директор.

### 9.3. Цель

Вести ручной учет оплат без онлайн-эквайринга.

### 9.4. Основной сценарий

1. Админ открывает `/admin/payments`.
2. Видит список оплат по ученикам.
3. Фильтрует или открывает неоплаченные оплаты.
4. Открывает конкретного ученика.
5. Создает или редактирует оплату.
6. Указывает сумму.
7. Указывает валюту.
8. Указывает срок оплаты.
9. Указывает период оплаты.
10. Выбирает статус `pending`, `paid`, `overdue` или `exempt`.
11. Сохраняет оплату.
12. Проверяет, что ученик видит актуальный статус в своем кабинете.

### 9.5. Результат

- Оплата ученика сохранена.
- Админ видит список должников.
- Ученик видит свой статус оплаты.
- Система не принимает платежи автоматически.

### 9.6. Важные проверки

- Онлайн-оплата не входит в MVP.
- Автоматические напоминания не входят в MVP.
- Статус оплаты меняется вручную.
- Преподаватель-одиночка может управлять оплатой своих учеников.

## 10. История: преподаватель-одиночка запускает свою работу

### 10.1. Ситуация

Преподаватель работает сам, без отдельного администратора и учебного центра.

### 10.2. Участник

Преподаватель-одиночка.

### 10.3. Цель

Начать пользоваться системой без сложной административной настройки.

### 10.4. Основной сценарий

1. Преподаватель-одиночка входит в систему.
2. Создает курс таджвида.
3. Создает группу.
4. Добавляет учеников.
5. Задает расписание.
6. Открывает ближайший урок.
7. Ведет посещаемость, прогресс, домашку и материалы.
8. При необходимости создает оплату ученику.
9. Ученик входит в кабинет и видит свои данные.

### 10.5. Результат

- Один человек может настроить и вести учебный процесс.
- Для старта не нужен отдельный директор или администратор.
- В интерфейсе не должно быть ощущения сложной школьной CRM.

## 11. История: админ смотрит состояние учебного процесса

### 11.1. Ситуация

Директору или админу нужно быстро понять, работает ли учебный процесс: есть ли группы, преподаватели, ученики, расписание, пропуски и долги.

### 11.2. Участник

Админ/директор.

### 11.3. Цель

Получить базовый контроль без сложной аналитики.

### 11.4. Основной сценарий

1. Админ открывает `/admin`.
2. Видит количество курсов, групп, преподавателей и учеников.
3. Видит активные группы.
4. Видит группы без расписания.
5. Видит группы без преподавателя.
6. Видит учеников с задолженностью.
7. Видит учеников с частыми пропусками.
8. Открывает нужную группу, ученика или оплату.

### 11.5. Результат

- Админ понимает общее состояние системы.
- Проблемные места можно открыть из панели.
- Сложные графики и отчеты не требуются для MVP.

## 12. История: материал доступен ученику

### 12.1. Ситуация

Преподаватель хочет дать ученикам текст, ссылку на видео или другой внешний ресурс без загрузки файлов в систему.

### 12.2. Участник

Преподаватель.

### 12.3. Цель

Добавить простой учебный материал и открыть его ученикам.

### 12.4. Основной сценарий

1. Преподаватель открывает материалы.
2. Создает текстовый материал или ссылку.
3. Привязывает материал к курсу, группе, уроку или домашнему заданию.
4. Выбирает, виден ли материал ученику.
5. Сохраняет материал.
6. Ученик открывает кабинет.
7. Ученик видит открытый материал.

### 12.5. Результат

- Материал сохранен.
- Материал можно использовать в учебном сценарии.
- Ученик видит только открытые материалы.

### 12.6. Не входит в MVP

- загрузка файлов;
- PDF-хранилище;
- аудио;
- видео;
- изображения;
- проверка доступности ссылок.

## 13. История: прогресс таджвида обновляется после урока

### 13.1. Ситуация

После урока преподаватель хочет обновить состояние конкретного ученика: какие правила он отслеживает у этого ученика, какой уровень по этим правилам и что нужно повторить.

### 13.2. Участник

Преподаватель.

### 13.3. Цель

Сохранить понятный прогресс ученика отдельно от оценки за урок.

### 13.4. Основной сценарий

1. Преподаватель открывает урок или карточку ученика.
2. Выбирает конкретного ученика.
3. Открывает блок прогресса ученика.
4. Видит правила, которые уже добавлены этому ученику.
5. При необходимости добавляет ученику новое правило.
6. По правилам ученика ставит или обновляет уровень: отлично, хорошо, удовлетворительно или плохо.
7. Добавляет или обновляет ошибки чтения у этого ученика.
8. Добавляет комментарий.
9. Заполняет поле "что нужно повторить".
10. Добавляет внутренний комментарий, если нужно.
11. Выбирает, что показывать ученику.
12. Сохраняет прогресс.
13. Ученик видит открытую часть прогресса в кабинете.

### 13.5. Результат

- Прогресс сохранен.
- Преподаватель видит подробную картину.
- Ученик видит только полезную и открытую часть.

## 14. Ошибочные и крайние ситуации

### 14.1. Нет расписания

Если у группы нет расписания:

- ближайшие уроки не создаются;
- группа должна быть видна админу как требующая настройки;
- преподаватель должен понимать, почему нет ближайших уроков.

### 14.2. Нет учеников в группе

Если группа пустая:

- урок можно видеть в расписании;
- журнал показывает, что учеников нет;
- админ должен иметь быстрый путь к добавлению учеников.

### 14.3. Урок еще не завершен

Если урок начат, но не завершен:

- пустые отметки не считаются итоговым присутствием;
- преподаватель может вернуться и дозаполнить урок;
- ученик не должен видеть непроверенные внутренние данные.

### 14.4. Ученик архивирован

Если ученик архивирован:

- он не должен появляться в активных списках по умолчанию;
- история его уроков и оплат не должна удаляться;
- старые записи журнала должны сохраняться.

### 14.5. Изменилось расписание

Если расписание группы изменилось:

- новые будущие уроки создаются по новым правилам;
- уже проведенные уроки и записи журнала не удаляются;
- админ и преподаватель должны понимать, какие уроки относятся к старому расписанию.

## 15. Сценарии, которые не входят в первый релиз

В первый релиз не входят:

- онлайн-оплата;
- автоматические уведомления;
- родительский кабинет;

## 16. Минимальный smoke-сценарий для проверки MVP

После реализации первый smoke-тест должен пройти такой путь:

1. Создать организацию.
2. Создать админа.
3. Создать преподавателя.
4. Создать ученика.
5. Создать курс таджвида.
6. Создать группу.
7. Назначить курс группе.
8. Назначить преподавателя группе.
9. Добавить ученика в группу.
10. Задать расписание.
11. Создать ближайший урок.
12. Войти как преподаватель.
13. Открыть группу.
14. Открыть урок.
15. Отметить посещаемость.
16. Добавить комментарий.
17. Добавить прогресс.
18. Добавить домашнее задание.
19. Добавить материал-ссылку.
20. Войти как ученик.
21. Увидеть ближайший урок.
22. Увидеть домашнее задание.
23. Увидеть материал.
24. Увидеть посещаемость.
25. Увидеть открытую часть прогресса.
26. Войти как админ.
27. Создать оплату ученику.
28. Войти как ученик.
29. Увидеть статус оплаты.

Если этот smoke-сценарий проходит, основа MVP работает.
